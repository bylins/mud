# Ручная проверка фикса #3178 (kZonedecay + kTicktimer)

Эта инструкция описывает шаги для иммортала, который хочет вживую убедиться, что:

1. Предмет с флагом `kZonedecay` рассыпается, если оказался **вне** своей родной зоны — независимо от того, лежит он в комнате, его кто-то несёт, надел или положил в контейнер.
2. Предмет с флагом `kZonedecay`, лежащий в **родной** зоне, не разрушается.
3. Обычный предмет с положительным таймером, лежащий в комнате (без флага `kTicktimer`), **не теряет** время — таймер «заморожен» до тех пор, пока с ним не повзаимодействует игрок.
4. После того как игрок подобрал предмет (выставился `kTicktimer`), таймер начинает капать.

## Подготовка

```bash
# В корне репозитория
cmake -S . -B build -DBUILD_TESTS=ON -DCMAKE_BUILD_TYPE=Release
make -C build circle -j$(($(nproc)/2))

# Запуск (порт можно любой свободный)
cd build
./circle -d small 4000
```

Подключаемся клиентом, заходим иммом, поднимаем уровень до Implementor (или
тем уровнем, что у вас прописан в `lib.template`).

Полезные команды (имена даны как в `interpreter.cpp`, можно сокращать):

| Команда                | Что делает                                                  |
|------------------------|-------------------------------------------------------------|
| `goto <vnum>`          | Перейти в комнату по vnum                                   |
| `load obj <vnum>`      | Загрузить объект себе в инвентарь (см. `do_load.cpp`)       |
| `load mob <vnum>`      | Загрузить моба в текущую комнату                            |
| `stat obj <name>`      | Показать характеристики объекта                             |
| `stat room`            | Текущая комната (зона, флаги)                               |
| `purge <name>`         | Уничтожить объект/моба                                      |
| `force <victim> <cmd>` | Заставить кого-то выполнить команду                         |
| `set obj <name> <…>`   | Изменить поле объекта (если применимо к зоне-вкладу)        |

`stat obj` важна — у объекта есть строка `..., "����������� ���� VNUM" : N` (см.
`do_stat.cpp:709`). Это поле `vnum_zone_from`, по которому проверяется зональная
принадлежность.

> **Важно**: команда `load obj` автоматически выставляет
> `vnum_zone_from = текущая зона иммортала`. То есть «родной» зоной для
> загруженного предмета становится та, где вы стоите в момент загрузки.

## Сценарии

Заранее выберите два vnum-а в **разных зонах**, например:

* комната `A_HOME = 100` в зоне X (родная зона);
* комната `A_FOREIGN = 30001` в зоне Y (чужая зона);
* любой объект `OBJ_VNUM` с флагом `kZonedecay` (если такого нет в зон-файлах,
  заведите тестовый через `oedit`: создайте предмет, в `extra flags`
  отметьте `ZONE-DECAY`).

Подставьте свои значения вместо `A_HOME`, `A_FOREIGN`, `OBJ_VNUM`.

### Сценарий 1: лежит в родной зоне — не рассыпается

```
goto A_HOME
load obj OBJ_VNUM
drop <name>
stat obj <name>
```

Убедитесь, что в `stat obj` поле `����������� ���� VNUM` совпадает с vnum зоны,
в которой вы стоите. После 5–10 минут реального времени (это
несколько `obj_point_update`-тиков по 60 секунд каждый):

```
look
stat obj <name>
```

Ожидаемо: предмет на месте, в `stat obj` таймер показывает `unlimited` (если
исходный `m_timer == 0`) либо своё начальное значение без декремента.

### Сценарий 2: брошен в чужой зоне — рассыпается на следующем тике

```
goto A_HOME
load obj OBJ_VNUM
goto A_FOREIGN
drop <name>
look
```

Подождите до **минуты** (один цикл `obj_point_update`). Должно прийти сообщение
вида `<предмет> рассыпался в труху, ...`, и `look` покажет пустую комнату.

### Сценарий 3 (главный для #3178): несут в чужой зоне — рассыпается

Это сценарий, ради которого пиар:

```
goto A_HOME
load obj OBJ_VNUM
inventory
goto A_FOREIGN
inventory
```

Подождите до минуты, повторите `inventory`. Предмет должен исчезнуть, придёт
сообщение о разрушении.

### Сценарий 4: надет в чужой зоне — рассыпается

Используется любой носимый предмет с `kZonedecay`. Если у объекта нет wear-flag,
дайте его моделкой через `oedit` (например, `wear flags`: HOLD).

```
goto A_HOME
load obj OBJ_VNUM
wear <name>
goto A_FOREIGN
equipment
```

Подождите до минуты, проверьте `equipment` — предмет должен исчезнуть.

### Сценарий 5: лежит в комнате с положительным таймером — таймер не капает

Возьмите обычный предмет (без `kZonedecay`) с заданным `timer` в зон-файле,
например `timer = 5` — это означает 5 минут MUD-времени. Положите его в комнату
и **не трогайте**:

```
goto A_HOME
load obj <vnum_with_timer_5>
drop <name>
stat obj <name>          # запомните таймер
```

Подождите 10 минут реального времени (≥ 5 циклов `obj_point_update`). Затем
снова:

```
stat obj <name>
```

Ожидаемо: таймер не уменьшился. До регрессии #3065 это было нормой; после
рефакторинга предмет тихо умирал. После фикса — снова заморожен.

### Сценарий 6: подняли — таймер начал капать

Продолжаем со сценарием 5. Теперь поднимаем предмет:

```
get <name>
stat obj <name>          # проверьте, что появился флаг TICKTIMER
```

Подождите 5+ минут. Снова:

```
stat obj <name>
```

Таймер должен уменьшиться. После полного истечения предмет исчезнет.

### Сценарий 7: положили в контейнер в чужой зоне — рассыпается

Дополнительный сценарий, чтобы проверить, что `up_obj_where` корректно
отрабатывает контейнеры:

```
goto A_HOME
load obj OBJ_VNUM
load obj <vnum_контейнера>
put <name_obj> <name_container>
goto A_FOREIGN
drop <name_container>
look
```

Подождите минуту, контейнер останется (он не `kZonedecay`), но предмет внутри
должен рассыпаться. Откройте/посмотрите контейнер: внутри пусто.

## Что наблюдать в логах

В отдельном терминале:

```bash
tail -F build/log/syslog | grep -i 'obj decay'
```

Сообщения `[Obj decay] for: ...` выходят при разрушении предмета через
`CheckObjDecay`. Сообщения о `kZonedecay`-разрушении — при срабатывании
секции 4 идут через стандартный путь `do_decay`/`act` и пишутся в комнату.

## Что считать пройденным

* Сценарии 1 и 5 — предмет **не** должен исчезать.
* Сценарии 2, 3, 4, 7 — предмет должен исчезать в течение одного
  `obj_point_update` (≤ 60 секунд) после того, как игрок попадает в чужую зону.
* Сценарий 6 — таймер должен видимо уменьшиться через несколько тиков.

Если в каком-то сценарии поведение отличается — записывайте `stat obj`,
снимайте лог `syslog` и сообщайте.
