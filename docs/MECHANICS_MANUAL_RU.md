# Руководство по настройке механик BRusMUD

Этот документ описывает **настраиваемые данными** игровые механики движка: где лежит их
конфигурация, как её править и перезагружать «на лету», и что означает каждый параметр.

Правило движка: настраиваемая механика описывается конфигом в `cfg/mechanics/…`, грузится через
`CfgManager`, перезагружается командой `reload …` и (если у неё есть схема) правится редактором
`vedun`. Логику, которую держат в коде (например, влияние способностей-фитов), в конфиг не выносят —
такие места отдельно оговорены.

Оглавление:

- [Поднятие нежити (Animate Dead)](#поднятие-нежити-animate-dead)

## Поднятие нежити (Animate Dead)

Механика конструктов чернокнижника: заклинание **«поднять труп»** превращает труп в подконтрольную
нежить. Тип нежити зависит от уровня трупа и мастерства заклинателя, число одновременно управляемых
существ ограничено «бюджетом управления», а характеристики поднятого существа масштабируются от силы
накладывания заклинания. Все эти параметры вынесены в один файл, чтобы гейм-дизайнеры могли их
балансировать без пересборки движка.

### 1. Где лежит и как применять

| | |
|---|---|
| Файл | `cfg/mechanics/animate_dead.xml` |
| Схема для `vedun` | `cfg/mechanics/animate_dead.scheme` |
| Идентификатор в `CfgManager` | `animate_dead` |
| Перезагрузка без рестарта | `reload animatedead` |
| Правка в игре | `vedun creature` (список), `vedun creature <vnum>` (правка тира) |
| Загрузка при старте | `MUD::CfgManager().LoadCfg("animate_dead")` в `db.cpp` (грузится при загрузке мира) |

Порядок работы дизайнера: правим `animate_dead.xml` (руками или через `vedun`) → `reload animatedead`
→ проверяем в игре. Изменения вступают в силу сразу для **новой** поднимаемой нежити.

> Файл — обычный ASCII/латиница (английские имена атрибутов), его можно править любым редактором.
> **Внимание:** в теле XML-комментариев `<!-- ... -->` нельзя использовать двойной дефис `--`
> (это запрещено спецификацией XML) — пишите одиночный `-`.

### 2. Структура файла

```xml
<animate_dead>
    <!-- бюджет управления = min(<skill>, budget_cap) - сумма весов удерживаемой нежити -->
    <control skill="kDarkMagic" budget_cap="75" />

    <creature vnum="0" id="kSkeleton" proto_vnum="3001" weight="8">
        <cost corpse_max_level="5" min_rating="0" />
        <scaling>
            <hp          beta="0.25" mode="mult" />
            <damage_dice beta="1.5"  mode="add"  cap="100" />
            <hitroll     beta="0"    mode="add" />
            <ac          beta="0"    mode="add" />        <!-- меньше AC = лучше; задайте cap= для нижней границы -->
            <skills      beta="0"    mode="add"  cap="200" />
            <saving      beta="0"    mode="add" />        <!-- все 4 спасброска; меньше = лучше -->
            <armor       beta="0"    mode="add" />
            <morale      beta="0"    mode="add" />
            <initiative  beta="0"    mode="add" />
        </scaling>
    </creature>

    <!-- остальные тиры, от слабого к сильному ... -->
</animate_dead>
```

Тиры (`<creature>`) перечисляются **от слабейшего к сильнейшему** — этот порядок и есть «лестница»,
по которой механика понижает выбор, когда сильный тир недоступен.

### 3. Блок `<control>` — бюджет управления

| Атрибут | Смысл |
|---|---|
| `skill` | Навык, задающий бюджет и «рейтинг» заклинателя. По умолчанию `kDarkMagic`. |
| `budget_cap` | Верхняя граница бюджета: даже при очень высоком навыке бюджет не превысит это значение. |

**Бюджет управления** = `min(навык, budget_cap) − Σ(вес уже удерживаемой нежити)`.

Каждый тир «стоит» свой `weight`. Пока в бюджете есть место под вес нового существа — его можно
поднять; если места нет даже под слабейший тир, поднятие срывается сообщением
«Вы не сможете управлять столькими последователями.».

> **Важно (скрытый порог входа):** вес слабейшего тира — это фактический минимальный навык, при
> котором можно поднять хоть кого-то. Если у слабейшего тира `weight=8`, то заклинатель с навыком
> ниже 8 не поднимет вообще ничего. Держите вес входного тира не выше навыка, ожидаемого у
> начинающего некроманта.

### 4. Тиры `<creature>` и блок `<cost>`

Атрибуты `<creature>`:

| Атрибут | Смысл |
|---|---|
| `vnum` | Индекс тира в конфиге (ключ для `vedun`); задаёт порядок лестницы. |
| `id` | Символьное имя тира, например `kSkeleton` (для списков и логов). |
| `proto_vnum` | VNUM моба-прототипа, который будет поднят. |
| `weight` | Стоимость одного такого существа в бюджете управления. |

Блок `<cost>` — условия доступности тира:

| Атрибут | Смысл |
|---|---|
| `corpse_max_level` | Наибольший уровень трупа-источника, дающий этот тир (полоса по уровню трупа). |
| `min_rating` | Наименьший «рейтинг» заклинателя, при котором тир разрешён (см. ниже). |
| `pick` | Относительные шансы выбора внутри общей верхней полосы (необязательный; `0`/отсутствует = одиночный тир). |

### 5. Как выбирается тир при поднятии

Когда игрок поднимает труп, тир определяется в три шага (`PickTier`):

1. **Полоса по уровню трупа.** Берётся слабейший тир, чей `corpse_max_level` покрывает уровень
   трупа. То есть слабый труп даёт слабую нежить, сильный — сильную.

2. **Случайный выбор внутри полосы.** Если несколько тиров делят одну верхнюю полосу
   (одинаковый `corpse_max_level`), между ними бросается жребий по весам `pick`. Так, например,
   один и тот же «высокоуровневый» труп с шансом 50/50 даёт «урончика» или «дыхателя».

3. **Гейт по рейтингу заклинателя.** Выбранный тир понижается по лестнице вниз, пока рейтинг
   заклинателя не дотянет до `min_rating` тира:

   ```
   рейтинг = уровень + псевдо-реморт + 4
   псевдо-реморт = max(0, (навык − SkillCapStart) / SkillCapIncrement)
   ```

   где `SkillCapStart` (порог навыка при 0 ремортов, обычно **80**) и `SkillCapIncrement`
   (прибавка кэпа навыка за реморт, обычно **5**) — константы движка. То есть рейтинг растёт от
   **навыка**, а не от числа ремортов: навык 80 → +0, навык 85 → +1, навык 90 → +2 и т.д.

   Пример: персонаж 30 уровня с навыком 80 → рейтинг = 30 + 0 + 4 = 34; при `min_rating=32` он
   пройдёт гейт и сможет поднимать верхние тиры.

После выбора тира срабатывает проверка бюджета (`FitUndeadTier`, шаг из раздела 3): если на
выбранный тир не хватает бюджета, он **понижается** до ближайшего тира, который влезает; если не
влезает даже слабейший — поднятие срывается.

### 6. Пример: почему «скелет», а не отказ

Персонаж «Чернок», 10 уровень, навык `kDarkMagic` = 10, поднимает труп 4 уровня:

1. Полоса: труп 4 уровня ≤ `corpse_max_level=5` скелета → выбран **скелет**.
2. Гейт: рейтинг = 10 + max(0, (10−80)/5) + 4 = 10 + 0 + 4 = 14 ≥ `min_rating=0` → скелет остаётся.
3. Бюджет: `min(10, 75) − 0 = 10`; вес скелета `8 ≤ 10` → скелет поднимается.

Второго скелета Чернок уже не удержит: остаток бюджета `10 − 8 = 2 < 8` → «Вы не сможете управлять
столькими последователями.». (Если бы вес скелета был `12`, как в первой версии, Чернок не поднял бы
даже одного — отсюда правило про «скрытый порог входа» из раздела 3.)

### 7. Масштабирование характеристик — блок `<scaling>`

Характеристики поднятого существа улучшаются относительно **прототипа** пропорционально
**компетентности накладывания** `C` (`SetupUndeadStats`). Каждая строка `<scaling>` — это одна
характеристика со своими параметрами:

| Атрибут | Смысл |
|---|---|
| `beta` | Коэффициент масштабирования. `0` = характеристика не меняется (берётся как у прототипа). |
| `mode` | `add` — прибавка; `mult` — множитель (см. формулы ниже). |
| `cap` | Граница результата: для «растущих» характеристик — потолок, для «убывающих» (AC, спасброски) — нижняя граница. Необязателен. |

Масштабируемые характеристики: `hp`, `damage_dice` (число костей урона), `hitroll`, `ac`,
`skills` (все умения прототипа, равномерно), `saving` (все 4 спасброска), `armor`, `morale`,
`initiative`. Характеристики `luck` нет намеренно — у мобов её не существует.

**Формулы.** Пусть `C` — компетентность (см. раздел 8):

- «Растущие» (`hitroll`, `armor`, `morale`, `initiative`, `skills`, `damage_dice`), `mode="add"`:
  `значение = прототип + round(beta · C)`, затем ограничивается сверху `cap`.
- «Убывающие» (`ac`, `saving`), где меньше = лучше, `mode="add"`:
  `значение = прототип − round(beta · C)`, затем ограничивается снизу `cap`.
- Здоровье `hp`, `mode="mult"`: `значение = прототип × (1 + beta · C)` (при `cap` — не выше него).
  Множитель сохраняет пропорции между тирами.
- `damage_dice` дополнительно зажимается в диапазон `1…100` (защита знакового `damnodice`).

Стартовые значения в конфиге воспроизводят историческое поведение: `hp beta=0.25 mode=mult`
(+25 % · C к здоровью) и `damage_dice beta=1.5 mode=add cap=100`. Остальные семь характеристик
поставлены на `beta=0` (выключены) — их можно включать и настраивать по вкусу.

### 8. Что такое компетентность `C`

`C` — та же «сила накладывания», что и у прочих заклинаний: сумма вклада навыка и профильной
характеристики (для нежити это Мудрость):

```
C = skill_coeff + stat_coeff
skill_coeff = ( min(навык, 75)·low_bonus + max(0, навык−75)·hi_bonus ) / 100
stat_coeff  = max(0, мудрость − порог) · stat_weight / 100
```

Коэффициенты `low_bonus`, `hi_bonus`, `порог` и `stat_weight` берутся из блока `<potency_roll>`
заклинания `kAnimateDead` в `cfg/…/spells.xml` — то есть «крутизна» масштабирования настраивается
там, а `beta` в `animate_dead.xml` лишь задаёт, насколько сильно данная характеристика реагирует
на `C`.

Практически: `C` растёт с навыком Тёмной магии и Мудростью и для опытного некроманта имеет порядок
единиц. Поскольку точное значение зависит от `<potency_roll>`, подбирайте `beta` **опытным путём**:
выставьте значение, поднимите существо разными по силе заклинателями и посмотрите итоговые
характеристики (`vstat`/`stat` по мобу).

### 9. Практические рецепты

- **Добавить новый тир.** `vedun creature new` создаст запись с первым свободным `vnum`; заполните
  `id`, `proto_vnum`, `weight`, блок `<cost>` и `<scaling>`. Не забудьте про порядок лестницы: тир
  должен стоять между соседями по силе.
- **Разрешить более раннее поднятие сильного тира.** Уменьшите его `min_rating` и/или `weight`.
- **Ограничить «толпу».** Поднимите `weight` тиров или снизьте `budget_cap` — управляемых существ
  станет меньше. Держите `weight` сильнейшего тира равным `budget_cap`, если хотите «максимальный
  навык = ровно одно топовое существо».
- **Сделать существо крепче/сильнее.** Поднимите `beta` у нужной характеристики; для «убывающих»
  (AC, спасброски) обязательно задайте `cap` как нижнюю границу, иначе значение уйдёт в минус без
  предела.
- **Расширить/сузить полосу уровней трупа.** Меняйте `corpse_max_level`: тир получают все трупы с
  уровнем не выше указанного (и выше `corpse_max_level` предыдущего тира).

### 10. Типичные ошибки

- **Вес входного тира выше стартового навыка** → начинающий некромант не поднимает ничего и видит
  «слишком много последователей», хотя нежити у него нет. Держите `weight` слабейшего тира ≤
  ожидаемого навыка новичка.
- **`beta` у «убывающей» характеристики без `cap`** → AC или спасброски уходят в бесконечный минус.
- **Нарушен порядок лестницы** (`<creature>` не по возрастанию силы) → понижение по гейту и бюджету
  выберет «не тот» тир. Перечисляйте от слабого к сильному.
- **`--` в комментарии XML** → строгий парсер отвергнет файл; используйте одиночный дефис.
- **Правка `min_rating` под реморты** — не нужно: гейт считает псевдо-реморт из навыка, реморты
  напрямую не участвуют.

### 11. Что осталось в коде (не в конфиге)

- Влияние способностей-фитов (например, «Ярость Тьмы» / `kFuryOfDarkness`, дающая прибавку к
  характеристикам нежити) применяется в коде поверх конфигурационного масштабирования.
- Само разбиение на три шага выбора (полоса → жребий → гейт) и формулы масштабирования — в коде
  (`src/gameplay/mechanics/animate_dead.cpp`); конфиг задаёт только их **данные**.
