# Bylins MUD — Руководство по конфигурации эффектов

*Для гейм-дизайнеров, описывающих эффекты в `lib/cfg/affects.xml` (эффекты
персонажей) и `lib/cfg/room_affects.xml` (эффекты комнат / выходов), а также их
файлы сообщений.*

**Эффект** — это длящееся воздействие на персонажа, комнату или выход: бафф,
дебафф, урон-во-времени (DoT), щит, метка призыва, статусный флаг. Если
[Руководство по заклинаниям](SPELL_MANUAL_RU.md) описывает *произнесение* (как
заклинание достигает целей и что делает *однократно*), то это руководство
описывает, что эффект делает *пока длится* и *когда с его носителем что-то
происходит*.

Раньше эффект был лишь именем + набором модификаторов статов, накладываемым одним
заклинанием, а любое поведение сверх изменения статов было зашито в C++. Сейчас
всё наоборот: **эффект владеет собственными данными** — своими флагами поведения,
модификаторами статов, весом магического щита и (главное дополнение) списком
**действий `<actions>`, управляемых триггерами**. Заклинание (или другой эффект,
или способность) лишь *накладывает* эффект; дальше эффект сам решает, что делать.

Руководство предполагает знакомство с общим словарём
[Руководства по заклинаниям](SPELL_MANUAL_RU.md) (`<damage>`, `<points>`,
`<area>`, `<affects>`, `<unaffect>`, бросок potency, перечисления
`EApply`/`ESaving`/`EResist`) — эти теги значат здесь то же самое. Новое — это
**слой триггеров/действий** (§5) и правила описания эффектов.

---

## 1. Где живут эффекты

| Файл | Хранит | Файл сообщений |
|---|---|---|
| `lib/cfg/affects.xml` | эффекты **персонажей** (`EAffect`) | `lib/cfg/messages/<lang>/affect_msg.xml` |
| `lib/cfg/room_affects.xml` | эффекты **комнат / выходов** (`ERoomAffect`) | `lib/cfg/messages/<lang>/room_affect_msg.xml` |

Раскладка файлов сообщений проверяется по схеме
`lib/cfg/messages/affect_msg.scheme`. Как и с заклинаниями, тестовый мир — это
копия; правьте отслеживаемые файлы в `lib.template/cfg` (или живые `lib/cfg`) и
перезагружайте, не редактируйте копию `small/` конкретной сборки.

**Перезагрузка:** `reload affects` / `reload roomaffects` (и перезагрузки
сообщений) подхватывают изменения без рестарта, как `reload spells`.

## 2. Стыковка заклинания и эффекта

Когда `<affects><affect id="kX"/>` заклинания (§8 Руководства по заклинаниям)
срабатывает, движок строит по одному экземпляру `Affect` **на каждый apply** и
передаёт владение эффекту:

| Принадлежит **эффекту** (`affects.xml`) | Принадлежит **наложению** (блоку `<affects>` заклинания / действия) |
|---|---|
| `<flags>` — флаги поведения (`EAffFlag`) | `<duration>` — длительность |
| `<apply>` — изменения статов/статуса | сохранённый **potency** (из броска potency) |
| `<shield>` — вес магического щита | `saving=` / `resist=` — контест на наложение |
| `<actions>` — всё, что эффект делает за свою жизнь | `battleflag=` — переопределение тайминга декремента (§5.6) |

Так два заклинания могут наложить *один* эффект с разной длительностью и силой,
но **поведение** эффекта задаётся один раз, в `affects.xml`.

> **Правило.** Если эффект — «число меняется, пока эффект активен» → это `<apply>`.
> Если это «что-то происходит, *когда* случается X» (тик, удар, добивание,
> истечение, входящая атака) → это `<action>` с `<trigger>`.

## 3. Элемент `<affect>`

```xml
<affect id="kBurning" buff="N">
    <flags val="kAfSameTime|kAfDispellable"/>
    <apply location="kInitiative"><modifier .../></apply>   <!-- 0..N -->
    <shield weight="10"/>                                    <!-- опционально -->
    <actions> … </actions>                                   <!-- опционально -->
</affect>
```

### 3.1 Атрибуты заголовка

| атрибут | значение |
|---|---|
| `id` | имя из перечисления `EAffect` (персонаж) или `ERoomAffect` (комната). Обязателен. |
| `buff` | `Y` = полезный, `N` = вредный. Определяет классификацию для снятия/лечения (снятие магии убирает дебаффы, отмена — баффы) и насилие/дружелюбность повторного наложения. По умолчанию — вредный. |
| `dispel_mod` *(issue.random-noise-rework)* | Аддитивный модификатор эффекта к **порогу состязания при снятии**, в процентных пунктах (Руководство по заклинаниям §9.3). Отрицательный = **труднее** снять, положительный = легче. По умолчанию `0`. Критические баффы (огне/ледо/возд. щит, санктуарий, призм. аура, полёт, водное дыхание) поставляются с `dispel_mod="-35"` → ~15 % на снятие при равной компетентности против ~50 % обычного бафа. |

Эффект может существовать с **пустым телом** (`<affect id="kCommander"/>`) — как
чистая метка-флаг, которую проверяет другой код. Добавление данных это не ломает.

## 4. Пассивные слои: `<flags>`, `<apply>`, `<shield>`

### 4.1 `<flags val="A|B|…">` — биты поведения `EAffFlag`

Биты тайминга декремента решают, **когда убывает длительность** — это самое
важное, потому что заодно решает, *какие циклы триггеров видят эффект*:

| флаг | длительность убывает… |
|---|---|
| *(нет)* | раз в **игровой час** (обновление вне боя). |
| `kAfBattledec` | раз в **боевой раунд**, пока носитель сражается. |
| `kAfPulsedec` | раз в **пульс** эффектов (быстрый цикл мобов/коротких баффов). |
| `kAfSameTime` | в такт обработке боевого DoT (для урона-во-времени). |

Прочие частые биты:

| флаг | эффект |
|---|---|
| `kAfDispellable` | снимается развеиванием магии / отменой. |
| `kAfCurable` | снимается лечащими заклинаниями / первой помощью. |
| `kAfUpdateDuration` | повторное наложение обновляет длительность до **большей**. |
| `kAfAccumulateDuration` | повторное наложение **складывает** длительности. |
| `kAfUpdateMod` | повторное наложение сохраняет **больший** модификатор. |
| `kAfMaterialize` | эффект можно материализовать как врождённый бафф моба. |
| `kAfEntanglement` | категория «ограничение движения» (её проверяют приёмы работы ног). |

> **Мультиэкземплярность.** Заклинание, накладывающее эффект с несколькими
> `<apply>`, создаёт **несколько экземпляров** (по одному на apply) с общей
> длительностью, истекающих вместе. Это важно для триггеров `kExpired`/`kDeath` —
> см. §5.5.

### 4.2 `<apply>` — изменения статов/статуса, принадлежащие эффекту

```xml
<apply location="kInitiative">
    <modifier min="0.0" dices_weight="0.0" alpha="0" beta="0.9" factor="1" cap="0"/>
</apply>
```

`location` — это `EApply` (`kHitroll`, `kAc`, `kInitiative`, `kSavingReflex`, …).
`<modifier>` использует **ту же формулу option-2**, что и §8.9 Руководства по
заклинаниям:

```
raw = min + ceil(dices·dices_weight·(1 + alpha·C) + beta·C)      C = skill_coeff + stat_coeff
raw = min(raw, cap)                                              (только при cap > 0)
modifier = factor · raw                                          (factor = -1 для дебаффов)
```

**`C` (компетентность) даёт то, что накладывает эффект.** Реальное произнесение
заполняет `C` из своего броска potency (навык + стат). *Триггерное действие
эффекта* (§5) берёт `C` из **сохранённого potency сработавшего эффекта** — так
эффект может наложить другой эффект, масштабирующийся от силы первого (так
масштабируется хасте-за-добивание у вампиризма, §7). Плоский эффект использует
только `min` (`beta=0`).

> **Эффект владеет своими apply.** Если эффект задаёт `<apply>`, каждое
> накладывающее его заклинание использует *их* (собственные apply заклинания
> игнорируются). Давайте apply эффекту; на заклинании оставляйте лишь
> `<duration>`/`saving`/`resist`. Старые эффекты без `<apply>` по-прежнему
> откатываются к apply заклинания.

#### 4.2.1 Настройка модификаторов через компетентность *(правила `beta`)*

Величина модификатора должна расти от **компетентности кастера (навык + стат)**, а
не напрямую от уровня/ремортов. Модификаторы почти всегда плоские
(`dices_weight=0`), так что формула сводится к `min + ceil(beta·C)`, где `beta` —
аддитивный вес компетентности (`alpha` важен только при `dices_weight≠0`).

Перевод из СТАРОЙ формулы по уровню/реморту:

> **1 реморт ≈ +5 к кэпу навыка + +1 к базовому стату.** Для типовой potency-roll
> (`low=3 hi=1.25`, стат `threshold=22 weight=0.5`) компетентность растёт на
> `(5·1.25 + 1·0.5)/100 = 0.0675` за реморт. Значит модификатору, который должен
> расти на `k` за реморт, нужен `beta ≈ k / 0.0675`.

Два правила:

* **`min ≥ 0`.** Подбирайте `min`/`beta` так, чтобы нетренированный кастер (`C=0`)
  получал `min·factor` (0 или чистую базу) — никогда не позволяйте дебаффу с
  `factor="-1"` инвертироваться в бафф для слабого кастера.
* **Якорь типового кастера.** Возьмите `beta ≈ |цель при R12 макс-навык| / C_R12`
  (округляя *вниз*, чтобы `min≥0`), затем `min = |цель при R12| − ceil(C_R12·beta)`.
  Пример — `kBless` (старое `-5 - R/3` на kSavingStability): `beta=2.8`, `min=0`,
  `factor=-1` → −7 при R0, **−9 при R12** (точно), −11 при R24.

#### 4.2.2 Стакирование

`<modifier stack="N"/>` ограничивает число стаков одного apply на цели. При
`stack="1"` (по умолчанию) повторное наложение обновляет/отклоняется по флагам
`kAfUpdate…`. При `stack>1` повторное наложение **добавляет стак** (накапливая
модификатор до `N·mod`, продлевая длительность при
`kAfAccumulateDuration`/`kAfUpdateDuration`). При развеивании снимается **один
стак** (модификатор пропорционально уменьшается), а не весь эффект — эффект
исчезает лишь со снятием последнего стака. Так «накапливающийся яд / нарастающий
дебафф» становится чистыми данными.

#### 4.2.3 Случайные пулы apply

Среди соседних apply с флагом `random="true"` за одно наложение выбирается
**ровно один** (равновероятно). Безусловные apply срабатывают всегда; случайные
делят один пул. Пример `kFailure`: один гарантированный штраф морали + один
случайный штраф стата из шести.

### 4.3 `<shield weight="N"/>`

Для семейства магических щитов (огонь/лёд/воздух): относительный вес, с которым
именно этот щит выбирается перехватить входящий удар, когда у носителя их
несколько. Имеет смысл только вместе с действием-возмездием `kWardDamage` (§5.4).

## 5. `<actions>` — слой триггеров *(новое ядро)*

`<actions>` эффекта — упорядоченный список блоков `<action>`. Каждый несёт
`<trigger>`, задающий **когда** он срабатывает, и одну или несколько **проявлений
эффекта**, задающих **что** он делает. Это тот же словарь действий, что у
`<talent_actions>` заклинания (§3.8 Руководства по заклинаниям) — `<damage>`,
`<points>`, `<affects>`, `<unaffect>`, `<side_spell>`, `<manual>`, `<area>` — плюс
одно проявление только для эффектов, `<damage_change>` (§5.4).

```xml
<actions>
    <action target="kTarFightSelf" base="tag" tag="amount">
        <trigger val="kPostHit"/>
        <points extra="100"><heal min="1" beta="0.10" .../></points>
    </action>
</actions>
```

### 5.1 Набор триггеров (`<trigger val="…">`)

| триггер | срабатывает, когда… | типичное применение | актор / примечания |
|---|---|---|---|
| `kPulse` | каждый пульс эффектов, в бою и вне | DoT / HoT / периодический эффект | без актора |
| `kBattlePulse` | пульс эффектов, **только пока в комнате идёт бой** | боевой DoT | без актора |
| `kPreHit` | носитель наносит удар (замах, до урона) | наездник-на-замахе | событие: оружие, навык |
| `kPostHit` | удар разрешился (после урона) | вампиризм, прок-на-удар | событие: amount, оружие, навык, actor=жертва |
| `kKill` | носитель нанёс **смертельный** удар (прямой или через DoT) | награда за добивание | actor=жертва (мертва — **не** целить в неё); цели `kTarFightSelf`/`kTarGroup` |
| `kExpired` | таймер/заряды эффекта истекли | эффект-при-спадении | без внешнего актора; цели на себя/союзников |
| `kDispell` | эффект **принудительно** снят (развеивание/отмена) | ужалить развеивающего | actor = развеивающий (`kTarActor`) |
| `kDeath` | носитель умирает (до окончательной смерти) | предотвращение смерти | actor=убийца; `<trigger return="0"/>` + самолечение *предотвращают* смерть |
| `kPoints` | носителя **восстановил** внешний кастер | реакция на лечение | actor=лекарь; `category=` фильтрует kHeal/kMoves/kThirst/kFull |
| `kWardDamage` | **входящий** экземпляр урона на носителе | пассивное изменение урона (санктуарий, призма) | см. §5.4 |
| `kWardAttack` | носитель — **цель** входящей (магической) атаки | отражение / поглощение произнесения | срабатывает раз за произнесение на входном гейте |
| `kEnter` / `kEnterPC` / `kEnterNPC` | вход персонажа / игрока / NPC (эффекты комнат) | эффект-при-входе / печать | actor = вошедший (`kTarActor`); `return="0"` запрещает вход |
| `kPick`/`kUnlock`/`kOpen`/`kClose`/`kLock` | команды с дверью (эффекты выходов) | ловушки / запечатанные двери | `return="0"` запрещает действие |

### 5.2 Атрибуты `<action>`

| атрибут | значение |
|---|---|
| `target` | `EActionTarget` (§5.3), по умолчанию `kTarSame`. На кого ложится эффект действия. |
| `base` | как масштабируется величина: по умолчанию `kCompetence` (potency); `tag` = значение из события. |
| `tag` | при `base="tag"` — поле события (напр. `amount` = нанесённый/восстановленный объём). |

Податрибуты `<trigger>`: `val` (обязателен), `category` (фильтр kPoints),
`return` (`"0"` = блокировать вызвавшее действие — для `kDeath`, `kEnter*`,
дверных триггеров).

### 5.3 Цели (`EActionTarget`)

`kTarSame` (цели предыдущего действия), `kTarFightSelf` (носитель),
`kTarFightVict` (противник носителя), `kTarGroup` / `kTarFoes` (нужен `<area>`),
`kTarRandomFoe` / `kTarRandomAlly`, `kTarMinions` (очарованные последователи),
`kTarActor` (вызвавший событие — вошедший / развеивающий / лекарь),
`kTarMaster` (хозяин очарования носителя — для эффектов на стороне хозяина, как
kSoulLink), `kTarRoomThis` (сама комната).

### 5.4 Проявления эффекта

Внутри `<action>` — те же стадии, что у заклинания (§5–8 Руководства по
заклинаниям), плюс:

* `<damage>` — нанести урон (`<damage>` на `kPulse`/`kBattlePulse` — это **DoT**;
  отражение на `kWardAttack` наносит урон обратно атакующему). Дайте эффекту свои
  сообщения `kDamageToChar/Vict/Room` (§6), иначе DoT будет молчаливым.
* `<points><heal>` — восстановить HP/движение (`kPulse`-лечение — это **HoT**;
  лечение на `kPostHit` — вампиризм, масштабируется от `base="tag" tag="amount"`).
* `<affects>` — наложить другой эффект (см. §5.6 про контроль длительности/флагов).
* `<unaffect>` — развеять/вылечить (грамматика §9 Руководства по заклинаниям).
* `<side_spell id="…">` — произнести целое заклинание как вложенный эффект.
* `<manual name="…">` — зашитый обработчик (выбор — данные; тело — C++).
* `<damage_change>` — **только для эффектов, `kWardDamage`**: пассивно
  масштабировать/править входящий экземпляр урона. Структура:
  ```xml
  <damage_change>
      <conditions>
          <type val="kPhysDmg"/>              <!-- kPhysDmg / kMagicDmg -->
          <flags missing="kIgnoreSanct"/>     <!-- пропустить, если удар игнорирует щит -->
      </conditions>
      <variation min="50" max="50" factor="-1"/>   <!-- здесь -50% -->
  </damage_change>
  ```
  Подформы семейства щитов: `<retaliation>` (отразить % обратно атакующему),
  `<reflection>` / `<absorption>` (отскок / поглощение произнесения), `<percent>`,
  `<element>`. Ими перенесли старую зашитую логику санктуария/призмы/огнещита —
  копируйте существующий эффект-щит как шаблон.

### 5.5 `<charges max="N"/>` — ограниченные срабатывания

**Триггерный** эффект может нести `<charges max="N"/>` внутри своего наложения
`<affects>` или собственного блока: эффект снимается после `N` срабатываний,
независимо от длительности. `-1` (по умолчанию) — без ограничения (только по
длительности). DoT с `<charges max="6"/>` тикает ровно 6 раз. Сочетайте с
действием `kExpired`, чтобы отреагировать на исчерпание зарядов.

> **Дедупликация.** DoT на `kPulse` и триггеры жизненного цикла (`kExpired`)
> срабатывают **один раз на тип эффекта за проход**, даже если у эффекта несколько
> экземпляров (мультиэкземплярные эффекты). Описывайте действие при
> истечении/тике один раз — оно не сработает дважды.

### 5.6 Контроль длительности и флагов у накладывающего `<affects>`

Когда действие эффекта накладывает другой эффект, его блок `<affects>` управляет
новым экземпляром:

* `<duration base="…" skill_divisor="…" min="…" max="…" skill="kX"/>` — формула
  длительности (§8.3 Руководства по заклинаниям). Атрибут **`skill="kX"`**
  специфичен для действий эффекта: у триггерного действия нет заклинания-источника,
  поэтому оно называет навык, чей бонус-новичка масштабирует длительность
  (например, похмелье, масштабируемое от `kHangovering`). Без `skill=` триггерное
  действие не масштабирует длительность по навыку.
* `battleflag="kAfBattledec|…"` — добавить (OR) флаги тайминга декремента новому
  экземпляру. Так действие может выдать обычно многочасовой эффект как короткий,
  **измеряемый в боевых раундах**: при `kAfBattledec` `<duration>` читается как
  **сырые боевые раунды** (без пересчёта часы→тики для игрока). Так вампиризм
  выдаёт хасте-за-добивание (§7) на 5–6 раундов.

## 6. Сообщения

Тексты эффектов лежат в `lib/cfg/messages/<lang>/affect_msg.xml` (и
`room_affect_msg.xml`), с ключами на каждый id эффекта. Движок читает ключи:

| семейство ключей | показывается, когда |
|---|---|
| `kAffImposedToChar/Vict/Room`, `kAffImposeFailToChar/Vict/Room` | эффект наложился / не наложился |
| `kAffExpiredToChar/Room` | эффект спал (kExpired) |
| `kAffDispelledToChar/Room` | эффект развеян (kDispell) |
| `kDamageToChar/Vict/Room` | сработало действие `<damage>` (дайте это DoT, иначе он молчит) |
| `kWardToChar/Vict/Room` | щит перехватил (kWardDamage/kWardAttack) |
| `kDeathToChar/Room` | сработало действие `kDeath` |
| `kTransformToChar/Vict/Room`, `kTransformCrit…` | флейвор трансформации/полиморфа |

Пустой ключ = стандартная строка движка (или тишина, для урона). Заполняйте хотя
бы наложение/спадение и любые ключи урона.

## 7. Эффекты комнат и выходов (`room_affects.xml`)

Эффекты комнат/выходов используют ту же грамматику
`<affect>`/`<flags>`/`<actions>`, но добавляют:

* **Триггеры входа и дверей** — `kEnter`/`kEnterPC`/`kEnterNPC` и
  `kPick`/`kUnlock`/`kOpen`/`kClose`/`kLock`. `<trigger return="0"/>` **запрещает**
  вход/действие с дверью (запечатанная комната, ловушка на замке). `kTarActor` =
  вошедший.
* **`<area>` + `<targets>` + `<distribution>`** — действие комнаты может задеть
  всех в комнате с затуханием по целям (грамматика §7 Руководства по заклинаниям).
* **`kService`-side_spell** — потиковый эффект комнаты часто выражается как
  `<side_spell>` к связанному сервисному заклинанию (`kTarRoomThis`).
* **`<target_conditions>`** — гейт действия по вошедшему/комнате (blocking/required).

## 8. Разобранные примеры (все в игре)

* **DoT — `kBurning`:** `kAfSameTime` + действие `kPulse`
  `<action target="kTarFightSelf">` с `<damage>`, масштабируемым `dices_weight` от
  сохранённого potency, плюс сообщения `kDamageTo*`. Тикает в бою и вне; ограничен
  `<charges>`.
* **Вампиризм + добивание — `kVampirism`:** `kPostHit`
  `<points><heal base="tag" tag="amount" beta="0.10">` (лечит 10% нанесённого
  урона) **и** `kKill` `<affects battleflag="kAfBattledec"><duration base="6">
  <affect id="kHaste"/>` — короткий боевой хасте в награду за добивание.
* **Бафф/дебафф стата — `kHaste`/`kSlowdown`:** один
  `<apply location="kInitiative">` (`beta=0.9`, `factor=+1`/`-1`) — масштаб от
  навыка кастера; пара складывается (итог = сумма), т.к. оба пишут в один `EApply`.
* **Щит — `kSanctuary`/`kFireShield`:** `kWardDamage` `<damage_change>` блокирует
  (−50% физ / +100% маг для санктуария; огнещит добавляет `<retaliation>`).
* **Цепочка при спадении — `kDrunked` → `kAbstinent`:** `kExpired`
  `<action target="kTarFightSelf"><affects><duration … skill="kHangovering"/>
  <affect id="kAbstinent"/>` — естественное протрезвление оставляет похмелье,
  масштабируемое по навыку (kAbstinent владеет своими apply на AC/HR/DR).
* **Эффект, изменённый способностью — `kSoulLink`:** *способность* патчит
  `kVampirism`, чтобы лечить ещё и хозяина миньона (`kTarMaster`). Патчинг на
  стороне способности — в [Руководстве по способностям](FEAT_MANUAL_RU.md); сам
  эффект не трогается.

## 10. Чек-лист для добавления / миграции эффекта

1. **Добавьте / найдите id `EAffect`** и его блок `<affect id= buff=>`.
2. **Пассивная часть → `<apply>`** (владейте apply; `<duration>` — на накладывающем).
3. **Выберите флаги декремента** (`kAfBattledec` для эффектов на боевые раунды;
   ничего — для многочасовых баффов) — это решает и то, какие циклы триггеров его
   видят.
4. **Поведенческая часть → `<actions>`**: выберите `<trigger>`, `target` и
   проявление (`<damage>` DoT / `<points>` HoT / `<affects>` наложение /
   `<damage_change>` щит). Масштабируйте через `base="tag"` или potency (`beta`).
5. **Сообщения** — хотя бы наложение/спадение плюс `kDamageTo*` для любого DoT.
6. **Эффект комнаты?** поместите в `room_affects.xml` и рассмотрите триггеры
   входа/дверей.
7. **Уберите зашитый C++**, который эффект заменяет (оставьте id как метку, если
   другой код всё ещё проверяет флаг).
8. **Проверьте** в mud-sim: наложите, протикайте/триггерните, подтвердите эффект +
   сообщения (учтите: sim прокручивает циклы боевых раундов, но не почасовое
   обновление — используйте `kAfBattledec`, чтобы наблюдать триггеры по истечению).
