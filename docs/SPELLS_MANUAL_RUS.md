
# Bylins MUD - Справочник заклинаний на основе данных

*Для игровых дизайнеров, создающих заклинания в файле `lib/cfg/spells.xml` и сопутствующем файле
`lib/cfg/spell_msg.xml`.*

В этом руководстве каталогизированы все теги, атрибуты и значения, которые сегодня распознает система заклинаний на основе данных
на сегодняшний день. После его прочтения вы сможете добавить новое
заклинание, настроить существующее или скомпоновать любой из механизмов - нанесение урона, исцеление,
влияние, снятие, накопление, ограничение по выравниванию, отражение - не прибегая к
C++.

Для ознакомления с мотивацией и историей изменений см. журнал коммитов на `sventovit.work`.
Этот документ сосредоточен исключительно на том, что *вы, как дизайнер, можете настроить*.

---

## 1. Краткий обзор процесса заклинания

Когда игрок произносит заклинание, движок выполняет следующие этапы **в указанном
порядке** для каждой цели. Любой этап может прервать весь процесс заклинания.

1. **Предпосылки заклинания** - мана, положение, сон, бой и т. д.
2. **Ограничения цели** (`<blocking>`, `<required>`, `<caster_blocking>`) -
   проверки на уровне действий, основанные на данных. Неудача незаметно прерывает заклинание на этой цели
   (групповые/массовые заклинания) или с сообщением <без эффекта> (одиночная цель).
3. **Отражение** (`<reflection>`) - если цель имеет флаг отражения
   или выравнивание, заклинание может отскочить обратно к заклинателю (один бросок на вероятность,
   охватывает все последующие этапы).
4. **`CastDamage`** - если установлен флаг `kMagDamage`.
5. **`CastUnaffects`** - если установлен флаг `kMagUnaffects`. Выполняется *до* применения эффекта, так что
   заклинание может рассеять существующее воздействие, а затем применить своё собственное.
6. **`CastAffect`** - если установлен флаг `kMagAffects`.
7. **`CastToPoints`** - если установлен флаг `kMagPoints` (восстановление HP/MV).
8. **`CastToAlterObjs`** - если установлен флаг `kMagAlterObjs` (изменение предмета).
9. **`CastSummon`** - если установлен флаг `kMagSummons` (оживление мертвых, клонирование и т. д.).
10. **`CastCreation`** - если установлен флаг `kMagCreations` (создание еды/света).
11. **`CastManual`** - если установлен флаг `kMagManual` (ручно написанный обработчик в
    `spells.cpp`).
12. **Реакция жертвы** - враждебность NPC / автоматическое применение заклинания обнаружения.

Этапы 4-11 управляются набором битов заклинания `<flags val="kMag:">`, поэтому одно
заклинание может запускать любое их подмножество в указанном выше порядке. Каждый этап может возвращать
код <break>, который завершает цепочку.

## 2. Структура файлов

### 2.1 `lib/cfg/spells.xml`

Определяет сами заклинания. Элемент верхнего уровня - `<spells>`. Каждое заклинание является
один дочерний элемент `<spell>`. Идентификаторы заклинаний берутся из перечисления `ESpell`
(`src/gameplay/magic/spells_constants.h`).

```xml
<spells>
    <spell id="kFireball" element="kFire" mode="kEnabled">
        <name rus="огненный шар" eng="fireball"/>
        <misc pos="kFight" violent="Y" danger="1"/>
        <mana max="80" min="60" change="1"/>
        <цели val="kTarCharRoom|kTarFightVict"/>
        <flags val="kMagDamage|kNpcDamagePc"/>
        <potency_roll> : </potency_roll>
        <success_roll> : </success_roll>          <!-- опционально -->
        <talent_actions>
            <action>
                <!-- gates: blocking / required / caster_blocking / reflection -->
                <!-- эффекты: урон / исцеление / область / влияет / не влияет -->
            </action>
        </talent_actions>
    </заклинание>
    :
</заклинания>
```

Кодировка - **KOI8-R**. Отредактируйте с помощью `iconv`, если ваш редактор не поддерживает эту кодировку
.

> **Примечание по поводу XML-ридера.** Парсер `DataNode` Bylins *не поддерживает внутренний текст
> *не поддерживает* внутренний текст. Каждое значение должно находиться в атрибуте (`val=":"`, `min=":"`,
> и т. д.). Пробелы внутри тега игнорируются.

### 2.2 `lib/cfg/spell_msg.xml`

Строки сообщений для каждого заклинания, индексируемые по идентификатору заклинания и типу сообщения (`ESpellMsg`).
См. 13.

### 2.3 Предостережение о снимках <малого мира>

`./circle -d small` считывает данные из `build/<dir>/small/cfg/`, что представляет собой
**моментальный снимок** каталога `lib/cfg/`, сделанный один раз во время запуска `meson setup`. После редактирования
`lib/cfg/*.xml`, обновите снимку перед запуском:

```bash
for f in spells.xml spell_msg.xml hit_msg.xml; do
  cp "lib/cfg/$f" "build/<your-build>/small/cfg/$f"
done
```

В противном случае ваши изменения не будут сохранены, и вы можете увидеть ложные сообщения SYSERR с <неверной
константы> SYSERR от старых значений.

## 3. Элемент `<spell>`

### 3.1 Атрибуты заголовка

| Атрибут | Обязателен | Описание |
|---|---|---|
| `id` | да | значение перечисления `ESpell`, например `kFireball`. |
| `element` | нет | `EElement`: `kFire`, `kAir`, `kWater`, `kEarth`, `kLife`, `kDark`, `kLight`, `kMind`, `kUndefined`. Используется для расчета сопротивления/коэффициентов элементов. |
| `mode` | нет | `EItemMode`: `kEnabled` (по умолчанию), `kDisabled`, `kService` (заглушка только для движка). |

### 3.2 `<name>`

```xml
<name rus="огненный шар" eng="fireball"/>
```

| Атрибут | Описание |
|---|---|
| `rus` | Русское отображаемое имя. Используется в игре. |
| `eng` | Английское название. Используется в инструментах администратора и журналах. |

### 3.3 `<misc>`

```xml
<misc pos="kFight" violent="Y" danger="1"/>
```

| Атрибут | Описание |
|---|---|
| `pos` | Минимальное положение заклинателя (`EPosition`): `kSleep`, `kRest`, `kSit`, `kFight`, `kStand`. |
| `violent` | `Y` / `N` / `A`. См. ниже. |
| `danger` | Целочисленный рейтинг угрозы, используемый ИИ мобов / свитками / ценообразованием. |

**`violent` (issue.ambiguous-spells).** Три состояния:

| Значение | Поведение |
|---|---|
| `N` | Чисто полезное заклинание. Никогда не вызывает PK, ответные действия, сопротивление AR, отражение магическим зеркалом / звуковым барьером или рассеивание с ограничением по мощности. |
| `Y` | Всегда агрессивно. Применяются все правила PK + ответные действия + защитные щиты + соревнование по рассеиванию + проверка GET_AR. |
| `A` | **Неоднозначно.** Решается для каждой цели: полезно для себя / очарованного питомца (по цепочке хозяина) / товарища по группе / NPC против NPC, полностью насильственно (с полной ответственностью за PK) для постороннего. `kDispellMagic` является текущим носителем - рассеивание из руки союзника проходит без соревнования; рассеивание от постороннего вызывает проверку силы. |

Вспомогательная функция, определяющая по целям, - это
`SpellInfo::IsViolentAgainst(caster, target)`. Статический предикат `IsViolent()`
сохраняет семантику `bool` для обратной совместимости - он возвращает `true` только
для `kYes`, поэтому любое место вызова, которое не было перенесено, рассматривает `A` как
<полезный по умолчанию> (более безопасный вариант без PK). `A` отклоняется при загрузке
при сочетании с `<flags val="kMagRoom">` (SYSERR-ограничивает до `Y`):
окраски комнат не имеют отношений к целям, которые нужно разрешить.

### 3.4 `<mana>`

```xml
<mana min="60" max="80" change="1"/>
```

| Атрибут | Описание |
|---|---|
| `min` | Стоимость маны на максимальном практическом уровне. |
| `max` | Стоимость маны на самом низком практическом уровне. |
| `change` | Снижение стоимости за каждый уровень по мере повышения уровня заклинателя. |

### 3.5 `<targets val=":">`

Разделенный вертикальной чертой набор флагов `ETarget`. Выберите хотя бы один из флагов <where to
искать>, а также любые флаги проверки.

| Флаг | Значение |
|---|---|
| `kTarIgnore` | Отсутствие аргумента цели - заклинания по области/массовые заклинания, мировые заклинания. |
| `kTarCharRoom` | Персонаж в той же комнате. |
| `kTarCharWorld` | Персонаж в любой точке мира (призыв/телепортация). |
| `kTarFightSelf` | Во время боя может быть нацелено на себя. |
| `kTarFightVict` | Во время боя можно выбрать в качестве цели текущего противника. |
| `kTarSelfOnly` | *Только проверка:* допускается только себя. |
| `kTarNotSelf` | *Только для проверки:* себя выбирать нельзя. |
| `kTarObjInv` | Предмет в инвентаре заклинателя. |
| `kTarObjRoom` | Предмет, лежащий на полу в комнате. |
| `kTarObjWorld` | Предмет в любом месте. |
| `kTarObjEquip` | Предмет, который носит заклинатель. |
| `kTarRoomThis` | Сама текущая комната (заклинания комнаты). |
| `kTarRoomDir` | Соседняя комната по направлению. |
| `kTarRoomWorld` | Любая комната в мире. |
| `kTarAllyOnly` | *Только проверка:* только себя или товарища по группе. Используется с `kTarCharRoom`. |
| `kTarMinionsOnly` | *Только для проверки:* только один из собственных NPC-последователей заклинателя (хозяин == заклинатель). |

### 3.6 `<flags val=":">`

Разделенный вертикальной чертой набор битов флагов `EMagic`. Они **направляют заклинание через
этапы** (все, что начинается с `kMag:`, включает этап) и **передают
метаданные игрового процесса** (PK / агро / тип урона).

| Маршрутизация этапов | Эффект |
|---|---|
| `kMagDamage` | Запускает `CastDamage`. |
| `kMagAffects` | Запустить `CastAffect`. |
| `kMagUnaffects` | Запустить `CastUnaffects`. |
| `kMagPoints` | Запустить `CastToPoints` (восстановление HP/MV). |
| `kMagAlterObjs` | Запустить `CastToAlterObjs` (трансформация предмета). |
| `kMagSummons` | Запустить `CastSummon`. |
| `kMagCreations` | Запустить `CastCreation`. |
| `kMagManual` | Запустить обработчик, написанный вручную в `spells.cpp` (также охватывает заклинания перемещения, такие как телепортация, возвращение, портал, перемещение). |
| `kMagGroups` | Заклинание нацелено на группу. |
| `kMagMasses` | Массовое заклинание - все NPC в комнате (кроме союзников). |
| `kMagAreas` | Зональное заклинание - все враги в комнате с затуханием. |
| `kMagWarcry` | Помечает заклинание как боевой клич (обходит магическую защиту). |
| `kMagNeedControl` | Требует присутствия питомца заклинателя. |
| `kMagRoom` | Заклинание комнаты, находится в `magic_rooms.cpp`. |

| Класс PK / агро / урона | Эффект |
|---|---|
| `kNpcDamagePc` | Урон в PvE: NPC может наносить урон персонажу. |
| `kNpcDamagePcMinhp` | Урон ограничен, чтобы у персонажа осталось некоторое количество HP (без убийства). |
| `kNpcAffectPc` | NPC может наложить эффект на персонажа (PK-шлюз). |
| `kNpcAffectNpc` | NPC может наложить эффект на другого NPC. |
| `kNpcUnaffectNpc` | Зеркало для снятия эффектов. |

### 3.7 `<potency_roll>` и `<success_roll>`

Оба имеют одинаковую форму (`talents_actions::Roll`). Каждое заклинание имеет **один
бросок на силу**, который общий для всех его эффектов (величина урона, модификатор эффекта,
бонус к длительности, бонус за дополнительные попадания, сила снятия).

```xml
<potency_roll>
    <dices ndice="6" sdice="4" adice="10"/>
    <base_skill id="kFireMagic" low_skill_bonus="3" hi_skill_bonus="0.75"/>
    <base_stat id="kWis" threshold="22" weight="5"/>
</potency_roll>
```

| Дочерний элемент / атрибут | Описание |
|---|---|
| `<dices ndice= sdice= adice=>` | Выражение с кубиками: `ndice` кубиков с `sdice` гранями плюс `adice` плоских. Любой нулевой атрибут означает <нет вклада от этой части> - `ndice="0" sdice="0" adice="N"` надежно возвращает `N`, а отсутствующий блок `<dices>` эквивалентен `0,0,0` (возвращает 0). Отрицательные значения также рассматриваются как ноль. |
| `<base_skill id= low_skill_bonus= hi_skill_bonus=>` | Добавляет коэффициент, основанный на навыке. Ниже порога новичка (навык 75) каждый пункт даёт `low_skill_bonus`; выше 75 каждый пункт даёт `hi_skill_bonus`. Итоговый коэффициент = `(low_skill·low_bonus + hi_skill·hi_bonus) / 100`. Опустите дочерний элемент, чтобы пропустить вклад навыка. |
| `<base_stat id= threshold= weight=>` | Добавляет коэффициент, основанный на характеристике. Вклад = `max(0, stat ? threshold) · weight / 100`. Опустите дочерний элемент, чтобы пропустить вклад характеристики. |

`<success_roll>` анализируется и вычисляется, но **еще не используется** никаким игровым
механика - это перспективный хук. Используйте ту же форму, если хотите, чтобы
заклинание несло в себе бросок на успех/попадание отдельно от его мощности.

`base_skill` из `<potency_roll>` также определяет продолжительность действия и
количество попаданий (см. 7 и 6.1).

#### Значение `cast_potency`

Везде в руководстве, где говорится о <мощности заклинания> - формуле
модификатора, сохраненной мощности эффекта, состязания по рассеиванию,
масштабирование урона / исцеления / дополнительных попаданий - использует один и тот же скаляр:

```
мощность_заклинания = бросок_кубиков_навыка + коэффициент_навыка + коэффициент_характеристики
```

Где:

* `RollSkillDices` = результат броска `<dices>` (один `ndice * sdice + adice`
  проба на каждое заклинание).
* `skill_coeff` = `(low_skill · low_skill_bonus + hi_skill · hi_skill_bonus) / 100`
  от `<base_skill>` (навык ограничен значением `kNoviceSkillThreshold = 75` для
  нижней части; все, что выше, идет в верхнюю часть).
* `stat_coeff` = `max(0, stat ? threshold) · weight / 100` от `<base_stat>`.

Заклинание, в котором пропущен какой-либо подблок, вносит вклад 0 с этой стороны. Заклинание с
пустым `<potency_roll>` выбрасывает фиксированное 0 - его эффекты приземляются с мощностью 0, его
развеивание всегда терпит неудачу в состязании (все равно применяется нижний порог удачи в 5 %).

### 3.8 `<talent_actions>`

Контейнер для одного или нескольких блоков `<action>`. Каждый `<action>` может объявлять любое
комбинацию:

* Блокировки: `<blocking>`, `<required>`, `<caster_blocking>`, `<reflection>`
* Эффектов: `<damage>`, `<points>`, `<area>`, `<affects>`, `<unaffect>`

Большинство заклинаний используют один `<action>`. Технически допускается использование нескольких действий
, но фреймворк обрабатывает только одно действие каждого типа эффекта, поэтому типичная
шаблон - это одно `<action>`, в котором находится все.

### 3.9 `<components>` - требования к заклинанию

```xml
<components>
    <verbal/>
    <weave/>
    <material any_of="1234|5678" cost="1"/>
</components>
```

Необязательный дочерний элемент `<spell>`. Каждый внутренний тег является маркером присутствия или
дескриптор компонента материала; отсутствие `<components>` означает <требований нет>.
Сегодня существует три вида:

| Тег | Атрибуты | Эффект |
|---|---|---|
| `<verbal/>` | нет | Для заклинания требуется произнести фразу. `kSilence` на заклинателе блокирует его (заклинатели, на которых наложено заклинание <Молчание>, все еще могут произносить невербальные заклинания, и `SaySpell` для них остается беззвучным). |
| `<weave/>` *(issue.weave-component)* | нет | Помечает действие как <настоящую магию>. Заклинание с плетением проваливается в комнате с `kNoMagic` - выдает `kCastForbiddenToChar` / `kCastForbiddenToRoom` точно так же, как это делал устаревший, управляемый данными `<blocking><room_flags val="kNoMagic"/>`. Заменяет 209 копий этого блокирующего блока - `<weave/>` теперь является единственным источником правды для определения <является ли это магией>. `<verbal/>` и `<weave/>` независимы (боевые крики являются вербальными, но **не** являются заклинаниями). |
| `<material .../>` | `any_of` / `all_of` / `where` / `cost` | Компонент, основанный на предметах. `any_of` / `all_of` представляют собой списки объектов и чисел, разделенные символом `\|`. `where` - это список `EFind`, разделенный символом `\|` (по умолчанию `kObjEquip\|kObjInventory\|kObjRoom` - сначала экипировка, затем инвентарь, затем комната). `cost` - расход заряда (по умолчанию 1; 0 = фокус только на присутствии; -1 = расходуется полностью). |

Исключение `MayCastInForbiddenRoom` (высшие боги, незачарованные NPC) обходит
как шлюз `<weave/>`, так и управляемый данными шлюз с флагом комнаты `<blocking>`
симметрично. Наборные для каждого заклинания `kCastForbiddenToChar` / `kCastForbiddenToRoom`
перекрывают нарратив kDefault.

## 4. Шлюзы целей

Все они находятся *внутри* `<talent_actions> ? <action>`, наряду с эффектами.
Они ограничивают **весь процесс применения** (каждый этап, каждый эффект), поэтому `<blocking>`,
, который срабатывает, прерывает нанесение урона *и* применение эффектов *и* исцеление для этой цели.

### 4.1 `<blocking>`

```xml
<blocking>
    <mob_flags val="kNoSleep"/>
    <affect_flags val="kHold|kSleep"/>
    <room_flags val="kNoMagic"/>
    <align val="kGood"/>
</blocking>
```

Заклинание отклоняется, если выполняется **любое одно** из перечисленных условий: флаг
флаг моба (только для NPC), флаг эффекта (любая цель), селектор ориентации
(выбор ориентации цели) или флаг комнаты (текущая комната CASTER). Каждая ось является
необязательна; объединяйте столько, сколько требуется для заклинания.

Используется форма с дочерними тегами, так как строки становятся длинными, когда
объединяются (например, блокировка `kNoMagic` теперь присутствует почти в каждом заклинании).
Старая форма атрибута (`<blocking mob_flags="..."/>`) была выведена из обращения.

| Дочерний тег | Значения |
|---|---|
| `<mob_flags val=":"/>` | Имена `EMobFlag`, разделенные символом `\|`. Распознаваемые: `kNoBlind`, `kNoSleep`, `kNoSilence`, `kNoHold`, `kNoBash`, `kNoCharm`, `kNoSummon`, `kNoFear`, `kCorpse`, `kResurrected`, `kMounting`, `kHelper`, `kClone`. |
| `<affect_flags val=":"/>` | Имена `EAffect`, разделенные символом `\|` (любой флаг эффекта, распознаваемый движком). |
| `<room_flags val=":"/>` | Имена `ERoomFlag`, разделенные символом `\|`. Распознаваемые: `kNoMagic`, `kPeaceful`, `kTunnel`, `kNoTeleportIn`, `kNoBattle`, `kGodsRoom`, `kForMono`, `kForPoly`. Проверяется по комнате заклинателя. |
| `<align val=":"/>` | `kGood`, `kEvil`, `kNeutral` - блокирует, если цель имеет данную ориентацию. |

При заклинании на одну цель блокировка со стороны цели посылает `kNoeffect`
заклинателя. При групповом/массовом заклинании блокировка со стороны цели не вызывает никаких сигналов для этой
цель (без спама по каждой цели). **Блокировка с флагом комнаты** работает иначе: она
срывает *все* заклинание до отправки сообщений по каждой цели и отправляет
`kCastForbiddenToChar` / `kCastForbiddenToRoom` (см. 15). Великие боги
и NPC, не находящиеся под чарами и не являющиеся опекунами, обходят блокировку флага комнаты
(`MayCastInForbiddenRoom`); заклинания боевого клича просто не используют `kNoMagic`
в качестве флага блокировки, поэтому для них такой обход не требуется.

### 4.2 `<required>`

```xml
<required>
    <mob_flags val="kCorpse"/>
    <affect_flags val="kCharmed"/>
    <align val="kEvil"/>
</required>
```

Такая же структура дочерних тегов, как у `<blocking>`, но заклинание отклоняется, если
цель **обладает всеми** перечисленными флагами/ориентацией. Используйте его для "только на
трупах> (оживить мертвых), <только на очарованных приспешниках>, <только на злых
существах> (рассеивание зла) и т. д. `<required>` в настоящее время не использует
`<room_flags>`.

### 4.3 `<caster_blocking>`

```xml
<caster_blocking>
    <caster align="kEvil" affect_flags="kHold|kCharmed"/>
</caster_blocking>
```

Отражает `<blocking>`, но проверяет **заклинателя** вместо жертвы.
Используется для отказа в заклинании, которое заклинатель не может использовать - например, `kDispelEvil`
блокируется, если сам заклинатель является злым. Всегда выдает `kNoeffect`
(без пропуска групповых/массовых заклинаний: блокировка со стороны заклинателя касается одного заклинателя,
а не цикл по целям).

**Схема** *(issue.caster-blocking-refine)*. В отличие от `<blocking>` / `<required>`
, которые используют форму с несколькими дочерними тегами (независимые дочерние элементы `<mob_flags val>` / `<affect_flags val>`
/ `<room_flags val>` / `<align val>`), `<caster_blocking>` сводит
свои оси в атрибуты одного дочернего элемента `<caster>`:

| Атрибут | По умолчанию | Описание |
|---|---|---|
| `align` | нет | `kGood` / `kEvil` / `kNeutral`. Блок срабатывает, когда заклинатель соответствует указанной ориентации (`IsGood` / `IsEvil` / `IsNeutral`). |
| `affect_flags` | нет | Список `EAffect`, разделенный символом `\|`. Блок срабатывает, когда заклинатель имеет **любой** из перечисленных флагов эффектов (например, `kHold`, чтобы удерживаемый заклинатель не мог активировать бафф, действующий только на себя). |

Оба атрибута являются опциональными и суммируются - любая совпадающая ось запускает
блок. Хранение данных совместно с `<blocking>` / `<required>` (та же
структура `FlagCondition`), поэтому внутренне данные попадают в `cond.align` и
`cond.affect_flags`; асимметрия схемы является преднамеренной (ограничение заклинателя
одна описательная запись вместо многоосевого И-соединения нестрогих условий).

### 4.4 `<reflection>` *(новая механика)*

```xml
<reflection affect_flags="kFireShield" align="kGood" prob="35"/>
```

Если исходная цель имеет любой из перечисленных флагов или соответствует
выравниванию, заклинание может **отскочить обратно к заклинателю** при броске на проценты
против `prob`. Заклинания, наложенные на себя, никогда не отскакивают.

| Атрибут | По умолчанию | Описание |
|---|---|---|
| `affect_flags` | (пусто) | Имена `EAffect`, разделенные символом `\|`. Любое совпадение запускает бросок на отражение. |
| `align` | `kAny` | `kGood`/`kEvil`/`kNeutral`. Соответствующая ориентация цели также запускает бросок. |
| `prob` | `20` | Процент вероятности того, что заклинание действительно отразится, когда флаг/выравнивание совпадают. |

Когда срабатывает отражение, выдаются три сообщения из `spell_msg.xml`:
`kReflectedToChar` (заклинателю), `kReflectedToVict` (первоначальной
цели), `kReflectedToRoom` (зрителям). После отражения заклинание
продолжает действовать на заклинателя - урон, эффект, задержка, все.

**Почему мы используем простой флаг + вероятность, а не соревнование по силе:** мобы и
объекты не имеют значения мощности, поэтому сравнение флага с мощностью
сегодня не имеет смысла. Регулятор `prob` - это рычаг разработчика для настройки
коэффициентов отражения для каждого заклинания.

## 5. `<damage>` - прямой урон

```xml
<damage saving="kReflex" prob="100">
    <amount min="0" dices_weight="1.0" competencies_weight="25"/>
    <hits skill_divisor="12" max="2" prob="100"/>
</damage>
```

| Атрибут / дочерний элемент | По умолчанию | Описание |
|---|---|---|
| `saving` | `kReflex` | `ESaving`: `kReflex`, `kStability`, `kWill`, `kCritical`, `kNone`. Успешная защита уменьшает урон вдвое. |
| `prob` | `100` | Процент вероятности того, что урон действительно будет нанесен (в противном случае происходит бесшумный промах). `prob<100` обходит RNG. |
| `<amount min= dices_weight= competencies_weight=>` | min=0, оба веса=1.0 | Итоговая сумма = `min + dices · dices_weight + (skill_coeff + stat_coeff) · competencies_weight`. Опустите тег, чтобы сохранить значения по умолчанию; пустой тег `<amount/>` подойдет. |
| `<hits skill_divisor= max= prob=>` | divisor=25, max=1, prob=20 | Поддержка многократных попаданий: `count = 1 + CalcExtraHits(...)`. Бонус за дополнительные попадания использует бросок на мощность заклинателя `base_skill`, масштабированный по формуле `min(skill, 75) / skill_divisor`, ограниченный значением `max`, срабатывающий с вероятностью `prob`%. `prob=0` означает равномерный случайный выбор между 0 и `extra`. Отсутствие тега означает одно попадание. |

## 6. `<points>` - восстановление: HP / ходы / жажда / полное

```xml
<points extra="N" prob="100">
    <heal   npc_coeff="3" min="0" dices_weight="1.0" competencies_weight="90"/>
    <moves                min="0" dices_weight="0.0" competencies_weight="20"/>
    <thirst               min="0" dices_weight="0.0" competencies_weight="0"/>
    <full                 min="0" вес_кубиков="0.0" вес_навыков="0"/>
</points>
```

Тег `<points>` (ранее `<heal>`, переименованный в issue.mag-points)
содержит до четырех необязательных внутренних тегов amount. Каждый из них независимо
восстанавливает (или уменьшает) одну категорию из того же броска на силу:

* **`<heal>`** - очки здоровья. Атрибут `extra="Y"` во внешнем теге `<points>` позволяет
  восстановлению превысить максимальный уровень ОЗ (перелечение). Имеет опциональный
  атрибут `npc_coeff` (по умолчанию `1.0`), который увеличивает величину, когда
  заклинатель является NPC - устаревшее поведение, предназначенное для того, чтобы исцеления, накладываемые на мобов,
  эффективным для миньонов.
* **`<moves>`** - очки движения. Такая же структура атрибутов, как у `<heal>`,
  за исключением `npc_coeff`.
* **`<thirst>`** - жажда (слот движка `THIRST`).
* **`<full>`** - сытость (слот движка `FULL`). Переименовано из `<cond>` в
  issue.point-bugs, чтобы освободить слово <cond> для общего
  триаду `{DRUNK, FULL, THIRST}`, а не только для голода.

| Внешний атрибут `<points>` | По умолчанию | Описание |
|---|---|---|
| `extra` | `N` | `Y` позволяет величине **исцеления** поднять HP выше максимального предела. Влияет только на `<heal>`; остальные категории уже насыщаются на своих естественных пределах. |
| `prob` | `100` | Процент вероятности срабатывания действия по всем очкам. Неудачный бросок восстанавливает ноль во всех четырёх категориях. |

| Внутренний атрибут количества | По умолчанию | Описание |
|---|---|---|
| `min` | `0` | Фиксированное значение. Может быть **отрицательным** - отрицательные числа истощают, а не восстанавливают (`<moves min="-30">` истощает 30 очков движения). |
| `dices_weight` | `0` | Вес кубиков при броске на силу. |
| `competencies_weight` | `0` | Вес на `(skill_coeff + stat_coeff)`. |
| `npc_coeff` (только `<heal>`) | `1.0` | Дополнительный мультипликативный бонус, когда заклинатель является NPC. По умолчанию `1.0` воспроизводит старое эффективное исцеление `*2` для мобов-заклинателей; установите `0`, чтобы отключить. |

### Условные обозначения для <жажда> / <сытость>

В движке поля <жажда>/<сытость> инвертированы - `0` означает полное насыщение,
`kMaxCondition` (48) - <голодание>. В XML используется знак, естественный для игрового процесса:

* **Положительное значение в XML** = восстановление (меньше жажды / меньший голод).
* **Отрицательное значение XML** = ухудшает состояние (усиливает жажду / голод).

`CastToPoints` инвертирует вычисленное значение перед вызовом
`gain_condition`, поэтому разработчики могут интерпретировать `<thirst min="50"/>` как
<восстановить 50 пунктов жажды>, не задумываясь об инвертированном поле движка
.

### Формула

Каждое значение, до ограничения:

```
количество = ceil(min + кубик · вес_кубика + компетенции · вес_компетенций)
сумма += сумма · (бонус к магической силе заклинателя / 100)
если заклинатель - NPC: сумма += сумма · npc_coeff   // 0 для категорий, не связанных с исцелением
```

Кубики / компетенции бросаются **один раз за каждое применение** из
`<potency_roll>` и используются для всех четырёх сумм - один навык
определяет все категории при одном заклинании.

### Описание по категориям (issue.point-bugs)

Каждая категория, отличная от нуля, выдает собственное сообщение - один `act()` на категорию
в порядке `heal ? moves ? thirst ? full`. Ключи:

| Ключ | Срабатывает, когда | Процент категории, передаваемый в `{intensity}` |
|---|---|---|
| `kHealToVict` | `<heal>` произвело ненулевое количество | `amount * 100 / max_hp` |
| `kMovesToVict` | `<moves>` произвело ненулевое количество | `amount * 100 / max_moves` (со знаком; отрицательное для истощений) |
| `kThirstToVict` | `<thirst>` дало ненулевое значение | улучшение: `removed * 100 / current`; ухудшение: `-(cond_after * 100) / kMaxCondition` |
| `kFullToVict` | `<full>` дало ненулевое значение | такая же форма, как у kThirstToVict |

Устаревший одноключевой `kPointsToVict` был выведен из обращения - пучок каждого заклинания
указывает нужные ключи по категориям (большинство заклинаний исцеления просто
переопределяют `kHealToVict`; заклинания, истощающие ресурсы, такие как будущее <exhaust>, будут
переопределяют `kMovesToVict`; и т. д.).

Таблица интенсивности (`lib/cfg/mechanics/points.xml`) содержит оценки `<improve>`
и `<degrade>` для каждой категории. При вычислении таблица просматривается
по убыванию и возвращает первый уровень `T`, такой, что `T ? percent` -
объединенный для обоих направлений. `<heal><degrade>` намеренно пуст
(по замыслу, восстановление может быть только положительным; отрицательное значение `<heal>` незаметно
не дает никакого эффекта). Пороги `<degrade>` для жажды/сытости являются **отрицательными**, поскольку
формула процента имеет знак.

## 7. `<area>` - масштабирование заклинаний по площади

```xml
<area>
    <decay cast="0.05" level="7" free_targets="3"/>
    <victims skill_divisor="15" dice_size="2" min_targets="5" max_targets="20"/>
</area>
```

`<decay>` контролирует, как эффективный уровень/стоимость заклинания уменьшается за каждую дополнительную
цель; `<victims>` контролирует, сколько целей выбирает заклинание.

| атрибут `<decay>` | Описание |
|---|---|
| `cast` | Коэффициент снижения, применяемый к стоимости на каждом этапе. |
| `level` | Уменьшение уровня заклинателя за каждую цель сверх `free_targets`. |
| `free_targets` | Первые N целей не требуют дополнительных затрат. |

| Атрибут `<victims>` | Описание |
|---|---|
| `skill_divisor` | Делитель навыка заклинателя, который масштабирует количество целей (?1). |
| `dice_size` | Количество граней кубика, определяющего случайное количество целей. |
| `min_targets` | Минимальное количество жертв. |
| `max_targets` | Максимальное значение количества жертв. |

Используется вместе с флагами `kMagAreas` / `kMagMasses`. `CalcTargetsQuantity =
min_targets + min(RollDices(skill / skill_divisor, dice_size), max_targets)`.

## 8. `<affects>` - применение магического эффекта

Это основной инструмент для баффов и дебаффов.

```xml
<affects type="kSleep" saving="kWill" resist="kMind" prob="100">
    <flags val="kAfBattledec|kAfDispellable|kAfCurable"/>
    <duration base="1" skill_divisor="15" min="1" max="6"/>
    <apply id="kSleep" location="kNone">
        <modifier min="0.0" вес_кубиков="0.0" вес_компетенций="0.0"
                  factor="1" stack="1"/>
    </apply>
    <reposition pos="kSleep" stop_fight="false"/>
    <lag base="2" bonus_divisor="-1"/>
</affects>
```

### 8.1 Атрибуты `<affects>`

| Атрибут | По умолчанию | Описание |
|---|---|---|
| `type` | обязателен | значение перечисления `ESpell` - **идентификатор эффекта**, записанный на цели. Используется в качестве ключа поиска для снятия/исцеления и для сообщений. |
| `saving` | `kReflex` | Спасение, которое жертва совершает, чтобы избежать воздействия. `kNone` означает отсутствие спасения. |
| `resist` | `kFire` | `EResist` - канал сопротивления, применяемый к продолжительности эффекта. |
| `prob` | `100` | Вероятность в процентах, с которой вообще срабатывает блок эффекта (в противном случае происходит незаметный промах). Параметры `<lag>` и `<reposition>` *зависят* от этой же вероятности - сбой также подавляет их. |
| `potency_weight` | `1.0` | *(issue.affects-potency-weight)* Масштаб для сохраненного значения `Affect::potency` (т. е. для `cast_potency`, а не для модификатора). Эффект приземляется на `cast_potency * potency_weight`; проверка на снятие в `DispelSucceeds` считывает это масштабированное значение обратно. **Пример использования:** заклинание с большим модификатором, где один и тот же `<potency_roll>` питает как формулу модификатора, *так и* сохраненную мощность, может стать неразрушаемым просто потому, что модификатор должен быть большим. `potency_weight="0.4"` позволяет автору развязать эти компоненты - сохранить сильный модификатор, а масштабировать только сохраненную мощность вниз до разрушаемого диапазона. Симметрично по духу с `<unaffect potency_weight=>` на стороне снятия заклинания (см. 9.1). |

### 8.2 `<flags val=":">` - биты `EAffFlag`

| Флаг | Значение |
|---|---|
| `kAfBattledec` | Продолжительность уменьшается за каждый раунд боя в дополнение к уменьшению за каждый тик. |
| `kAfDeadkeep` | Эффект сохраняется на трупе / после смерти. |
| `kAfPulsedec` | Уменьшение при каждом импульсе вместо каждого тика. |
| `kAfSameTime` | Одинаковая продолжительность для ПК и НПЦ (без преобразования тиков ПК). |
| `kAfUpdateDuration` | Повторное применение перезаписывает продолжительность (берется большее из старого и нового значений). |
| `kAfAccumulateDuration` | Повторное применение добавляется к существующей длительности. |
| `kAfUpdateMod` | Повторное применение перезаписывает модификатор. |
| `kAfDispellable` | **Подлежит снятию** (например, `kDispellMagic`). |
| `kAfCurable` | **Поддается исцелению** (например, `kRemovePoison`). |
| `kAfMustBeHandled` | Только для эффектов комнаты: этот эффект имеет обработчик периодических тиков в коде (например, тик яда kDeadlyFog, падение метеоритов kMeteorStorm, попытки захвата kBlackTentacles). Цикл эффектов комнаты вызывает `HandleRoomAffect` при каждом тике, когда этот бит установлен. Эффекты персонажей в настоящее время не используют этот флаг. |
| `kAfUnique` | Только для эффектов комнаты: перед наложением удалите любое предыдущее применение этого же заклинания этим же заклинателем. Используется kRuneLabel (<одна метка руны в мире на одного заклинателя>). |

`kAfDispellable` / `kAfCurable` являются **единственным достоверным источником информации** о том,
о том, может ли эффект быть снят с помощью данного снятия/исцеления. Если отключить оба,
эффект будет постоянным до естественного истечения срока действия.

`kAfMustBeHandled` и `kAfUnique` являются свойствами только эффектов комнаты;
влияния на персонажей в настоящее время их не используют. Они заменили член
`must_handled` и локальный булевый параметр `only_one` для каждого вызова, который
ранее находились в `CallMagicToRoom`.

### 8.3 `<duration>` (масштабируется по навыку)

```xml
<duration base="2" skill_divisor="5" min="0" max="0"/>
```

| Атрибут | Описание |
|---|---|
| `base` | Фиксированная базовая продолжительность (в часах, преобразованная в тики с учетом единицы измерения персонажа). |
| `skill_divisor` | Бонус за каждый час, равный `min(skill,75) / skill_divisor`, с использованием показателя мощности заклинания `base_skill`. `0` означает отсутствие масштабирования по навыку. |
| `min` / `max` | Ограничения в часах. `0` с любой стороны означает <нет ограничения с этой стороны> (соответствует прежнему поведению). |

Бонус к навыку ограничен порогом новичка (75), деленным на
`skill_divisor`, поэтому навыки монстров не могут бесконечно увеличивать продолжительность эффектов.

### 8.4 `<apply>` - одно изменение характеристики / статуса на каждый `<apply>`

```xml
<apply id="kPoisoned" location="kStr" random="false">
    <modifier min="2.0" вес_кубиков="0.0" вес_компетенций="0.0"
              factor="-1" stack="3"/>
</apply>
```

| Атрибут `<apply>` | Описание |
|---|---|
| `id` | Флаг `EAffect`, накладываемый этим применением (например, `kSleep`, `kPoisoned`, `kInvisible`). `kUndefined` для применения, влияющего только на характеристики, без флага. |
| `location` | `EApply`: место, куда попадает модификатор. `kNone` для применения, содержащего только флаг, `kStr`/`kDex`/:, `kHitroll`, `kSavingReflex`, `kResistDark`, `kManaRegen`, `kCastSuccess` и т. д. |
| `random` | `Y`/`N` (по умолчанию N). При значении Y это применение **не всегда применяется**: см. 8.7. |

| Атрибут `<modifier>` | По умолчанию | Описание |
|---|---|---|
| `min` | `0.0` | Нижняя граница величины модификатора. |
| `dices_weight` | `0.0` | Вес вклада кубиков в бросок на силу. |
| `competencies_weight` | `0.0` | Вес `(skill_coeff + stat_coeff)`. |
| `factor` | `1` | Коэффициент знака/масштаба. Используйте `-1` для дебаффов (штраф к силе, штраф к спасению и т. д.). |
| `cap` | `0` | Необязательная верхняя граница исходной величины **до** применения коэффициента. `0` (по умолчанию) = без ограничения. Используется `kForbidden` (cap=100, отражая СТАРЫЙ `MIN(100, :)`) и аурами стихий (cap=30, насыщение около R15). |
| `stack` | `1` | **Ограничение на накопление** - см. 8.5. |

Формула:

```
raw      = min + ceil(competencies · competencies_weight + dices · dices_weight)
если cap > 0: raw = min(raw, cap)            # опциональное ограничение перед коэффициентом
модификатор = коэффициент · raw
```

Ограничение ограничивает **величину raw**, поэтому дебафф с `factor="-1" cap="30"`
ограничен интервалом от `-30` до `-min`. Используйте `cap` для эффектов с <жестким ограничением>,
равновесие ранее обеспечивалось вызовами `std::min(:)` в коде.

### 8.5 Накопление *(новая механика)*

`stack` - это максимальное количество наслоений, которое одно применение может создать на одной
цели. При значении `stack="1"` (по умолчанию) повторное применение заклинания либо обновляет
или отказывается в повторном применении (в зависимости от флагов `kAfUpdate:` эффекта). При
`stack="3"`, повторное применение **добавляет стек** до 3, накапливая модификатор.

**Что накапливают стеки:**

* Модификатор увеличивается: стек 2 достигает `2·модификатор`, стек 3 - `3·модификатор` и т. д.
* Продолжительность также увеличивается, если это предусмотрено установленным флагом
  (`kAfAccumulateDuration` или `kAfUpdateDuration`).
* Когда эффект снимается (например, с помощью `kDispellMagic`), один стек
  *уменьшается* вместо полного исчезновения эффекта - при этом модификатор уменьшается
  пропорционально, количество слоев уменьшается на один, а эффект сохраняется. Только
  тогда, когда удаляется последний накопленный эффект, он исчезает.

Это позволяет <накапливать яд>, <накапливать потерю боевого духа> или <накапливать дебаффы>
описать исключительно с помощью XML.

Пример реализации - яд, который накапливается до 3 раз:

```xml
<affects type="kSlowPoison" saving="kCritical" resist="kDark">
    <flags val="kAfAccumulateDuration|kAfCurable"/>
    <duration base="0" skill_divisor="3" min="0" max="0"/>
    <apply id="kPoisoned" location="kStr">
        <modifier min="2.0" вес_кубиков="0.0" вес_компетенций="0.0"
                  factor="-1" stack="3"/>
    </apply>
    <apply id="kPoisoned" location="kPoison">
        <modifier min="30.0" вес_кубиков="0.0" вес_компетенций="0.0"
                  factor="1" stack="3"/>
    </apply>
</affects>
```

### 8.6 `<reposition>` - принудительное перемещение / остановка боя

```xml
<reposition pos="kSleep" stop_fight="false"/>
```

| Атрибут | По умолчанию | Описание |
|---|---|---|
| `pos` | (нет) | `EPosition`. Жертва перемещается в эту позицию при попадании эффекта. Опустите, чтобы пропустить изменение позиции, но сохранить `stop_fight`. |
| `stop_fight` | `N` | Если `Y`, все, кто сражается с жертвой, прекращают сражаться с ней (эффект <мирный>). |

Полезно для сна, оглушения, умиротворения, сбивания с ног - перемещает в данные то, что раньше было
жестко запрограммированные сдвиги после применения в данные.

### 8.7 `<lag>` - боевая задержка жертвы

```xml
<lag base="6" bonus_divisor="-1"/>
```

| Атрибут | Описание |
|---|---|
| `base` | Фиксированная задержка в раундах боя при попадании эффекта. |
| `bonus_divisor` | Если `> 0`, коэффициент низкого мастерства заклинателя добавляется в виде дополнительных раундов в виде `low_skill_coeff / bonus_divisor`. `? 0` означает фиксированную задержку (без масштабирования по мастерству). |

### 8.8 Пул случайного применения

Среди элементов-собратьев `<apply random="true">` внутри одного `<affects>` **именно
один** выбирается случайным образом при каждом заклинании (равномерная выборка из резервуара).
применения с флагом random и безусловные свободно смешиваются в одном `<affects>` -

Пример (из `kFailure`): одно гарантированное снижение морального духа + одно случайное снижение характеристик

    </apply>
    <apply id="kUndefinded" location="kCon" random="true">

### 8.9 Масштабирование модификаторов в зависимости от компетенции заклинателя

Философия новой системы заключается в том, что **величина модификатора растет вместе с
компетентностью заклинателя (навык + характеристика), а не с уровнем заклинателя или
**. Старые формулы для отдельных заклинаний, такие как `-1 - R/2` или `(L+R)/3`,
переформулированы с помощью `competencies_weight`.

Правило преобразования, используемое при переводе из СТАРЫХ формул:

> **1 реморт = +5 к максимальному значению навыка + +1 к базовому показателю**
>
> (У вновь созданного персонажа предел навыка равен 80, а базовый показатель = классу
> по умолчанию; каждое преобразование/перерождение повышает предел на 5 и
> базовый показатель на 1.)

Для стандартного броска на силу (`low=3 hi=1.25`,
`threshold=22 weight=0.5`) компетенции при каждом перерождении растут на
`(5·1,25 + 1·0,5)/100 = 0,0675`. Таким образом, модификатор, который должен увеличиваться на
`k` за каждый реморт, требует `competencies_weight ? k / 0.0675`.

**Правило min ? 0.** Выберите `min` и `competencies_weight` так, чтобы
необученный заклинатель (компетенции = 0) получал модификатор `min·factor` = 0
(или небольшой чистой базовой величиной). Это позволяет избежать ситуации, когда нетренированные заклинатели
случайно получать бафф с обратным знаком от заклинания дебаффа.

**Рекомендуемый алгоритм** (используется для большей части масштабирования `<modifier>`
в `spells.xml`):

```
cw_exact   = |target at R=12 max-skill| / competencies_at_R12_max
cw         = floor(cw_exact * 10) / 10        # округление в меньшую сторону, чтобы min ? 0
min        = |цель при R=12| - ceil(competencies_at_R12 * cw)
```

Это точно фиксирует типичного заклинателя на R=12 (максимальный уровень навыка 140, ключевой показатель
порог+12) точно, позволяет модификатору монотонно масштабироваться с
компетентностью в любом направлении, а также дает нетренированным заклинателям чистую
нулевой базовый уровень.

**Пример.** Старая формула `kBless` была `-5 - R/3` в
`kSavingStability`. С типовыми значениями potency_roll
(компетенции ? 3,12 при R=12 (максимальный навык)), `cw_exact = 9/3,12 ? 2,88`,
поэтому `cw = 2,8`, а `min = 9 - ceil(3,12·2,8) = 9 - 9 = 0`. В текущем
XML выглядит так:

```xml
<apply id="kBless" location="kSavingStability">
    <modifier min="0.0" dices_weight="0.0"
              competencies_weight="2.8" factor="-1"/>
</apply>
```

Проверка:
- Необученный заклинатель (компетенции = 0): модификатор = `-1·(0 + 0)` = 0.
- R=0 максимальный навык (компетенции ? 2,31): модификатор = `-1·ceil(6,47)` = ?7.
- R=12 типичный (компетенции ? 3,12): модификатор = `-1·ceil(8,74)` = **?9** ?
- R=24, максимальный (компетенции ? 3,93): модификатор = `-1·ceil(11,0)` = ?11.

Наклон немного пологий, чем у OLD, равный `-1/3` на реморт, но R=12
точно совпадает, и компетентность определяет значение от начала до конца.

**Когда `min` оказывается отрицательным.** При агрессивных бросках на потенциал (большие
`low_skill_bonus`/`hi_skill_bonus` или большим весом `weight`),
вышеуказанный алгоритм может дать `min < 0`. *Избегайте этого.* Отрицательное значение `min`
означает, что модификатор нетренированного заклинателя становится `-1·(отрицательное)` = положительным
- что меняет знак заклинания на противоположный. Например, дебафф с `min=-7
factor=-1` дает нетренированному магу *бафф* `+7` на спасения врага. Исправить можно:

- уменьшите `competencies_weight`, пока `min ? 0` (более плавное масштабирование), или
- приведя бросок мощности заклинания к стандартной форме, чтобы
  вычисления получались без перебора.

У нескольких заклинаний в коде необычно агрессивные
потенциальные броски (`kPatronage`, `kEviless`, `kCurse`) и в настоящее время используют
модификаторы с фиксированным `min` без `competencies_weight`. Они остаются в таком
до тех пор, пока их броски_мощности не будут нормализованы.

## 9. `<unaffect>` - снимает / исцеляет *(новая механика)*

```xml
<unaffect affect_flags="kAfCurable" potency_weight="1.0" prob="100">
    <blocking      any_of="kGodsShield"
                   all_of=""/>
    <breaking      any_of="kSanctuary"/>
    <remove_anyway any_of="kQuestMark"/>
    <remove        all_of="kInvisible|kCamouflage|kHide"
                   breaking_by_failure="true"/>
</unaffect>
```

### 9.1 Атрибуты `<unaffect>`

| Атрибут | По умолчанию | Описание |
|---|---|---|
| `affect_flags` | `kAfCurable\|kAfDispellable` | **На какие эффекты может повлиять данное снятие.** Влияние подходит только в том случае, если оно несет хотя бы один из этих битов `EAffFlag`. Ограничьте до `kAfCurable` для исцелений (исцеления не снимают), `kAfDispellable` для `kDispellMagic` (не исцеляет от ядов) и т. д. |
| `potency_weight` | `1.0` | Коэффициент, умножающий силу заклинания снятия при сопоставлении с зафиксированной силой эффекта. См. 9.3. |
| `prob` | `100` | Вероятность в процентах, с которой вообще срабатывает блокировка эффекта (в противном случае происходит бесшумный пропуск). |

### 9.2 Четыре подблока

Каждый из четырех дочерних элементов содержит два списка `ESpell`:

| Блок | Когда проверяется | Эффект |
|---|---|---|
| `<blocking>` | Первый | Если присутствует какой-либо из перечисленных эффектов, **все записи `<remove>` пропускаются** (но `<remove_anyway>` все равно выполняется). Цепочка *не* разрывается. |
| `<breaking>` | Первый | Если присутствует какой-либо из перечисленных эффектов, **цепочка заклинаний разрывается** после этого этапа (последующие этапы не выполняются). |
| `<remove_anyway>` | Второй | Устраняет эффекты **независимо от блокировки**. Полезно для <всегда снимать отметку квеста, даже если она защищена>. |
| `<remove>` | Третий (только если не заблокировано) | Обычный список удаления - эти эффекты снимаются, когда `<blocking>` не сработало. |

Внутри каждого блока:

| Атрибут | Значение |
|---|---|
| `any_of` | Список `ESpell`, разделенный символом `\|`. Блок срабатывает, если присутствует **любой** из них (для блокировки/разрушения) или снимает **первый из присутствующих** (для блоков удаления). Односимвольное значение `"*"` является **подстановочным знаком** - см. 9.5. |
| `all_of` | Список `ESpell`, разделенный символом `\|`. Для блокировки/прерывания срабатывает только при одновременном наличии **всех** элементов. Для блоков удаления рассеивает **все** перечисленные эффекты, которые присутствуют. Односимвольное значение `"*"` является **подстановочным знаком** - см. 9.5. |
| `breaking_by_failure` *(только remove / remove_anyway)* | Если `true`, **неудачная проверка потенциала** при любом снятии эффекта из этого блока разрывает цепочку заклинания (последующие этапы не выполняются). По умолчанию `false` (неудачная проверка проходит незаметно). |

Смешивание `"*"` с явными именами (например, `any_of="*\|kPoison"`) отклоняется
во время разбора с выводом предупреждения - явные имена являются излишними, если
имеется подстановочный знак, и семантика является неоднозначной. Используйте либо одно, либо другое.

### 9.3 Рассеивание с проверкой мощности *(новая механика)*

Каждый эффект, наложенный с помощью `CastAffect`, **записывает силу заклинания** (
значение броска в момент наложения) и флаг дебаффа (соответствует
`violent` заклинания). Когда `CastUnaffects` пытается удалить эффект,
удаление зависит от состязания по силе.

**Математика**, по сравнению. Обе стороны используют одну и ту же формулу `cast_potency`
(3.7); единственный параметр - это `potency_weight` каждой стороны:

```
stored_potency = cast_potency(affect_spell, caster_at_impose) · <affects potency_weight>     (время наложения, 8.1)
dispel_potency = cast_potency(dispel_spell, caster_at_dispel) · <unaffect potency_weight>    (contest-time, 9.1)
```

Рассеивание удается, когда `dispel_potency > stored_potency`. Полный набор правил:

1. **Ненасильственное снятие + эффект, не являющийся дебаффом** (очищение баффа с помощью
   ненасильственным исцелением) - **проверка не проводится**, эффект всегда снимается. Это
   путь, позволяющий доброжелательному `kRemovePoison` снять собственные
   случайные баффы без сопротивления. Для неоднозначного (`A`) снятия, такого как
   `kDispellMagic`, предикат <ненасильственный> разрешается для каждой цели
   (3.3): от союзника снятие проходит; от постороннего оно
   в борьбу, описанную ниже.
2. **Фиксированный минимальный шанс удачи 5 %** - независимо от силы, у снятия заклинания всегда есть
   5 % шансов на успех. Это позволяет даже начинающему целителю оставаться эффективным против
   сильного воздействия.
3. **Соревнование по силе** - в противном случае `dispel_potency > stored_potency`.

Таким образом, яд высокоуровневого некроманта противостоит слабому исцелению новичка, но
лечение всё равно может победить благодаря 5 %-ному шансу. Два регулятора `potency_weight`
независимы - регулятор со стороны эффекта (issue.affects-potency-weight) позволяет
заклинаниям с большим модификатором оставаться поддающимися рассеиванию, записывая намеренно более слабую
сохраненную силу, чем предполагает исходный бросок; регулятор со стороны рассеивания
(issue.#3342) позволяет разработчику сделать конкретное заклинание исцеления сильнее или слабее
в отношении целевых эффектов.

**Пример.** kFireball с `<affects potency_weight="0.4">` и
`<potency_roll>`, равным 60, наносит эффект горения с `stored = 60 * 0.4 = 24`.
kDispellMagic с `<unaffect potency_weight="0.5">`, бросающий 50, соревнуется с
`dispel = 50 * 0.5 = 25`. Разоружение побеждает (25 > 24) - но лишь с небольшим перевесом; чуть
меньший бросок со стороны рассеивания привел бы к провалу состязания и необходимости полагаться на 5 %
.

Неудачная проверка силы выдает `kNoeffect` заклинателю (для чистого заклинания рассеивания
- заклинание `kMagAffects`, которое также рассеивает, при неудаче ничего не сообщает
и по-прежнему оказывает собственное воздействие).

### 9.4 Путь разрыва цепочки *(новая механика)*

Два способа разорвать цепочку заклинаний на этапе отсутствия эффекта:

1. **Разрыв при наличии эффекта** - `<breaking any_of=":">` находит имеющийся
   эффект; цепочка разрывается без попытки удаления.
2. **Разрыв при неудачном удалении** - `<remove : breaking_by_failure="true">`
   пытается рассеять, и проверка силы не проходит ни на одном из кандидатов.

Разорванная цепочка означает, что последующие этапы - аффект, урон, очки, изменение и т. д. -
**не выполняются**. Используйте это для шаблонов <сначала раскрыть, потом наложить дебафф>: если
жертва сопротивляется снятию невидимости, последующий дебафф
также должен пропустить.

### 9.5 Подстановочный знак `"*"` *(новая механика)*

`any_of="*"` и `all_of="*"` расширяют возможности `<remove>` / `<remove_anyway>` с
<фиксированного списка названий заклинаний> до <всего, что соответствует
фильтру `affect_flags`>. Эти два подстановочных знака имеют разную семантику -
соответствуя расширенной семантике явного списка:

| Шарговый символ | Выбор |
|---|---|
| `any_of="*"` | Выбрать **один** подходящий эффект, случайным образом. (Выборка осуществляется по методу резервуара из списка эффектов жертвы.) |
| `all_of="*"` | Поставить в очередь **все** подходящие аффекты жертвы для удаления. |

<Подходящий> означает `affect_matches(victim->aff, unaffect.affect_flags)` -
т. е. аффект содержит хотя бы один из битов `EAffFlag`, перечисленных в
окружающем атрибуте `<unaffect affect_flags=:>`. Затем каждый кандидат
, выбранный подстановочным знаком, затем проходит через то же правило
`DispelSucceeds` (9.3), и `breaking_by_failure` работает так же
, как и для явных списков.

**Используйте это для:**
- общих заклинаний типа <снять одно случайное магическое действие> / <излечить все яды>
  (само `kDispellMagic` теперь выражается именно так - см. ниже),
- будущие сфероспецифические рассеивания: помечайте эффекты новым битом `EAffFlag`
  (например, `kAfSorcerySphere`, `kAfHazeSphere`) и напишите заклинание снятия
  с помощью `<unaffect affect_flags="kAfHazeSphere"><remove all_of="*"/>`
  , чтобы удалить все эффекты сферы тумана.

**Пример миграции `kDispellMagic`:**

```xml
<spell id="kDispellMagic" element="kLight" mode="kEnabled">
    <!-- : обычные поля : -->
    <flags val="kMagUnaffects"/>
    <talent_actions>
        <action>
            <unaffect affect_flags="kAfDispellable">
                <remove any_of="*"/>
            </unaffect>
        </action>
    </talent_actions>
</spell>
```

Это заменило специальный хелпер `DispelRandomAffect()` в `magic.cpp`
(теперь удален) - путь выполнения теперь идентичен любому другому
рассеиванию, только с кандидатами, отобранными с помощью подстановочных знаков, вместо явного
списка заклинаний.

**Гипотетическое <лекарство от всех ядов> (с использованием подстановочного знака плюс более узкого
фильтр `affect_flags`):**

```xml
<spell id="kCureAllPoisons" element="kLife" mode="kEnabled">
    <!-- : обычные поля : -->
    <flags val="kMagUnaffects"/>
    <talent_actions>
        <action>
            <unaffect affect_flags="kAfCurable">
                <remove all_of="*"/>
            </unaffect>
        </action>
    </талант_действия>
</заклинание>
```

Каждый эффект, имеющий атрибут `kAfCurable` (в настоящее время это яды, а также любые будущие
<излечимые> эффекты) проходит проверку на силу; каждый из них снимается или отражается
независимо друг от друга.

### 9.6 Перекрестная ссылка: краткий обзор цепочки снятия эффектов

Для каждого заклинания, выполняющего `<unaffect>`:

1. Применяется фильтр `prob` (если пропущено, то пропускается без уведомления).
2. Оцените `<blocking>` и `<breaking>` (с поддержкой подстановочных знаков).
3. Сбор кандидатов из `<remove_anyway>` (всегда) и `<remove>`
   (только если не заблокировано) - явные списки или подстановочные знаки.
4. Если кандидаты существуют, выполните опциональную проверку действия PK, затем для
   кандидату: `DispelSucceeds` ? `RemoveAffectAndAnnounce` или
   запишите попытку, которой было оказано сопротивление.
5. Если кандидат `breaking_by_failure` встретил сопротивление или присутствовало условие `<breaking>`,
   , верните `kBreak`, чтобы остановить цепочку заклинания.
6. В противном случае верните `kSuccess`, и заклинание переходит к следующему этапу.

## 11. `CastToPoints`, `CastToAlterObjs`, `CastSummon`, `CastCreation`, `CastManual`

Эти этапы управляются флагом `kMag:` и запускают специальную логику в
`magic.cpp`. Они **частично управляются данными**:

* `CastToPoints` (HP / MV / жажда / полное восстановление) полностью управляется данными
  через блок заклинания `<points>` (issue.mag-points). Все четыре внутренних
  величины (`<heal>`, `<moves>`, `<thirst>`, `<full>`) используют один
  бросок на силу и один шлюз `prob`. Каждое ненулевое значение запускает
  **собственное** сообщение - по одному `act()` на каждую категорию в heal/moves/thirst/full
  порядок, с клавишами `kHealToVict` / `kMovesToVict` / `kThirstToVict` /
  `kFullToVict` (см. 6 <Нарратив по категориям>). Устаревшая одиночная
  `kPointsToVict` была удалена в issue.point-bugs.
* `CastToAlterObjs` запускает обработчик для каждого заклинания в коде (благословение, проклятие,
  невидимость, яд, зачарование, ремонт и т. д.) и считывает его сообщения из
  `spell_msg.xml` (`kAlterObjToChar`, `kEnchantNotWeapon`, :).
* `CastSummon` (оживить мертвого, клонировать, вызвать стража, :) - это в основном код, с
  сообщениями в `spell_msg.xml` (`kSummonToRoom`, `kSummonFail`,
  `kSummonNoCorpse` и т. д.).
* `CastCreation`, `CastManual` - это этапы, состоящие исключительно из кода.

Настраиваемые на основе данных параметры для этих этапов: `<flags>`,
`<targets>`, `<misc>`, `<mana>` и сообщения заклинания.

### 11.5 Влияние на комнату (`kMagRoom`) - тот же блок `<affects>`, **продолжительность сырых импульсов**

Заклинания комнаты (флаг `kMagRoom`, вызываемый через `CallMagicToRoom` в
`magic_rooms.cpp`) используют **ту же схему `<affects>`, что и эффекты персонажей
(8). Читатель извлекает продолжительность, флаг боя и модификатор из
`talent.GetAffect()` и собирает из них `Affect<ERoomApply>`.
есть два отличия от семантики эффектов персонажей:

1. **Единицей измерения продолжительности являются ИСХОДНЫЕ ИМПУЛЬСЫ, а не часы.** Для эффектов комнаты
   используется значение `talent.GetDurationBase() + CalcNoviceSkillBonus(...)`
   без множителя `kSecsPerMudHour / kSecsPerPlayerAffect`
   для персонажей-заклинателей. Почему: тики эффектов комнаты срабатывают за каждый импульс, и у некоторых
   длительных эффектов в помещении (kDeadlyFog=8, kMeteorStorm=3,
   kThunderstorm=7) составляют менее часа, что невозможно
   невозможно выразить целыми числами. Считыватель эффектов помещения соответствует существующему
   конвенции, ориентированной на импульсы.
2. **Перечисление флагов эффектов - `ERoomAffect`, а не `EAffect`.** Большинство
   эффектов в комнате сегодня не устанавливают флаг (kNone). Читатель заполняет
   `af[0].affect_type` из атрибута `id=` прикладного модуля, но
   парсер `<apply id=...>` в настоящее время типизирован под `EAffect`, поэтому
   вывод флага эффекта комнаты через XML еще не полностью налажен - установите
   `id="kUndefinded"` для apply в случае эффектов комнаты, если вы не уверены,
   , что хотите флаг, типизированный как char-affect.

**Биты Battleflag, имеющие значение для эффектов комнаты:**

| Флаг | Эффект |
|---|---|
| `kAfUpdateDuration` | Повторное применение собственного эффекта комнаты обновляет его продолжительность (`max(old, new)`). Без этого повторное применение к себе не дает никакого эффекта. |
| `kAfAccumulateDuration` | Повторное применение суммирует продолжительности вместо обновления. |
| `kAfMustBeHandled` | Эффект имеет обработчик кода на каждый тик в `HandleRoomAffect` (например, тик яда kDeadlyFog, падение метеоритов kMeteorStorm, попытки захвата kBlackTentacles). Цикл эффектов комнаты вызывает обработчик при каждом импульсе, когда этот бит установлен. |
| `kAfUnique` | Перед наложением удалите любое предыдущее применение этого же заклинания этим же заклинателем (эффект комнаты <только один в мире на заклинателя>, используется kRuneLabel). |

**Проверка компонентов материала является универсальной.** Любое заклинание комнаты, имеющее
настроенный компонент в `ProcessMatComponents` (в настоящее время
`kHypnoticPattern`, `kFascination`, `kEnchantWeapon`) прерывает заклинание
, если компонент отсутствует. Проверка выполняется в начале
`CallMagicToRoom` до чтения каких-либо данных о воздействии - в том же месте, что и
эквивалентной проверки в `CastAffect`. Заклинания без настроенного
компонентом попадают в ветвь по умолчанию в
`ProcessMatComponents`, и заклинание продолжается.

**Переопределения на стороне кода** по-прежнему требуются (это механики, которые
не может выразить):

| Переопределение | Где | Что делает |
|---|---|---|
| Модификатор мана-кастера | Случай `kForbidden` | `IS_MANA_CASTER(ch)` ? модификатор = 95 (константа), переопределяя формулу. |
| Ограничение модификатора | Случай `kForbidden` | Ограничение `min(100, ...)` (формула модификатора в XML может дать результат выше предела). |
| Сообщение об уровне модификатора | Случай `kForbidden` | Трехуровневое описание качества печати (>99 / >79 / остальные) зависит от модификатора во время выполнения. |
| Путь провала | Случай `kRuneLabel` | Проверка флага комнаты (`kPeaceful`/`kTunnel`/`kNoTeleportIn`) отменяет эффект и выдает описание, характерное для провала. |
| Блокировка из-за несовместимости комнаты | Случай `kBlackTentacles` | Флаги комнаты `kForMono` / `kForPoly` прерывают заклинание. |

**Создание нового заклинания комнаты:**

```xml
<spell id="kMyRoomSpell" element="kEarth" mode="kEnabled">
    <name rus="..." eng="..."/>
    <misc pos="kStand" violent="N" danger="1"/>
    <mana .../>
    <targets val="kTarIgnore"/>
    <flags val="kMagRoom"/>          <!-- это то, что проходит через CallMagicToRoom -->
    <potency_roll>
        <!-- Требуется только в том случае, если продолжительность или модификатор зависят от навыка. -->
        <base_skill id="kEarthMagic" low_skill_bonus="0" hi_skill_bonus="0"/>
    </potency_roll>
    <talent_actions>
        <action>
            <affects type="kMyRoomSpell" saving="kNone" resist="kVitality">
                <!-- Продолжительность в ИСХОДНЫХ ИМПУЛЬСАХ. -->
                <duration base="20" skill_divisor="5" min="0" max="0"/>
                <flags val="kAfMustBeHandled"/>
                <!-- Необязательный <apply> для модификатора; опускается для эффектов, использующих только флаги. -->
            </affects>
        </action>
    </talent_actions>
</spell>
```

Если заклинанию требуется логика на каждый такт (`kAfMustBeHandled`), добавьте случай в
`HandleRoomAffect`. Если требуется какое-либо из вышеупомянутых переопределений на стороне кода,
добавьте минимальный вариант в переключатель `CallMagicToRoom` - но сохраните все
простые параметры (продолжительность / флаг боя / формула модификатора) в
XML.

## 12. Сообщения (`lib/cfg/spell_msg.xml`)

Строки сообщений для каждого заклинания, индексируемые по идентификатору заклинания (sheaf) и типу `ESpellMsg`
. `id="kDefault"` - это общий набор, который используется в качестве резервного варианта для сообщений, если
конкретное заклинание не переопределило.

```xml
<spell_msg>
    <sheaf id="kDefault">
        <message type="kNoeffect" val="Заклинание не оказало никакого эффекта."/>
        <message type="kAffExpired" val="Магия, наложенная на вас, рассеялась."/>
        <message type="kCantCastPosition" val="Вам нужно встать."/>
        :
    </sheaf>
    <sheaf id="kFireball">
        <message type="kAffImposedToChar" val="Огонь обжигает вас!"/>
        <message type="kAffImposedToRoom" val="$n окутывает $N3 пламенем!"/>
        <message type="kAffDispelledToChar" val="Пламя гаснет."/>
        <message type="kAffExpired" val="Жар, охвативший вас, утихает."/>
    </sheaf>
    :
</spell_msg>
```

### Общие ключи сообщений

| Ключ | Когда выводится | Аудитория |
|---|---|---|
| `kNoeffect` | Общий результат <без эффекта>. | Заклинатель |
| `kAffImposedToChar` | Эффект от этого заклинания подействовал. | Пострадавший персонаж |
| `kAffImposedToRoom` | То же самое, для зрителей. | Комната |
| `kAffDispelledToChar` | Эффект **этого типа** был снят. Sheaf ориентируется на **удаленный** эффект, а не на заклинание снятия. | Исцеленный персонаж |
| `kAffDispelledToRoom` | То же самое, для наблюдателей. | Комната |
| `kAffExpired` | Эффект естественным образом истек на цели. | Пострадавший персонаж |
| `kAffDispelledToOwner` | Ваш эффект был снят с того, на кого вы его наложили. | Исходный заклинатель |
| `kHealToVict` | Заклинание `<heal>` нанесло ненулевое количество урона при применении к `<points>`. | Цель |
| `kMovesToVict` | `<moves>` произвело ненулевое количество в заклинании `<points>`. | Цель |

| `kFullToVict` | `<full>` произвело ненулевое количество в заклинании `<points>`. | Цель |
| `kReflectedToChar` | Отражение: заклинание отскочило обратно. | Заклинатель |
| `kReflectedToVict` | Отражение: цель, которая отразила. | Первоначальная жертва |
| `kReflectedToRoom` | Отражение: зрители. | Комната |
| `kCastForbiddenToChar` | Неудача из-за блокировки комнаты: комната заклинателя имеет флаг `<blocking><room_flags val>` (`kNoMagic` и т. д.). | Заклинатель |

| `kCantCastPosition`, `kCantCastSleeping`, : | Нарушения предварительных условий заклинания. | Заклинатель |
| `kCantCastNotAlly` | Ворота цели `kTarAllyOnly` (3.5) - жертва не является самим собой / товарищем по группе. | Заклинатель |

| `kNoTarget`, `kWrongTarget` | Аргумент цели заклинателя не найден (такой цели нет / тип цели отклонен `<targets val>`). | Заклинатель |
| `kItemNoPrototype`, `kItemCreatedToChar`, `kItemCreatedToRoom`, `kItemCreationFailToChar` | Результаты `CastCreation`. | Разное |
| `kSummonToRoom`, `kSummonFail`, `kSummonNoCorpse`, `kResurrectNoPower`, `kResurrectProtected` | Результаты призыва / воскрешения. | Разное |
| `kEnchantNotWeapon`, `kEnchantMagic`, `kEnchantSetItem`, `kEnchantMono`, `kEnchantPoly`, `kEnchantOther` | Результаты зачарования оружия. | Заклинатель |
| `kFightDeathToChar`, `kFightHitToChar`, `kFightMissToChar` (+ `ToVict`, `ToRoom`) | Сообщения об уроне в бою, индексируемые по `ESpell`. | Разное |
| `kCustomMsgOne` : `kCustomMsgFive` | Общие слоты для каждого заклинания, предназначенные для одноразовых строк, которые не заслуживают собственной константы перечисления. Используйте их для второй/третьей и т. д. строки, необходимых для конкретного заклинания (например, вторичное описание неудачи для kSummon). | Разное |

Сообщения хранятся **без** конечного `\r\n` - `act()` и
`SendMsgToChar` добавляют его самостоятельно.

При поиске сообщения сначала проверяется собственный набор заклинания, затем используется
`kDefault`. Исключения, о которых стоит знать:

* **Наложение** (`kAffImposedToChar` / `kAffImposedToRoom`) и **отражение**
  (`kReflectedToChar` / `kReflectedToVict` / `kReflectedToRoom`) ищутся
  ищутся напрямую в наборе **без перехода на kDefault** - отсутствующий ключ остается
  не выдает ошибку, а не возвращает неправильное значение по умолчанию.
* **Рассеивание нарратива** (`kAffDispelledToChar` / `kAffDispelledToRoom`)
  изначально также осуществлялся напрямую через пучок, но *(issue.dispel-default-msg)* теперь проходит
  через поиск с учетом kDefault. Слой kDefault несет общий
  резерв, поэтому каждое снятие заклинания выдает *что-то* - ранее большинство эффектов
  (kSanctuary / kBless / kArmor / kFly / и т. д.) удалялись без предупреждения
  , поскольку их собственные наборы никогда не создавали эти ключи. Переопределения для отдельных эффектов
  (kCurse, kPoison, kBlindness, kHold, kSilence, kFever, kDeafness,
  4 варианта яда, kPatronage) по-прежнему имеют приоритет, если присутствуют.
* `kCastForbiddenToChar` переключается на `kDefault` (каждое заклинание использует
  общую строку "Ваша магия потерпела неудачу:" по умолчанию), а
  `kCastForbiddenToRoom` рассматривается как опциональный - пустой результат означает
  <ничего не объявлять в комнате>, позволяя заклинаниям типа `kRuneLabel`
  переопределяться только тогда, когда им требуется собственное описание.

## 13. Рабочие примеры

### 13.1 Простое заклинание урона с дополнительными ударами

```xml
<spell id="kMagicMissile" element="kAir" mode="kEnabled">
    <name rus="волшебная стрела" eng="magic missile"/>
    <misc pos="kFight" violent="Y" danger="1"/>
    <mana max="40" min="30" change="1"/>
    <targets val="kTarCharRoom|kTarFightVict"/>
    <flags val="kMagDamage|kNpcDamagePc"/>
    <potency_roll>
        <dices ndice="6" sdice="4" adice="10"/>
        <base_skill id="kFireMagic" low_skill_bonus="3" hi_skill_bonus="0.75"/>
        <base_stat id="kWis" threshold="22" weight="5"/>
    </potency_roll>
    <talent_actions>
        <action>
            <damage saving="kReflex">
                <amount min="0" вес_кубиков="1.0" вес_компетенций="25"/>
                <попадания skill_divisor="12" max="2" prob="100"/>
            </damage>
        </action>
    </таланты_действий>
</заклинание>
```

* Спасительный бросок `kReflex`, при успехе урон уменьшается вдвое.
* Количество = `кубики · 1,0 + компетенции · 25`.
* До 2 дополнительных попаданий, срабатывающих при каждом заклинании (`prob=100`), с бонусом за дополнительное попадание
  бонус, масштабируемый как `min(kFireMagic, 75) / 12`.

### 13.2 Дебафф с броском на спасение, перемещением и задержкой

```xml
<spell id="kSleep" element="kMind" mode="kEnabled">
    <name rus="сон" eng="sleep"/>
    <misc pos="kFight" violent="Y" danger="0"/>
    <mana max="70" min="55" change="1"/>
    <targets val="kTarCharRoom|kTarFightVict"/>
    <flags val="kMagAffects|kNpcAffectPc"/>
    <potency_roll>
        <dices ndice="5" sdice="9" adice="12"/>
        <base_skill id="kMindMagic" low_skill_bonus="3" hi_skill_bonus="1.25"/>
        <base_stat id="kInt" threshold="22" weight="0.5"/>
    </potency_roll>
    <talent_actions>
        <action>
            <blocking>
                <mob_flags val="kNoSleep"/>
                <affect_flags val="kHold"/>
                <room_flags val="kNoMagic"/>
            </blocking>
            <affects type="kSleep" saving="kWill" resist="kMind">
                <reposition pos="kSleep"/>
                <duration base="1" skill_divisor="15" min="1" max="6"/>
                <flags val="kAfBattledec|kAfDispellable|kAfCurable"/>
                <apply id="kSleep" location="kNone">
                    <модификатор мин="0.0" вес_кубиков="0.0"
                              competencies_weight="0.0" factor="1"/>
                </apply>
            </affects>
        </action>
    </талант_действия>
</заклинание>
```

* Целями являются NPC с флагом `kNoSleep` или любой, кто находится под эффектом `kHold`
  обладают иммунитетом (затвор срабатывает до любого броска).
* Успешный бросок на спасение от `kWill` предотвращает сон.
* При успешном наложении заклинания жертва падает в положение `kSleep`.
* Продолжительность: `1 + min(kMindMagic, 75) / 15` часов, ограничено `[1..6]`.
* `kAfDispellable|kAfCurable` означает, что как `kDispellMagic`, так и
  лечения типа `kRemovePoison` могут попытаться снять эффект (при условии
  ).

### 13.3 Чистое исцеление (с ограничением по силе)

```xml
<spell id="kRemovePoison" element="kLife" mode="kEnabled">
    <name rus="лечить яд" eng="remove poison"/>
    <misc pos="kFight" violent="N" danger="0"/>
    <mana max="60" min="45" change="2"/>
    <targets val="kTarCharRoom|kTarFightSelf|kTarObjInv|kTarObjRoom"/>
    <flags val="kMagUnaffects|kMagAlterObjs|kNpcUnaffectNpc"/>
    <potency_roll>
        <dices ndice="5" sdice="9" adice="12"/>
        <base_skill id="kLifeMagic" low_skill_bonus="3" hi_skill_bonus="1.25"/>
        <base_stat id="kWis" threshold="22" weight="0.5"/>
    </potency_roll>
    <talent_actions>
        <action>
            <unaffect affect_flags="kAfCurable">
                <remove all_of="kPoison|kAconitumPoison|kScopolaPoison|kBelenaPoison|kDaturaPoison"/>
            </unaffect>
        </action>
    </talent_actions>
</spell>
```

* `affect_flags="kAfCurable"` - не затрагивает эффекты, которые являются только
  `kAfDispellable` (например, благословения).
* `<remove all_of=":">` - пробует удалить все перечисленные яды, присутствующие на цели.
* Каждое удаление - это состязание в силе. Заклинание `remove poison` начинающего священника
  может потерпеть неудачу против `яда` мастера-убийцы (высокая зарегистрированная сила),
  но всегда имеет 5 % шанса на успех.

### 13.4 Раскрытие с разрывом цепи

```xml
<spell id="kGlitterDust" element="kEarth" mode="kEnabled">
    <name rus="блестящая пыль" eng="glitterdust"/>
    <misc pos="kFight" violent="Y" danger="5"/>
    <mana max="120" min="100" change="3"/>
    <targets val="kTarIgnore"/>
    <flags val="kMagMasses|kMagAffects|kMagUnaffects|kNpcAffectPc"/>
    <potency_roll>
        <dices ndice="5" sdice="9" adice="12"/>
        <base_skill id="kEarthMagic" low_skill_bonus="3" hi_skill_bonus="1.25"/>
        <base_stat id="kWis" threshold="22" weight="0.5"/>
    </potency_roll>
    <talent_actions>
        <action>
            <area>
                <decay cast="0.05" level="3" free_targets="5"/>
                <victims skill_divisor="15" dice_size="3"
                         min_targets="5" max_targets="20"/>
            </area>
            <unaffect>
                <remove all_of="kInvisible|kCamouflage|kHide"
                        breaking_by_failure="true"/>
            </unaffect>
            <affects type="kGlitterDust" saving="kReflex" resist="kEarth">
                <flags val="kAfDispellable|kAfCurable"/>
                <duration base="4" skill_divisor="0" min="0" max="0"/>
                <apply id="kGlitterDust" location="kSavingReflex">
                    <modifier min="0.0" вес_кубиков="0.0"
                              competencies_weight="4.4" factor="1"/>
                </apply>
            </affects>
        </action>
    </талант_действия>
</заклинание>
```

* `<unaffect>` выполняется первым (по порядку цепочки): пытается снять
  `kInvisible|kCamouflage|kHide`.
* Если проверка эффективности **завершится неудачей** для любого из них на данной цели,
  `breaking_by_failure="true"` приводит к разрыву цепочки заклинаний **для этой
  цели** - дебафф <блестящая пыль> не накладывается.
* Если снятие заклинаний проходит успешно (или эффекты отсутствовали), дебафф
  накладывается как обычно - штраф к броску спасения на 4 часа.
* Штраф зависит от компетентности заклинателя: при компетентности ? 3,12
  (навык 140 / мудрость 34, типичный заклинатель с R=12) модификатор спасательного броска
  модификатор составляет `+ceil(3.12·4.4) = +14`; нетренированный заклинатель (навык 0,
  характеристика на пороге) получает `+ceil(0·4.4) = 0` - см. 8.9.

### 13.5 Заклинания с ограничением по выравниванию

```xml
<spell id="kDispelEvil" element="kLight" mode="kEnabled">
    <name rus="рассеять зло" eng="dispel evil"/>
    <misc pos="kFight" violent="Y" danger="2"/>
    <mana max="120" min="100" change="2"/>
    <targets val="kTarCharRoom|kTarFightVict"/>
    <flags val="kMagDamage|kNpcDamagePc"/>
    <potency_roll>
        <dices ndice="6" sdice="9" adice="20"/>
        <base_skill id="kLightMagic" low_skill_bonus="3" hi_skill_bonus="1.25"/>
        <base_stat id="kWis" threshold="22" weight="0.5"/>
    </potency_roll>
    <talent_actions>
        <action>
            <blocking>
                <room_flags val="kNoMagic"/>
            </blocking>
            <required>
                <align val="kEvil"/>
            </required>
            <блокирующий_заклинатель>
                <align val="kEvil"/>
            </блокирующий_заклинателя>
            <damage saving="kStability">
                <amount min="0" вес_кубиков="1.0" вес_компетенций="122"/>
            </damage>
        </action>
    </talent_actions>
</заклинание>
```

* `<required><align val="kEvil"/></required>` - *цель* должна быть злой, иначе `kNoeffect`.
* `<caster_blocking><align val="kEvil"/></caster_blocking>` - *заклинатель* не должен быть
  злым, иначе `kNoeffect` (старое поведение <гнева света>, теперь управляемое данными).
* `<blocking><room_flags val="kNoMagic"/></blocking>` - универсальный провал заклинания в комнате NOMAGIC
  добавлено ко всем заклинаниям, не относящимся к Warcry; выдает `kCastForbidden*` (см. 15).

### 13.6 Шаблон отражения

В настоящее время ни одно заклинание не использует `<reflection>` в рабочей версии (оно подключено и
готово). Вот как существо с огненным щитом может отражать холодные заклинания:

```xml
<spell id="kIceBolt" element="kWater" mode="kEnabled">
    <!-- : обычные поля : -->
    <talent_actions>
        <action>
            <reflection affect_flags="kFireShield|kFireAura" prob="35"/>
            <damage saving="kReflex">
                <amount min="0" вес_кубиков="1.0" вес_компетенций="80"/>
            </damage>
        </action>
    </talent_actions>
</заклинание>
```

* Цели, обладающие `kFireShield` или `kFireAura`, имеют **35 %** шанс при каждом
  заклинания отразить его обратно на заклинателя.
* Как только отражение срабатывает, заклинатель сам получает урон (с учетом
  с учетом всех своих защитных характеристик). Три сообщения
  (`kReflectedToChar`/`Vict`/`Room`) описывают отражение.
* Принадлежность к лагерю: добавьте `align="kEvil"` в тот же тег, чтобы нежить изначально
  отражали священные заклинания и т. д.

### 13.7 Накопление яда (гипотетически)

Настоящий `kPoison` сегодня не суммируется - этот пример иллюстрирует
механику. Фактический `kPoison` использует `competencies_weight` для масштабирования
урон от яда (без накопления); здесь мы добавляем `stack="3"`, чтобы показать
как был бы реализован вариант с накоплением.

```xml
<spell id="kPoison" element="kDark" mode="kEnabled">
    <!-- : обычные поля : -->
    <talent_actions>
        <action>
            <blocking affect_flags="kGodsShield"/>
            <affects type="kPoison" saving="kCritical" resist="kDark">
                <duration base="0" skill_divisor="3" min="0" max="0"/>
                <flags val="kAfSameTime|kAfAccumulateDuration|kAfCurable"/>
                <apply id="kPoisoned" location="kStr">
                    <modifier min="2.0" вес_кубиков="0.0"
                              вес_компетенций="0.0"
                              factor="-1" stack="3"/>
                </apply>
                <apply id="kPoisoned" location="kPoison">
                    <modifier min="0.0" вес_кубиков="0.0"
                              вес_компетенций="11.5"
                              factor="1" stack="3"/>
                </apply>
            </affects>
        </action>
    </талант_действия>
</заклинание>
```

* Повторное применение к той же цели **накапливается до 3 раз**: сила
  убывает на `-2`, `-4`, `-6` (фиксированное значение за каждый накопленный эффект), а
  урон от яда увеличивается в `1*`, `2*`, `3*` раз по сравнению со значением за одно применение
  (где значение за одно применение равно `ceil(компетенции·11,5)` - у типичного заклинателя с R=12
  типичный заклинатель ? 36 на наложение).
* Продолжительность накапливается (`kAfAccumulateDuration`).
* `kAfCurable` (без `kAfDispellable`) - `kRemovePoison` работает, а простой
  `kDispellMagic` - нет.
* Рассеивание `kRemovePoison` **удаляет один стек**, а не
  всё это - три средства, чтобы полностью нейтрализовать тройной яд
  (при условии, что каждое испытание на силу пройдет успешно).

## 15. Краткий чек-лист для добавления нового заклинания

1. **Выберите идентификатор `ESpell`.** Добавьте его в `spells_constants.h`, если он действительно
   новое (задача инженера). **Создаете прототип?** Вместо этого используйте один из зарезервированных
   `kTestOne : kTestFive` - см. 17 - чтобы экспериментировать с
   только с XML, не затрагивая код.
2. **Добавьте элемент `<spell>`** в файл `lib/cfg/spells.xml`. Обязательные атрибуты:
   заголовок, `<name>`, `<misc>`, `<mana>`, `<targets>`, `<flags>`.
3. **Выберите этапы**, указав флаги `kMag:` в
   `<flags val>`. Допускается несколько этапов.
4. **Создайте `<potency_roll>`** - как минимум один из `<dices>`, `<base_skill>`,
   `<base_stat>`. Без броска заклинание наносит урон нулевой силы
   .
5. **Напишите блок `<talent_actions>`** - сначала условия, затем эффекты.
6. **Добавьте сообщения для каждого заклинания** в `lib/cfg/spell_msg.xml`: как минимум
   сообщения о наложении / снятии / истечении, если заклинание добавляет эффекты, или
   сообщения с описанием урона, если оно наносит урон.
7. **Обновите снэпшот сборки** (см. 2.3) и запустите тест:
   `./circle -d small <port>` - ищите строки SYSERR, в которых упоминается
   идентификатор вашего заклинания или имена его тегов.

## 16. Что в данном руководстве **пока** не освещается

* **Ручно написанные обработчики** (заклинания `kMagManual`, `CastSummon`,
  `CastCreation`, `CastToAlterObjs` в переключателях) - они по-прежнему находятся в
  `spells.cpp` и `magic.cpp`. Их постепенный перенос в
  систему, управляемую данными, - работа в процессе.
* **Использование броска на успех** - `<success_roll>` анализируется и оценивается
  во время заклинания, но пока не подключен к принятию решения об успехе заклинания.
  Классическая, основанная на процентах функция `CalcCastSuccess` по-прежнему определяет успех.
* **Модификаторы, связанные с навыками** - такие способности, как `kMagicArrows`, которые усиливают
  определенные заклинания (например, удваивает дополнительные попадания `kMagicMissile`), по-прежнему
  жестко запрограммированы в `CalcExtraHits` и подобных вспомогательных функциях.

Если вам нужен какой-либо из них, создайте заявку, и мы сможем определить объем работ.

## 17. Тестовые константы заклинаний для прототипирования

Чтобы дизайнеры могли вносить изменения в XML нового заклинания **без перекомпиляции
движка**, постоянно зарезервировано пять слотов-заполнителей `ESpell`:

| Константа | id | Русское отображаемое имя |
|---|---|---|
| `kTestOne`   | 359 | <тест 1> |
| `kTestTwo`   | 360 | <тест 2> |
| `kTestThree` | 361 | <тест 3> |
| `kTestFour`  | 362 | <тест 4> |
| `kTestFive`  | 363 | <тест 5> |

Каждый слот поставляется в `lib/cfg/spells.xml` в виде минимального заглушки:

```xml
<spell id="kTestOne" mode="kTesting">
    <name rus="тест 1" eng="test 1"/>
</spell>
```

`mode="kTesting"` исключает слот из списков заклинаний классов в производственной среде
(он загружен, но не предоставлен), при этом позволяя бессмертному принудительно произнести
его для проверки.

**Рабочий процесс:**

1. Выберите свободный слот (`kTestOne` : `kTestFive`).
2. Отредактируйте его элемент `<spell>` в `spells.xml` - присвойте ему реальные
   `<misc>`, `<mana>`, `<targets>`, `<flags>`, `<potency_roll>` и
   блок `<talent_actions>`, которые вы хотите попробовать. По желанию замените
   русское название на рабочее название вашего прототипа (оставьте
   `mode="kTesting"`).
3. Добавьте сообщения для каждого заклинания в файл `spell_msg.xml` под тем же id.
4. Обновите снэпшот сборки (см. 2.3) и запустите тест:
   `./circle -d small <port>`, войдите в систему как бессмертный, принудительно примените его
   на манекене.
5. Проведите итерацию только по XML - пересбор сборки не требуется после обновления
   .
6. Когда прототип проходит проверку, инженер переименовывает константу +
   XML `id=` на окончательное имя (например, `kTestOne` ? `kFrostshield`),
   переключает режим в `kEnabled` и добавляет его в соответствующий список классов
   списков заклинаний. Это один проход по переименованию - вся работа по проектированию
   , уже выполненная для тестового слота, переносится дословно.

Сделайте тестовые слоты кратковременными: верните их к минимальному заглушке, как только ваш
прототип будет готов (или как только вы от него откажетесь), чтобы следующий разработчик
нашел их пустыми.
