# Bylins MUD — Руководство по конфигурации заклинаний

*Для гейм-дизайнеров, работающих с заклинаниями в `lib/cfg/spells.xml` и сопутствующем
файле `lib/cfg/spell_msg.xml`.*

В этом руководстве описаны все теги, атрибуты и значения, которые поддерживает
система заклинаний на основе данных. После прочтения вы сможете добавить новое
заклинание, настроить существующее или сочетать любые механики — урон, лечение,
эффекты, снятие эффектов, наложение/стак, ограничение по мировоззрению, отражение —
не прикасаясь к коду на C++.

История изменений и мотивация решений — в журнале коммитов на `sventovit.work`.
Данное руководство описывает исключительно то, *что вы, дизайнер, можете настраивать*.

---

## 1. Конвейер применения заклинания

Когда игрок произносит заклинание, движок выполняет следующие этапы **строго по
порядку** для каждой цели. Любой этап может прервать всю цепочку.

1. **Предусловия применения** — мана, позиция, сон, состояние боя и т.д.
2. **Шлюзы цели и кастера** (`<target_conditions>` на действие;
   `<caster_conditions>` на заклинание) — проверки `<blocking>`/`<required>` на
   уровне данных. Шлюз цели молча прерывает обработку данной цели (массовые/групповые
   заклинания) или выдаёт «нет эффекта» (одиночная цель); шлюз кастера или блок по
   комнате NOMAGIC прерывает всё применение.
3. **Отражение** (`<reflection>`) — если цель несёт флаг отражения или
   нужное мировоззрение, заклинание может отразиться обратно в кастера (один
   бросок вероятности, охватывает все последующие этапы).
4. **`CastDamage`** — если выставлен флаг `kMagDamage`.
5. **`CastUnaffects`** — если выставлен флаг `kMagUnaffects`. Выполняется *до*
   наложения эффекта, чтобы заклинание могло сначала снять старый эффект, а
   затем наложить свой.
6. **`CastAffect`** — если выставлен флаг `kMagAffects`.
7. **`CastToPoints`** — если выставлен флаг `kMagPoints` (восстановление HP/сил).
8. **`CastToAlterObjs`** — если выставлен флаг `kMagAlterObjs` (модификация предмета).
9. **Призыв** — если действие несёт `<summon>` (§3.8.5: хранитель / огненный
   защитник / клон); выполняется один раз на действие, не на каждую цель.
10. **`CastCreation`** — если выставлен флаг `kMagCreations` (создать еду/свет).
11. **`CastManual`** — если выставлен флаг `kMagManual` (обработчик в `spells.cpp`).
12. **Реакция жертвы** — враждебность NPC / автоприменение обнаружения.

Этапы 4–11 управляются битовым полем `<flags val="kMag…">` заклинания, так что
одно заклинание может запускать любое их подмножество в указанном порядке.
Каждый этап может вернуть код «прервать», завершающий цепочку.

Список выше — это то, что выполняется для **одного `<action>`** заклинания.
`<talent_actions>` заклинания — это **упорядоченный список** действий, у каждого
свой набор целей и свои стадии; движок выполняет их сверху вниз (действие-снаружи,
цель-внутри), а реакция жертвы (этап 12) срабатывает один раз, после **последнего**
действия. Одно-действенное заклинание ведёт себя ровно как список выше. См. §3.8 о
модели мультидействия и §13.8 — проработанный пример (`kSacrifice`).

## 2. Структура файлов

### 2.1 `lib/cfg/spells.xml`

Определяет сами заклинания. Корневой элемент — `<spells>`. Каждое заклинание —
один дочерний элемент `<spell>`. Идентификаторы заклинаний берутся из перечисления
`ESpell` (`src/gameplay/magic/spells_constants.h`).

```xml
<spells>
    <spell id="kFireball" element="kFire" mode="kEnabled">
        <name rus="огненный шар" eng="fireball"/>
        <misc pos="kFight" violent="Y" danger="1"/>
        <mana max="80" min="60" change="1"/>
        <targets val="kTarCharRoom|kTarFightVict"/>
        <flags val="kMagDamage|kNpcDamagePc"/>
        <potency_roll> … </potency_roll>
        <success_roll> … </success_roll>          <!-- необязательно -->
        <talent_actions>
            <action>
                <!-- шлюзы: target_conditions (blocking/required) / reflection -->
                <!-- эффекты: damage / heal / area / affects / unaffect -->
            </action>
        </talent_actions>
    </spell>
    …
</spells>
```

Кодировка — **KOI8-R**. При редактировании через редактор без поддержки KOI8-R
используйте `iconv`.

> **Примечание по XML-парсеру.** Парсер `DataNode` в Bylins **не поддерживает
> текстовые узлы**. Все значения должны находиться в атрибутах (`val="…"`, `min="…"` и
> т.д.). Пробелы внутри тега игнорируются.

### 2.2 `lib/cfg/spell_msg.xml`

Строки сообщений для каждого заклинания, ключи — id заклинания и тип сообщения
(`ESpellMsg`). См. §13.

### 2.3 Предостережение о снимке «малого мира»

`./circle -d small` читает данные из `build/<dir>/small/cfg/`, которая является
**снимком** `lib/cfg/`, сделанным один раз во время `meson setup`. После
редактирования `lib/cfg/*.xml` обновите снимок перед запуском:

```bash
for f in spells.xml spell_msg.xml hit_msg.xml; do
  cp "lib/cfg/$f" "build/<your-build>/small/cfg/$f"
done
```

Иначе ваши правки не загрузятся, и в логах могут появляться ложные ошибки
SYSERR «incorrect constant» от старых значений.

## 3. Элемент `<spell>`

### 3.1 Атрибуты заголовка

| Атрибут | Обязательный | Описание |
|---|---|---|
| `id` | да | Значение перечисления `ESpell`, например `kFireball`. |
| `element` | нет | `EElement`: `kFire`, `kAir`, `kWater`, `kEarth`, `kLife`, `kDark`, `kLight`, `kMind`, `kUndefined`. Используется в расчётах сопротивления и коэффициентов стихии. |
| `mode` | нет | `EItemMode`: `kEnabled` (по умолчанию), `kDisabled`, `kService` (заглушка только для движка). |

### 3.2 `<name>`

```xml
<name rus="огненный шар" eng="fireball"/>
```

| Атрибут | Описание |
|---|---|
| `rus` | Отображаемое имя на русском. Используется в игре. |
| `eng` | Имя на английском. Используется в инструментах администратора и логах. |

### 3.3 `<misc>`

```xml
<misc pos="kFight" violent="Y" danger="1"/>
```

| Атрибут | Описание |
|---|---|
| `pos` | Минимальная позиция кастера (`EPosition`): `kSleep`, `kRest`, `kSit`, `kFight`, `kStand`. |
| `violent` | `Y` / `N` / `A`. См. ниже. |
| `danger` | Целочисленный рейтинг угрозы, используемый ИИ мобов, свитками и ценообразованием. |

**`violent` (issue.ambiguous-spells).** Три состояния:

| Значение | Поведение |
|---|---|
| `N` | Чисто полезное заклинание. Никогда не вызывает ПК, ответные атаки, сопротивление AR, отражение магического зеркала / звукового барьера или состязание при снятии зависящего от силы эффекта. |
| `Y` | Всегда враждебное. Применяются правила ПК, ответные атаки, защитные броски, состязание при снятии эффекта и проверка GET_AR. |
| `A` | **Двусмысленное.** Разрешается отдельно для каждой цели: полезное для себя / очарованного питомца (по цепочке хозяев) / членов группы / NPC против NPC; полностью враждебное (с полной ответственностью за ПК) — для посторонних. `kDispellMagic` — первый носитель этого значения: снятие эффекта от союзника проходит без состязания; снятие от постороннего вызывает проверку силы. |

Вспомогательная функция `SpellInfo::IsViolentAgainst(caster, target)` принимает
решение для каждой цели во время выполнения. Статический предикат `IsViolent()`
сохраняет семантику `bool` для обратной совместимости — он возвращает `true`
только для `kYes`, поэтому участки кода без миграции воспринимают `A` как
«по умолчанию полезное» (более безопасный вариант без ПК). `A` отклоняется при
загрузке в сочетании с `<flags val="kMagRoom">` (SYSERR-зажим до `Y`): комнатные
заклинания не имеют отношения к конкретной цели для разрешения.

### 3.4 `<mana>`

```xml
<mana min="60" max="80" change="1"/>
```

| Атрибут | Описание |
|---|---|
| `min` | Стоимость маны на максимально достижимом уровне. |
| `max` | Стоимость маны на минимально возможном уровне. |
| `change` | Уменьшение за каждый уровень по мере роста кастера. |

### 3.5 `<targets val="…">`

Битовое поле флагов `ETarget`, разделённых `|`. Выберите хотя бы один флаг
«где искать» и любые нужные проверочные флаги.

| Флаг | Значение |
|---|---|
| `kTarIgnore` | Без аргумента цели — площадная/массовая атака, мировые заклинания. |
| `kTarCharRoom` | Персонаж в той же комнате. |
| `kTarCharWorld` | Персонаж в любой точке мира (призыв/телепортация). |
| `kTarFightSelf` | В бою можно нацелиться на себя. |
| `kTarFightVict` | В бою можно нацелиться на текущего противника. |
| `kTarSelfOnly` | *Только проверка:* разрешено только нацелиться на себя. |
| `kTarNotSelf` | *Только проверка:* нельзя нацелиться на себя. |
| `kTarObjInv` | Предмет в инвентаре кастера. |
| `kTarObjRoom` | Предмет на полу в комнате. |
| `kTarObjWorld` | Предмет в любой точке мира. |
| `kTarObjEquip` | Предмет, надетый кастером. |
| `kTarRoomThis` | Текущая комната (комнатные заклинания). |
| `kTarRoomDir` | Соседняя комната по направлению. |
| `kTarRoomWorld` | Произвольная комната в мире. |
| `kTarAllyOnly` | *Только проверка:* только себя или члена группы. Используется вместе с `kTarCharRoom`. |
| `kTarMinionsOnly` | *Только проверка:* только один из собственных последователей-NPC кастера (мастер == кастер). |

### 3.6 `<flags val="…">`

Битовое поле флагов `EMagic`, разделённых `|`. Они одновременно **направляют
заклинание через этапы** (всё, начинающееся с `kMag…`, включает соответствующий
этап) и **несут игровые метаданные** (ПК / агро / тип урона).

| Маршрутизация по этапам | Эффект |
|---|---|
| `kMagDamage` | Запустить `CastDamage`. |
| `kMagAffects` | Запустить `CastAffect`. |
| `kMagUnaffects` | Запустить `CastUnaffects`. |
| `kMagPoints` | Запустить `CastToPoints` (восстановление HP/сил). |
| `kMagAlterObjs` | Запустить `CastToAlterObjs` (преобразование предмета). |
| `kMagCreations` | Запустить `CastCreation`. |
| `kMagManual` | Запустить обработчик в `spells.cpp` (также охватывает заклинания перемещения: телепорт, возврат, портал, перемещение). |
| `kMagGroups` | Заклинание действует на группу. |
| `kMagMasses` | Массовая атака — каждый NPC в комнате (кроме союзников). |
| `kMagAreas` | Площадная атака — каждый враждебный в комнате с ослаблением. |
| `kMagWarcry` | Помечает заклинание как боевой клич (игнорирует магическую защиту). |
| `kMagNeedControl` | Требует присутствия питомца кастера. |
| `kMagRoom` | Комнатное заклинание, описано в `magic_rooms.cpp`. |

| ПК / агро / класс урона | Эффект |
|---|---|
| `kNpcDamagePc` | Урон в направлении PvE: NPC может нанести урон игроку. |
| `kNpcDamagePcMinhp` | Урон ограничен так, чтобы у игрока оставалось немного HP (не убивает). |
| `kNpcAffectPc` | NPC может наложить эффект на игрока (шлюз ПК). |
| `kNpcAffectNpc` | NPC может наложить эффект на другого NPC. |
| `kNpcUnaffectNpc` | Зеркальный флаг для снятия эффектов. |

### 3.7 `<potency_roll>` и `<success_roll>`

Оба имеют одинаковую структуру (`talents_actions::Roll`). Каждое заклинание
имеет **один бросок силы**, общий для всех его эффектов (величина урона,
модификатор эффекта, бонус к длительности, бонус к количеству ударов, сила снятия).

```xml
<potency_roll>
    <dices ndice="6" sdice="4" adice="10"/>
    <base_skill id="kFireMagic" low_skill_bonus="3" hi_skill_bonus="0.75"/>
    <base_stat id="kWis" threshold="22" weight="5"/>
</potency_roll>
```

| Дочерний элемент / атрибут | Описание |
|---|---|
| `<dices ndice= sdice= adice=>` | Выражение броска: `ndice` кубиков с `sdice` гранями плюс `adice` фиксированных. Нулевой атрибут означает «без вклада от этой части». `ndice="0" sdice="0" adice="N"` всегда возвращает `N`; отсутствие блока `<dices>` эквивалентно `0,0,0` (возвращает 0). Отрицательные значения трактуются как 0. |
| `<base_skill id= low_skill_bonus= hi_skill_bonus=>` | Добавляет коэффициент на основе навыка. Ниже порога новичка (навык 75) каждый пункт даёт `low_skill_bonus`; выше 75 — `hi_skill_bonus`. Итоговый коэффициент = `(low_skill·low_bonus + hi_skill·hi_bonus) / 100`. Пропустите элемент, чтобы исключить вклад навыка. |
| `<base_stat id= threshold= weight=>` | Добавляет коэффициент на основе характеристики. Вклад = `max(0, стат − threshold) · weight / 100`. Пропустите элемент, чтобы исключить вклад характеристики. |

`<success_roll>` разбирается и вычисляется, но **пока не используется** ни одной
игровой механикой — это задел на будущее. Применяйте ту же структуру, если хотите,
чтобы заклинание имело отдельный бросок успеха/попадания, независимый от силы.

`base_skill` из `<potency_roll>` также влияет на длительность эффекта и количество
дополнительных ударов (см. §7 и §6.1).

#### Компетентность `C`, величина эффекта и скаляр `cast_potency`

Бросок даёт три числа — `RollSkillDices` (случайная выборка кубиков), `skill_coeff`
и `stat_coeff`. Из них строятся **два разных** значения; не путайте их.

* `RollSkillDices` = результат броска `<dices>` (одна выборка `ndice × sdice + adice` за применение).
* `skill_coeff` = `(low_skill · low_skill_bonus + hi_skill · hi_skill_bonus) / 100`
  из `<base_skill>` (навык ограничен значением `kNoviceSkillThreshold = 75` для
  нижней части; всё выше идёт в верхнюю).
* `stat_coeff` = `max(0, стат − threshold) · weight / 100` из `<base_stat>`.

**Компетентность** — это часть навык+стат, в руководстве обозначается `C`:

```
C = skill_coeff + stat_coeff
```

**Величина эффекта** (урон, лечение/очки, модификатор эффекта) — это **не** простая
сумма броска и `C`. Используется формула option-2 (§5, §6, §8.4):

```
amount = min + кубики · dices_weight · (1 + alpha · C) + beta · C
```

Кубики (случайный бросок) **умножаются** на `(1 + alpha · C)`, поэтому вес броска
*растёт* с компетентностью, а не превращается в ничтожную долю большой аддитивной
суммы. `beta · C` — отдельный плоский член компетентности. В этом и весь смысл
переработки option-2: чисто аддитивное `бросок + C` делало бы бросок бессмысленным
при высоком навыке/стате — option-2 этого избегает. (Ставьте `alpha=0`, только если
вам намеренно нужно прежнее плоское поведение, например для плоских модификаторов
эффектов.)

**`cast_potency`** — это *отдельный* скаляр, и он **действительно** равен простой
аддитивной сумме:

```
cast_potency = RollSkillDices + skill_coeff + stat_coeff
```

Он используется **только как относительная величина силы**, никогда — как величина
эффекта: это сила, сохранённая на эффекте (`cast_potency × <affects potency_weight>`),
и состязание при снятии, которое затем пытается его удалить (§9.3). Аддитивность здесь
безвредна, потому что состязание сравнивает два применения, масштабирующихся одинаково.
Количество дополнительных ударов — третья, отдельная вещь: оно масштабируется как
`min(навык, 75) / skill_divisor`, а не от `cast_potency` (§5).

Отсутствующий под-блок вносит вклад 0. Заклинание с пустым `<potency_roll>` бросает
плоский 0 — его эффекты накладываются с силой 0, снятие всегда проигрывает
состязание (но 5 % удачи всё равно применяется), а любая величина сводится к своему `min`.

### 3.8 `<talent_actions>` — упорядоченный список действий *(мультидействие)*

Контейнер для **упорядоченного списка** блоков `<action>`. Каждый `<action>` —
это самостоятельное под-применение: он определяет **собственный** список целей и
запускает **собственные** стадии, а действия выполняются **сверху вниз**. Каждый
`<action>` может содержать любое сочетание:

* Шлюзов: `<target_conditions>` (обёртка для пер-действенных `<blocking>`/`<required>`),
  `<reflection>`. (Шлюз со стороны кастера — это `<caster_conditions>` на уровне заклинания, §4.3.)
* Эффектов: `<damage>`, `<points>`, `<area>`, `<affects>`, `<unaffect>`,
  `<side_spell>`, `<manual_cast>`.

Внутри одного действия стадии эффектов всегда выполняются в фиксированном порядке:

> **Damage → Unaffect → Affect → Points → SideSpell → Manual**

Большинство заклинаний по-прежнему используют один `<action>`, но теперь
заклинание может выстроить цепочку из нескольких — каждое нацелено на свой набор
целей и способно масштабироваться от результата предыдущего действия. См.
**§3.8.1–3.8.4** ниже и проработанный пример в **§13.8 (`kSacrifice`)**.

#### Как выполняется список действий (действие-снаружи, цель-внутри)

* **Действие 0 (*входное*)** использует **уровень заклинания** для области охвата —
  флаги `kMag…` выбирают список целей (`kMagAreas`/`kMagMasses` → враги,
  `kMagGroups` → союзники, иначе единственная выбранная цель). Оно выполняет полный
  конвейер по каждой цели: шлюзы цели, `<reflection>`, проверку бога, mtrigger
  применения **и** стадии эффектов, после чего запоминает каждую успешно
  затронутую цель для реакции.
* **Действия 1, 2, …** определяют свой список целей по атрибуту `target=`
  (§3.8.1) и выполняют **только** свои стадии эффектов плюс собственные
  `<blocking>`/`<required>`. Они **не** повторяют отражение, mtrigger или реакцию
  жертвы.
* **Реакция жертвы** (ответ NPC / авто-обнаружение) откладывается до завершения
  **всех** действий — всё заклинание это одно событие, поэтому дебаф в действии 1
  не может заставить цель ответить раньше, чем нанесёт урон действие 2.
* Если действие **убивает** цель (или снятие разрывает на ней цепочку),
  пропускаются только оставшиеся стадии **этого** действия на **этой** цели —
  остальной список действий и другие цели этого действия продолжаются.

Для одно-действенного заклинания выполняется только входное действие, ровно как
прежде, — механизм полностью обратно совместим.

#### 3.8.1 `target="…"` — выбор цели для действия

Не-входной `<action>` может нести атрибут `target` (`EActionTarget`, по умолчанию
`kTarSame`), задающий, **на кого** он действует:

| значение | что выбирает | нужен `<area>`? |
|---|---|---|
| `kTarSame` *(по умолч.)* | список целей **предыдущего** действия | наследует |
| `kTarFightSelf` | сам кастер | нет |
| `kTarFightVict` | текущий противник кастера | нет |
| `kTarGroup` | соратники в комнате | **да** |
| `kTarFoes` | все враги в комнате | **да** |
| `kTarRandomFoe` | один случайный враг в комнате | нет |
| `kTarRandomAlly` | один случайный союзник в комнате | нет |
| `kTarMinions` | очарованные NPC-последователи кастера (миньоны) в комнате | да (для >1) |
| `kTarRoomThis` | сама комната (`ctx.rvict`); действие накладывает свой эффект на комнату | нет |

`kTarGroup`/`kTarFoes`/`kTarMinions` охватывают нескольких персонажей, поэтому им
нужен блок `<area>` (§7), задающий сколько целей и с каким затуханием.

#### 3.8.2 `base="…"` и `reset` — масштабирование от результата прошлого действия *(cast-chain)*

По умолчанию каждое действие масштабирует свой эффект от **компетентности**
кастера (коэффициенты навыка и стата из броска силы). Не-входное действие может
вместо этого масштабироваться от **того, что произвело предыдущее действие**,
через `base` (`EActionBase`):

| значение `base` | чем заменяет скаляр компетентности |
|---|---|
| `kCompetence` *(по умолч.)* | реальная компетентность кастера (навык + стат) |
| `kDamage` | суммарный урон по HP, нанесённый за применение |
| `kPoints` | суммарно восстановленные очки (HP/силы/…) |
| `kAffects` | суммарная **сила** наложенных эффектов |
| `kDispelled` | суммарная **сила** снятых эффектов |

Аккумуляторы **суммируются по целям действия**. `base` заменяет только скаляр
компетентности в формуле — собственные кости и веса действия по-прежнему
применяются, поэтому для чистого переноса 1:1 проектировщик ставит
`dices_weight="0"` и `beta="1.0"` (см. A2/A3 `kSacrifice` в §13.8).
`reset="true"` обнуляет выбранный аккумулятор **после** выполнения этого
действия, чтобы более позднее действие начало подсчёт заново.

#### 3.8.3 `<side_spell id="…"/>` — запуск полного вложенного применения

```xml
<side_spell id="kHold"/>
```

Запускает **полное вложенное применение** другого заклинания на текущей цели — с
его *собственными* бросками, силой и конфигурацией эффектов. Допускается несколько
тегов `<side_spell>` (выполняются в порядке записи). Так массовое/групповое
заклинание становится тонкой обёрткой над своей одноцелевой базой: `kMassHold` —
это просто охват `<area>` плюс `<side_spell id="kHold"/>`, переиспользующий всё
определение `kHold` вместо его копирования. **Защита от циклов** отклоняет любое
заклинание, уже присутствующее в текущей цепочке применения, поэтому `A → B → A`
(и более глубокие петли) не могут уйти в бесконечную рекурсию. Вложенное
применение наследует пер-целевой коэффициент области, поэтому массовое затухание
доходит и до делегированного эффекта.

#### 3.8.4 `<manual_cast handler="…"/>` — ручной обработчик из данных

```xml
<manual_cast handler="SpellMentalShadow"/>
```

Называет обработчик, написанный вручную и зарегистрированный в `spells.cpp` (имя
должно совпадать с реестром, иначе при загрузке выводится `unknown <manual_cast>
handler`). Он выполняется как стадия **Manual** действия, поэтому ручной
обработчик теперь шлюзуется и упорядочивается как любая другая стадия, а не
вызывается лишь по старому флагу `kMagManual`.

#### 3.8.5 `<summon>` — призвать зачарованного прислужника *(issue.summon-pipeline)*

```xml
<summon base_fail="50" min_fail="0" handler="SetupKeeperStats">
    <mob vnum="-3021" competence_weight="16" keeper="Y"/>
</summon>
```

Создаёт моба **в комнате применяющего** и зачаровывает его на применяющего. Как и
действие над комнатой (§3.8.1 `kTarRoomThis`), `<summon>` выполняется **один раз на
действие — не на каждую цель**, поэтому диспетчеризуется на уровне действия (до
пер-целевого пути) и может питаться результатом предыдущего действия через `base=`
(§3.8.2): входное призывание читает реальную компетентность `C`, а сцепленное —
свой аккумулятор `base=` (например, прислужник, чья сила растёт с нанесённым уроном).
Это пришло на замену старому флагу `kMagSummons`, которого больше нет.

Атрибуты `<summon>`:

| Атрибут | Значение |
|---|---|
| `base_fail` | Базовый шанс провала, в процентах. |
| `min_fail` | Нижняя граница шанса провала (по умолчанию `0`). |
| `handler` | Необязательный именованный обработчик пост-призыва (см. ниже). |

Шанс провала равен `pfail = clamp(min_fail, base_fail − C·competence_weight, 100)`,
где `C` — компетентность действия (§3.7). Компетентный применяющий гонит провал к
`min_fail`; слабый — к `base_fail`. Способность `kFavorOfDarkness` перебрасывает
провал с шансом ¼ (срабатывает лишь 1 из 4). Бессмертные не проваливаются никогда.

Атрибуты `<mob>`:

| Атрибут | Значение |
|---|---|
| `vnum` | **Отрицательный** виртуальный номер для чтения (напр. `-3021` грузит моба `3021`). |
| `competence_weight` | Насколько сильно `C` снижает провал (тот самый `k` выше). Задавайте под ту шкалу, которую подаёт в действие `base=`. |
| `keeper` | `Y` → прислужник получает `EAffect::kHelper` + `ESkill::kRescue=100` (страж, спасающий хозяина). |

Прислужник зачаровывается (`ESpell::kCharm`) на длительность от мудрости + фазы
луны, его золото/опыт обнуляются, он размещается и становится последователем. Затем
`handler` выполняет специфичную для заклинания пост-обработку — имя должно совпадать
с записью в реестре `kSummonHandlers` в `magic.cpp` (сигнатура `(ch, mob, ctx,
duration)`), иначе при загрузке выводится `unknown summon handler`. Текущие
обработчики: `SetupKeeperStats`, `SetupFirekeeperStats`, `CloneCascade`.

Используйте `<summon>` для призывов **по vnum**, чей провал масштабируется с `C`
(хранитель / огненный защитник / клон). Призывы из трупа (поднять/оживить мертвеца —
моб и провал на основе телосложения/уровня берутся из трупа-источника) и не-мобные
призывы (ангел-хранитель, ментальная тень) не укладываются в эту модель данных и
живут в обработчиках `<manual_cast>` (§3.8.4).

### 3.9 `<components>` — требования к применению

```xml
<components>
    <verbal/>
    <weave/>
    <material any_of="1234|5678" cost="1"/>
</components>
```

Необязательный дочерний элемент `<spell>`. Каждый внутренний тег — маркер
наличия или дескриптор материального компонента; отсутствие `<components>`
означает «нет требований». Сегодня существует три вида:

| Тег | Атрибуты | Эффект |
|---|---|---|
| `<verbal/>` | нет | Применение требует произнесения фразы. `kSilence` на кастере блокирует (заглушённые кастеры всё равно могут применять неречевые заклинания, а `SaySpell` для них остаётся безмолвным). |
| `<weave/>` *(issue.weave-component)* | нет | Помечает действие как «настоящую магию». Заклинание-плетение не удаётся в комнате с `kNoMagic` — выдаёт `kCastForbiddenToChar` / `kCastForbiddenToRoom` так же, как прежний `<blocking><room_flags val="kNoMagic"/>`. Заменяет 209 копий этого блокирующего блока — `<weave/>` теперь единственный источник истины для «это магия». `<verbal/>` и `<weave/>` независимы (боевые кличи речевые, но **не** плетения). |
| `<material .../>` | `any_of` / `all_of` / `where` / `cost` | Компонент на основе предмета. `any_of` / `all_of` — списки vnum объектов через `\|`. `where` — список `EFind` через `\|` (по умолчанию `kObjEquip\|kObjInventory\|kObjRoom`). `cost` — количество зарядов (по умолчанию 1; 0 = только наличие фокуса; -1 = уничтожается целиком). |

Исключение `MayCastInForbiddenRoom` (старшие боги, неочарованные NPC) обходит
и шлюз `<weave/>`, и шлюз флагов комнаты `<blocking>`. Покоморнатные ключи
`kCastForbiddenToChar` / `kCastForbiddenToRoom` переопределяют повествование kDefault.

## 4. Шлюзы цели и кастера

Применение ограничивается переиспользуемыми блоками условий — `<blocking>`
(отклонить, если совпадает **хотя бы одно** условие) и `<required>` (отклонить,
если не выполнены **все**). Они никогда не пишутся «голыми»; каждый оборачивается
одним из двух контейнеров:

- **`<target_conditions>`** — дочерний элемент `<action>`. Его `<blocking>` /
  `<required>` проверяют **цель** и ограничивают **данное действие** для текущей
  цели (все его стадии). Большинство заклинаний одно-действенные, поэтому на
  практике это ограничивает всё применение для данной цели.
- **`<caster_conditions>`** — элемент уровня **заклинания** (дочерний для `<spell>`,
  не для `<action>`). Его `<blocking>` / `<required>` проверяют **кастера** и
  выполняются один раз, до цикла по целям — см. §4.3.

«Голые» `<blocking>` / `<required>` прямо под `<action>` **больше не принимаются**;
всегда оборачивайте их. (`<caster_blocking>`, одно-тег шлюз кастера, удалён — его
работу теперь выполняет `<caster_conditions>`.)

### 4.1 `<blocking>`

```xml
<target_conditions>
    <blocking>
        <mob_flags val="kNoSleep"/>
        <affect_flags val="kHold|kSleep"/>
        <room_flags val="kNoMagic"/>
        <align val="kGood"/>
    </blocking>
</target_conditions>
```

Применение отклоняется, если совпадает **хотя бы одно** из перечисленных условий:
флаг моба (только NPC), флаг эффекта (любая цель), выравнивание (мировоззрение
цели) или флаг комнаты (текущая комната КАСТЕРА). Каждая ось необязательна;
комбинируйте столько, сколько нужно заклинанию.

Форма дочерних тегов используется потому, что строки становятся длинными при
сочетании нескольких осей. Старая форма атрибута (`<blocking mob_flags="..."/>`)
упразднена.

| Дочерний тег | Значения |
|---|---|
| `<mob_flags val="…"/>` | Имена `EMobFlag` через `\|`. Допустимые: `kNoBlind`, `kNoSleep`, `kNoSilence`, `kNoHold`, `kNoBash`, `kNoCharm`, `kNoSummon`, `kNoFear`, `kCorpse`, `kResurrected`, `kMounting`, `kHelper`, `kClone`. |
| `<affect_flags val="…"/>` | Имена `EAffect` через `\|` (любой флаг эффекта, поддерживаемый движком). |
| `<room_flags val="…"/>` | Имена `ERoomFlag` через `\|`. Допустимые: `kNoMagic`, `kPeaceful`, `kTunnel`, `kNoTeleportIn`, `kNoBattle`, `kGodsRoom`, `kForMono`, `kForPoly`. Проверяется против комнаты кастера. |
| `<align val="…"/>` | `kGood`, `kEvil`, `kNeutral` — блокировать, если цель имеет данное мировоззрение. |

При заклинании с одной целью блокировка со стороны цели выдаёт кастеру `kNoeffect`.
При групповом/массовом — блокировка для данной цели происходит молча. **Блокировка
по флагу комнаты** иная: она прерывает *всё* применение до цикла по целям и
выдаёт `kCastForbiddenToChar` / `kCastForbiddenToRoom` (см. §15). Старшие боги и
неочарованные/не-опекаемые NPC обходят этот блок (`MayCastInForbiddenRoom`); заклинания
боевого клича просто не перечисляют `kNoMagic` как блокирующий флаг.

### 4.2 `<required>`

```xml
<required>
    <mob_flags val="kCorpse"/>
    <affect_flags val="kCharmed"/>
    <align val="kEvil"/>
</required>
```

Та же структура дочерних тегов, что и у `<blocking>`, но применение отклоняется,
если цель **не имеет** всех перечисленных флагов/мировоззрения. Используйте для
«только на трупах» (поднять мертвеца), «только на очарованных слугах», «только
на злых существах» (рассеять зло) и т.д. `<required>` не поддерживает `<room_flags>`.

### 4.3 `<caster_conditions>` — шлюз по **кастеру**

```xml
<caster_conditions>
    <blocking>
        <align val="kEvil"/>
    </blocking>
</caster_conditions>
```

Блок уровня **заклинания** — дочерний элемент `<spell>`, *не* `<action>`. Он
оборачивает те же `<blocking>` / `<required>`, что и `<target_conditions>` (§4.1,
§4.2), но они проверяют **кастера**, а не цель, и выполняются **один раз**, до цикла
по целям — поэтому неудача прерывает *всё* применение и выдаёт `kNoeffect` (без
молчаливого пропуска группы/массы: шлюз кастера касается одного кастера, а не цикла
по целям).

Для кастера осмысленны только оси `<align>` и `<affect_flags>` (`<mob_flags>` /
`<room_flags>` парсер принимает, но для кастеров сегодня не используются). Семантика
как на стороне цели: `<blocking>` отклоняет при **любом** совпадении; `<required>`
отклоняет, если у кастера нет **всех**.

Используйте, чтобы отказать в применении, которое кастер не может выполнить.
`kDispelEvil` несёт блок выше, поэтому злой кастер не может его применить;
удержанного кастера можно запереть от самобафа через
`<blocking><affect_flags val="kHold"/></blocking>`; а `<required>` может, наоборот,
требовать у кастера наличие флага.

> Заменяет старую одно-теговую форму `<caster_blocking><caster align= affect_flags=/>` —
> шлюз кастера теперь использует ту же вложенную структуру `<blocking>` / `<required>`,
> что и сторона цели, так что они симметричны.

### 4.4 `<reflection>` *(новая механика)*

```xml
<reflection affect_flags="kFireShield" align="kGood" prob="35"/>
```

Если исходная цель несёт хотя бы один из перечисленных флагов или имеет
указанное мировоззрение, заклинание может **отразиться обратно в кастера** с
вероятностью `prob` (процентный бросок). Самоприменение никогда не отражается.

| Атрибут | По умолчанию | Описание |
|---|---|---|
| `affect_flags` | (пусто) | Имена `EAffect` через `\|`. Совпадение любого вызывает бросок отражения. |
| `align` | `kAny` | `kGood`/`kEvil`/`kNeutral`. Совпадающее мировоззрение цели также вызывает бросок. |
| `prob` | `20` | Вероятность фактического отражения при совпадении флага/мировоззрения (в процентах). |

При срабатывании отражения выдаются три сообщения из `spell_msg.xml`:
`kReflectedToChar` (кастеру), `kReflectedToVict` (исходной цели),
`kReflectedToRoom` (свидетелям). После отражения заклинание продолжается
против кастера — урон, эффект, задержка, всё.

**Почему используется простой флаг + вероятность, а не состязание сил:** у
эффектов мобов и объектов нет значения силы, поэтому сравнение флага и силы
сегодня не имеет смысла. Параметр `prob` — рычаг дизайнера для настройки
частоты отражения для каждого заклинания.

## 5. `<damage>` — прямой урон

```xml
<damage saving="kReflex" prob="100">
    <amount min="0" dices_weight="1.0" alpha="0.5" beta="12.5"/>
    <hits skill_divisor="12" max="2" prob="100"/>
</damage>
```

| Атрибут / дочерний элемент | По умолчанию | Описание |
|---|---|---|
| `saving` | `kReflex` | `ESaving`: `kReflex`, `kStability`, `kWill`, `kCritical`, `kNone`. Успешный спасбросок вдвое уменьшает урон. |
| `prob` | `100` | Вероятность того, что урон будет нанесён (молчаливый промах иначе). `prob<100` не задействует ГСЧ. |
| `<amount min= dices_weight= alpha= beta=>` | min=0, dices_weight=1.0, alpha=0, beta=1.0 | Итоговое количество (issue.potency-formula) = `min + кубики · dices_weight · (1 + alpha · C) + beta · C`, где `C = skill_coeff + stat_coeff`. `alpha` мультипликативно масштабирует кубики с компетентностью, чтобы случайный разброс рос вместе с результатом (субквадратично). `beta` — аддитивный член компетентности. `alpha=0` сводит формулу к прежней аддитивной (Формула A). Опустите тег, чтобы сохранить значения по умолчанию. |
| `<hits skill_divisor= max= prob=>` | divisor=25, max=1, prob=20 | Поддержка нескольких ударов: `count = 1 + CalcExtraHits(...)`. Бонус дополнительных ударов использует `base_skill` броска силы, масштабируется `min(skill, 75) / skill_divisor`, ограничен значением `max`, срабатывает с вероятностью `prob`%. `prob=0` означает равномерный случайный выбор от 0 до `extra`. Отсутствие тега — одиночный удар. |

## 6. `<points>` — восстановление: HP / силы / жажда / сытость

```xml
<points extra="N" prob="100">
    <heal   npc_coeff="3" min="0" dices_weight="1.0" alpha="0.5" beta="45"/>
    <moves                min="0" dices_weight="0.0" alpha="0" beta="20"/>
    <thirst               min="0" dices_weight="0.0" alpha="0" beta="0"/>
    <full                 min="0" dices_weight="0.0" alpha="0" beta="0"/>
</points>
```

Тег `<points>` (ранее `<heal>`, переименован в issue.mag-points) содержит до
четырёх необязательных дочерних тегов количества. Каждое количество независимо
восстанавливает (или уменьшает) одну категорию на основе одного броска силы:

* **`<heal>`** — очки здоровья. `extra="N"` на внешнем `<points>` —
  целочисленный предел сверхлечения **в процентах сверх** максимума HP
  (issue.points-extra-cap): `0` ограничивает максимумом, `20` разрешает 120%, `33`
  — 133%. Имеет необязательный атрибут `npc_coeff` (по умолчанию `1.0`), который
  усиливает количество, когда кастер — NPC.
* **`<moves>`** — очки движения. Та же структура атрибутов, что у `<heal>`, без `npc_coeff`.
* **`<thirst>`** — жажда (слот движка `THIRST`).
* **`<full>`** — голод (слот движка `FULL`). Переименован из `<cond>` в issue.point-bugs.

| Атрибут внешнего `<points>` | По умолчанию | Описание |
|---|---|---|
| `extra` | `0` | Целочисленный предел сверхлечения **в процентах сверх** максимума HP (issue.points-extra-cap). `0` = строгий предел по максимуму; `20` = до 120%; `33` = до 133%. Касается только `<heal>`; остальные категории насыщаются на своих естественных пределах. |
| `prob` | `100` | Вероятность срабатывания всего действия. Неудача обнуляет все четыре категории. |

| Атрибут дочернего количества | По умолчанию | Описание |
|---|---|---|
| `min` | `0` | Фиксированное количество. Может быть **отрицательным** — отрицательные числа отнимают вместо восстановления. |
| `dices_weight` | `0` | Базовый масштаб на кубики броска силы. |
| `alpha` | `0` | Мультипликативный: насколько компетентность `C` усиливает кубики (субквадратичный рост разброса). `0` = прежнее аддитивное поведение. |
| `beta` | `0` | Аддитивный вес на `C = (skill_coeff + stat_coeff)` (прежний `competencies_weight`). |
| `npc_coeff` (только `<heal>`) | `1.0` | Дополнительный мультипликатор для кастера-NPC. |

### Знаковое соглашение для жажды / сытости

В движке поля жажды/голода инвертированы — `0` означает полное насыщение,
`kMaxCondition` (48) — «умирает от голода». XML использует игровой знак:

* **Положительное значение XML** = восстановление (меньше жажды / голода).
* **Отрицательное значение XML** = ухудшение (больше жажды / голода).

`CastToPoints` инвертирует вычисленное количество перед вызовом `gain_condition`.

### Формула

Каждое количество, до ограничений:

```
amount = ceil(min + кубики · dices_weight · (1 + alpha · C) + beta · C)   // C = skill_coeff + stat_coeff
amount += amount · (бонус силы заклинания кастера / 100)
если кастер — NPC: amount += amount · npc_coeff   // 0 для категорий, кроме лечения
```

Кубики / компетенции бросаются **один раз за применение** из `<potency_roll>`
и разделяются между всеми четырьмя категориями.

### Повествование по категориям (issue.point-bugs)

Каждая ненулевая категория выдаёт собственное сообщение. Ключи:

| Ключ | Срабатывает когда | Категория процента для `{intensity}` |
|---|---|---|
| `kHealToVict` | `<heal>` дало ненулевое количество | `amount * 100 / max_hp` |
| `kMovesToVict` | `<moves>` дало ненулевое количество | `amount * 100 / max_moves` (со знаком; отрицательное — для уменьшения) |
| `kThirstToVict` | `<thirst>` дало ненулевое количество | улучшение: `removed * 100 / current`; ухудшение: `-(cond_after * 100) / kMaxCondition` |
| `kFullToVict` | `<full>` дало ненулевое количество | то же, что kThirstToVict |

Устаревший единый ключ `kPointsToVict` удалён.

Таблица интенсивности (`lib/cfg/mechanics/points.xml`) содержит уровни `<improve>`
и `<degrade>` для каждой категории. Разрешение проходит по таблице в убывающем
порядке и возвращает первый уровень `T` такой, что `T ≤ процент`.

## 7. `<area>` — сколько целей и пер-целевое затухание

> **Смена схемы (issue.area-cast).** Старые дочерние элементы `<victims>` /
> `<decay>` удалены. Теперь у `<area>` два дочерних элемента — `<targets>`
> (сколько) и `<distribution>` (пер-целевое затухание). Площадные и групповые
> применения используют один конвейер: групповое применение — это просто
> площадное по списку союзников.

```xml
<area>
    <targets skill_divisor="15" dice_size="2" min="5" max="20"/>
    <distribution type="kStepped" decay="0.05" free_targets="3"/>
</area>
```

Блок `<area>` нужен любому действию, чья выборка цели охватывает более одного
персонажа — входному действию заклинания `kMagAreas`/`kMagMasses`/`kMagGroups`
или более позднему действию с `target="kTarFoes"` / `kTarGroup` / `kTarMinions`
(§3.8.1).

### `<targets>` — количество

| атрибут | по умолч. | смысл |
|---|---|---|
| `skill_divisor` | 1 | делитель навыка кастера, масштабирующий количество (≥1) |
| `dice_size` | 1 | грани случайного кубика количества целей |
| `min` | 1 | нижняя граница количества |
| `max` | 1 | предел **бонуса** (итог ∈ `[min, min+max]`); **`max ≤ 0` означает «без предела»** — весь список целей |
| `stat_weight` | 0 | необязательный довесок от вторичного стата (`stat_coeff` броска силы); 0 = ровно историческое количество |

> **Каждый атрибут необязателен** и откатывается к значению по умолчанию выше.
> (Отсутствующий атрибут **не** должен читаться как пустая строка — раньше это
> бросало исключение и молча удаляло всё действие; исправлено в
> issue.area-spell-bug. Поэтому `<targets max="-1"/>` сам по себе допустим и
> означает «все, без масштабирования».)

`CalcTargetsQuantity = min + min(RollDices(skill / skill_divisor, dice_size) +
ceil(stat_weight · stat_coeff), max)`. Реально поражается
`N = min(CalcTargetsQuantity, размер списка)` (или весь список при `max ≤ 0`).

### `<distribution>` — пер-целевой коэффициент `f_j`

`type` ∈ `{kUniform (по умолч.), kLinear, kStepped}`. Для j-й цели (нумерация с 1,
цель №1 = основная жертва / первый союзник, поражается первой) из `N` коэффициент
`f_j` масштабирует **и** величину эффекта, **и** эффективную успешность применения:

| `type` | `f_j` |
|---|---|
| `kUniform` | `1` — ничего не затухает (все цели в полную силу) |
| `kLinear` | `(N − j + 1) / N` — №1 в полную силу, линейно вниз до `1/N` |
| `kStepped` | `1` при `j ≤ free_targets`, иначе `max(0, 1 − decay·(j − free_targets))` |

`decay` и `free_targets` читаются **только** для `kStepped`. `f_j` всегда в
`[0, 1]` и никогда не усиливает. **NPC-кастеры обходят** затухание (`f_j = 1`
везде). Способность `kMultipleCast` смягчает затухание `kStepped` (`decay ×= 0.6`).

## 8. `<affects>` — наложение магического эффекта

Это основной механизм для бафов и дебафов.

```xml
<affects type="kSleep" saving="kWill" resist="kMind" prob="100">
    <flags val="kAfBattledec|kAfDispellable|kAfCurable"/>
    <duration base="1" skill_divisor="15" min="1" max="6"/>
    <apply id="kSleep" location="kNone">
        <modifier min="0.0" dices_weight="0.0" alpha="0" beta="0.0" factor="1" stack="1"/>
    </apply>
    <reposition pos="kSleep" stop_fight="false"/>
    <lag base="2" bonus_divisor="-1"/>
</affects>
```

### 8.1 Атрибуты `<affects>`

| Атрибут | По умолчанию | Описание |
|---|---|---|
| `type` | обязательный | Значение перечисления `ESpell` — **идентичность эффекта**, записываемая на цели. Используется как ключ для снятия/лечения и сообщений. |
| `saving` | `kReflex` | Спасбросок жертвы для избегания эффекта. `kNone` — без спасброска. |
| `resist` | `kFire` | `EResist` — канал сопротивления, применяемый к длительности эффекта. |
| `prob` | `100` | Вероятность срабатывания блока эффекта (молчаливый промах иначе). `<lag>` и `<reposition>` также привязаны к этой вероятности. |
| `potency_weight` | `1.0` | *(issue.affects-potency-weight)* Масштаб сохранённого значения `Affect::potency` (т.е. `cast_potency`). Эффект накладывается с силой `cast_potency * potency_weight`; состязание при снятии читает это масштабированное значение. **Сценарий применения:** заклинание с большим модификатором, где тот же `<potency_roll>` питает и формулу модификатора, и сохранённую силу, может стать неснимаемым. `potency_weight="0.4"` позволяет разделить их — держать модификатор сильным, уменьшая лишь сохранённую силу. |
| `tick_spell` | *(нет)* | *Только для комнатных эффектов.* `ESpell` `kService`-заклинания, чьи действия обработчик тика выполняет каждый пульс (см. §11.5). Отсутствует → нет тика на основе данных (эффект пассивный или использует `tick_handler`). |
| `tick_handler` | *(нет)* | *Только для комнатных эффектов.* Имя C++-обработчика тика — механизм manual-cast для тиков, невыразимых данными (напр. погода `kThunderstorm`). Регистрируется в таблице обработчиков тика комнат; получает эффект (может читать его `duration`). Имеет приоритет над `tick_spell`. |

### 8.2 `<flags val="…">` — биты `EAffFlag`

| Флаг | Значение |
|---|---|
| `kAfBattledec` | Длительность убывает с каждым раундом боя в дополнение к каждому тику. |
| `kAfDeadkeep` | Эффект сохраняется на трупе / после смерти. |
| `kAfPulsedec` | Убывание на каждом пульсе вместо каждого тика. |
| `kAfSameTime` | Одинаковая длительность для игрока и NPC (без пересчёта тиков). |
| `kAfUpdateDuration` | Повторное применение заменяет длительность (наибольшая из старой/новой). |
| `kAfAccumulateDuration` | Повторное применение добавляет к существующей длительности. |
| `kAfUpdateMod` | Повторное применение заменяет модификатор. |
| `kAfDispellable` | **Можно снять через рассеивание** (например, `kDispellMagic`). |
| `kAfCurable` | **Можно снять через лечение** (например, `kRemovePoison`). |
| `kAfMustBeHandled` | Только для комнатных эффектов: эффект тикает каждый пульс. `HandleRoomAffect` выполняет его `tick_spell` (§8.1, §11.5), если задан, иначе его `tick_handler` (именованный C++-обработчик, напр. погода `kThunderstorm`). Свитча по заклинаниям нет. |
| `kAfUnique` | Только для комнатных эффектов: перед наложением удалить предыдущее применение того же заклинания тем же кастером. |

`kAfDispellable` / `kAfCurable` — **единственный источник истины** о том, может
ли данный эффект быть снят конкретным рассеиванием/лечением. Уберите оба — и
эффект постоянный до естественного истечения.

### 8.3 `<duration>` (масштабируемая по навыку)

```xml
<duration base="2" skill_divisor="5" min="0" max="0"/>
```

| Атрибут | Описание |
|---|---|
| `base` | Фиксированная базовая длительность (в часах, пересчитывается в тики для игроков). |
| `skill_divisor` | Бонус в часах за `min(skill,75) / skill_divisor`, используя `base_skill` броска силы. `0` — без масштабирования по навыку. |
| `min` / `max` | Ограничения в часах. `0` с любой стороны означает «без ограничения с этой стороны». |

### 8.4 `<apply>` — одно изменение стата / статуса на `<apply>`

```xml
<apply id="kPoisoned" location="kStr" random="false">
    <modifier min="2.0" dices_weight="0.0" alpha="0" beta="0.0" factor="-1" stack="3"/>
</apply>
```

| Атрибут `<apply>` | Описание |
|---|---|
| `id` | Флаг `EAffect`, налагаемый этим apply (например, `kSleep`, `kPoisoned`, `kInvisible`). `kUndefinded` — для apply только стата без флага. |
| `location` | `EApply`: куда попадает модификатор. `kNone` — только флаг, без стата. |
| `random` | `Y`/`N` (по умолчанию N). При Y этот apply применяется не всегда: см. §8.7. |

| Атрибут `<modifier>` | По умолчанию | Описание |
|---|---|---|
| `min` | `0.0` | Минимальная величина модификатора. |
| `dices_weight` | `0.0` | Базовый масштаб на кубики броска силы. |
| `alpha` | `0.0` | Мультипликативный: компетентность `C` усиливает кубики. `0` (обычное значение для модификаторов, они плоские) = прежнее аддитивное поведение. |
| `beta` | `0.0` | Аддитивный вес на `C = (skill_coeff + stat_coeff)` (прежний `competencies_weight`). |
| `factor` | `1` | Итоговый знак/масштаб. Используйте `-1` для дебафов (штраф к силе, штраф к спасброску и т.д.). |
| `cap` | `0` | Необязательный верхний предел сырой величины **до** factor. `0` = без ограничения. |
| `stack` | `1` | **Максимальное число стаков** — см. §8.5. |

Формула:

```
raw      = min + ceil(кубики · dices_weight · (1 + alpha · C) + beta · C)   // C = компетентность = skill_coeff + stat_coeff
если cap > 0: raw = min(raw, cap)            # необязательное ограничение до factor
modifier = factor · raw
```

### 8.5 Стакирование *(новая механика)*

`stack` — максимальное число стаков, которое apply может накопить на одной цели.
При `stack="1"` (по умолчанию) повторное применение либо обновляет, либо
отказывает в повторном наложении. При `stack="3"` повторное применение **добавляет
стак** до 3, накапливая модификатор.

**Что накапливают стаки:**

* Модификатор растёт: стак 2 достигает `2·modifier`, стак 3 — `3·modifier` и т.д.
* Длительность также продлевается, если это задано флагами эффекта.
* При снятии (например, через `kDispellMagic`) **один стак снимается**, а не весь
  эффект — модификатор уменьшается пропорционально. Только при снятии последнего
  стака эффект исчезает.

Пример — яд, стакирующийся до 3 раз:

```xml
<affects type="kSlowPoison" saving="kCritical" resist="kDark">
    <flags val="kAfAccumulateDuration|kAfCurable"/>
    <duration base="0" skill_divisor="3" min="0" max="0"/>
    <apply id="kPoisoned" location="kStr">
        <modifier min="2.0" dices_weight="0.0" alpha="0" beta="0.0" factor="-1" stack="3"/>
    </apply>
    <apply id="kPoisoned" location="kPoison">
        <modifier min="30.0" dices_weight="0.0" alpha="0" beta="0.0" factor="1" stack="3"/>
    </apply>
</affects>
```

### 8.6 `<reposition>` — принудительная позиция / остановка боя

```xml
<reposition pos="kSleep" stop_fight="false"/>
```

| Атрибут | По умолчанию | Описание |
|---|---|---|
| `pos` | (нет) | `EPosition`. Жертва принудительно переводится в эту позицию при наложении. Опустите, чтобы пропустить изменение позиции, но сохранить `stop_fight`. |
| `stop_fight` | `N` | При `Y` все, кто сражался с жертвой, прекращают бой. |

### 8.7 `<lag>` — задержка боя жертвы

```xml
<lag base="6" bonus_divisor="-1"/>
```

| Атрибут | Описание |
|---|---|
| `base` | Фиксированная задержка в боевых раундах при наложении. |
| `bonus_divisor` | При `> 0` коэффициент низкого навыка кастера добавляется как `low_skill_coeff / bonus_divisor` дополнительных раундов. `≤ 0` — фиксированная задержка (без масштабирования). |

### 8.8 Случайные пулы apply

Из `<apply random="true">` в одном `<affects>` **ровно один** выбирается
случайно за применение (равномерная выборка). Безусловные apply всегда
срабатывают; случайные делят один пул.

Пример (из `kFailure`): гарантированный штраф морали + один случайный штраф стата из шести.

```xml
<affects type="kFailure" saving="kWill" resist="kDark">
    <flags val="kAfDispellable|kAfCurable"/>
    <duration base="2" skill_divisor="5" min="0" max="0"/>
    <apply id="kUndefinded" location="kMorale">
        <modifier min="5.0" alpha="0" beta="1.0" factor="-1"/>
    </apply>
    <apply id="kUndefinded" location="kStr" random="true">
        <modifier min="0.0" alpha="0" beta="0.11" factor="-1"/>
    </apply>
    <!-- … аналогичные apply для kDex, kInt, kWis, kCon, kCha … -->
</affects>
```

### 8.9 Масштабирование модификаторов с компетентностью кастера

Философия новой системы: **величина модификатора растёт с компетентностью кастера
(навык + стат), а не с его уровнем или числом перерождений напрямую**. Старые
формулы вида `-1 - R/2` или `(L+R)/3` переводятся через `beta` (аддитивный вес
компетентности). Модификаторы почти всегда плоские (`dices_weight=0`), и тогда
формула — это просто `min + ceil(beta · C)`, а `beta` — это в точности
переименованный прежний `competencies_weight`, поэтому правила настройки ниже
переносятся без изменений. `alpha` важен только при `dices_weight ≠ 0`.

Правило пересчёта при переводе из старых формул:

> **1 перерождение = +5 к максимуму навыка + +1 к базовому стату**
>
> (Новый персонаж имеет максимум навыка 80 и базовый стат по умолчанию класса;
> каждое трансформирование/перерождение повышает максимум на 5 и стат на 1.)

Для стандартного броска силы (`low=3 hi=1.25`, `threshold=22 weight=0.5`)
компетенции за перерождение растут на `(5·1.25 + 1·0.5)/100 = 0.0675`. Значит,
модификатор, растущий на `k` за перерождение, требует `beta ≈ k / 0.0675`.

**Правило min ≥ 0.** Выбирайте `min` и `beta` так, чтобы
необученный кастер (компетенции = 0) получал модификатор `min·factor` = 0. Это
предотвращает случайный инвертированный знак у дебафа.

**Рекомендуемый алгоритм** (используется для большинства `<modifier>` в `spells.xml`):

```
cw_exact   = |цель при R=12 макс.навыке| / competencies_at_R12_max
cw         = floor(cw_exact * 10) / 10        # округлить вниз, чтобы min ≥ 0
min        = |цель при R=12| - ceil(competencies_at_R12 * cw)
```

**Пример.** Старая формула `kBless` была `-5 - R/3` для `kSavingStability`.
При стандартном броске силы (компетенции ≈ 3.12 при R=12 макс.навыке) `cw_exact =
9/3.12 ≈ 2.88`, поэтому `cw = 2.8` и `min = 9 - ceil(3.12·2.8) = 0`. Текущий XML:

```xml
<apply id="kBless" location="kSavingStability">
    <modifier min="0.0" dices_weight="0.0" alpha="0" beta="2.8" factor="-1"/>
</apply>
```

Проверка:
- Необученный кастер (компетенции = 0): модификатор = `-1·(0 + 0)` = 0.
- R=0 макс.навык (компетенции ≈ 2.31): модификатор = `-1·ceil(6.47)` = −7.
- R=12 типовой (компетенции ≈ 3.12): модификатор = `-1·ceil(8.74)` = **−9** ✓
- R=24 макс. (компетенции ≈ 3.93): модификатор = `-1·ceil(11.0)` = −11.

Наклон чуть мягче, чем старый `-1/3` за перерождение, но R=12 совпадает точно, и
значением управляет компетентность от начала до конца.

**Когда `min` оказывается отрицательным.** При агрессивных бросках силы (большие
`low_skill_bonus`/`hi_skill_bonus` или большой `weight` стата) алгоритм может дать
`min < 0`. *Избегайте этого.* Отрицательный `min` превращает дебаф в баф для
необученного мага. Исправьте:

* уменьшив `beta` (более мягкое масштабирование), или
* нормализовав бросок силы до стандартной формы.

Несколько заклинаний с необычно агрессивными бросками силы (`kPatronage`,
`kEviless`, `kCurse`) сейчас используют плоский `min` без `beta` — они оставлены так
до нормализации их бросков силы.

## 9. `<unaffect>` — снятие эффектов / лечение *(новые механики)*

```xml
<unaffect affect_flags="kAfCurable" potency_weight="1.0" prob="100">
    <blocking      any_of="kGodsShield" all_of=""/>
    <breaking      any_of="kSanctuary"/>
    <remove_anyway any_of="kQuestMark"/>
    <remove        all_of="kInvisible|kCamouflage|kHide" breaking_by_failure="true"/>
</unaffect>
```

### 9.1 Атрибуты `<unaffect>`

| Атрибут | По умолчанию | Описание |
|---|---|---|
| `affect_flags` | `kAfCurable\|kAfDispellable` | **Какие эффекты может затронуть это снятие.** Эффект доступен, только если несёт хотя бы один из этих битов `EAffFlag`. |
| `potency_weight` | `1.0` | Мультипликатор на силу заклинания снятия при состязании с записанной силой эффекта. |
| `prob` | `100` | Вероятность срабатывания блока снятия (молчаливый пропуск иначе). |

### 9.2 Четыре под-блока

Каждый из четырёх дочерних элементов содержит два списка `ESpell`:

| Блок | Когда проверяется | Эффект |
|---|---|---|
| `<blocking>` | Первым | Если присутствует любой перечисленный эффект, **все `<remove>` пропускаются** (но `<remove_anyway>` всё равно выполняется). Цепочка *не* прерывается. |
| `<breaking>` | Первым | Если присутствует любой перечисленный эффект, **цепочка прерывается** после этого этапа. |
| `<remove_anyway>` | Вторым | Эффекты для снятия **независимо от блокировки**. |
| `<remove>` | Третьим (только если не заблокировано) | Обычный список снятия — снимаются, если `<blocking>` не сработал. |

В каждом блоке:

| Атрибут | Значение |
|---|---|
| `any_of` | Список `ESpell` через `\|`. Срабатывает при наличии **любого** (для blocking/breaking) или снимает **первый найденный** (для remove). Одиночное значение `"*"` — **маска**, см. §9.5. |
| `all_of` | Список `ESpell` через `\|`. Для blocking/breaking — срабатывает только когда присутствуют **все**. Для remove — снимает **каждый** найденный. Значение `"*"` — **маска**. |
| `breaking_by_failure` *(только remove / remove_anyway)* | При `true` **неудачная проверка силы** для любого кандидата прерывает цепочку. |

### 9.3 Снятие с проверкой силы *(новая механика)*

Каждый эффект, наложенный через `CastAffect`, **записывает силу применения** и
флаг дебафа. При попытке снятия через `CastUnaffects` срабатывает состязание сил.

**Математика**, обе стороны используют одну формулу `cast_potency`:

```
stored_potency = cast_potency(spell_effect, caster_at_impose) · <affects potency_weight>
dispel_potency = cast_potency(dispel_spell, caster_at_dispel) · <unaffect potency_weight>
```

Снятие успешно при `dispel_potency > stored_potency`. Полные правила:

1. **Ненасильственное снятие + небаф** — **без проверки**, эффект всегда снимается.
2. **Гарантированный шанс 5 %** — независимо от силы.
3. **Состязание сил** — иначе `dispel_potency > stored_potency`.

**Проработанный пример.** `kFireball` с `<affects potency_weight="0.4">` и
броском 60 накладывает эффект горения с `stored = 60 × 0.4 = 24`. `kDispellMagic`
с `<unaffect potency_weight="0.5">` и броском 50 состязается при `dispel = 50 × 0.5 = 25`.
Снятие победит (25 > 24) — но лишь с небольшим перевесом.

### 9.4 Путь разрыва цепочки *(новая механика)*

Два способа прервать цепочку применения из этапа снятия:

1. **Разрыв по наличию эффекта** — `<breaking any_of="…">` находит присутствующий эффект; цепочка прерывается без попытки снятия.
2. **Разрыв по неудаче снятия** — `<remove … breaking_by_failure="true">` пытается снять, и проверка силы провалилась.

Прерванная цепочка означает: последующие этапы — эффект, урон, очки и т.д. — **не выполняются**.

### 9.5 Маска `"*"` *(новая механика)*

`any_of="*"` и `all_of="*"` расширяют `<remove>` / `<remove_anyway>` от
«фиксированного списка» до «всего, соответствующего фильтру `affect_flags`».

| Маска | Выборка |
|---|---|
| `any_of="*"` | Выбрать **один** доступный эффект случайно (равномерная выборка). |
| `all_of="*"` | Поставить в очередь **каждый** доступный эффект жертвы. |

«Доступный» означает, что эффект несёт хотя бы один из битов `EAffFlag` из
атрибута `affect_flags` окружающего `<unaffect>`.

**Использование:**
- Универсальные заклинания «рассеять одну случайную магию» / «вылечить все яды»
- Снятия по сфере: добавьте новый бит `EAffFlag` и используйте маску.

**Пример миграции `kDispellMagic`:**

```xml
<spell id="kDispellMagic" element="kLight" mode="kEnabled">
    <!-- … обычные поля … -->
    <flags val="kMagUnaffects"/>
    <talent_actions>
        <action>
            <unaffect affect_flags="kAfDispellable">
                <remove any_of="*"/>
            </unaffect>
        </action>
    </talent_actions>
</spell>
```

### 9.6 Сводка конвейера снятия

Для каждого применения, запускающего `<unaffect>`:

1. Применить шлюз `prob` (молчаливый пропуск при неудаче).
2. Проверить `<blocking>` и `<breaking>` (с поддержкой маски).
3. Собрать кандидатов из `<remove_anyway>` (всегда) и `<remove>` (если не заблокировано).
4. При наличии кандидатов — для каждого: `DispelSucceeds` → `RemoveAffectAndAnnounce` или зафиксировать сопротивление.
5. Если кандидат с `breaking_by_failure` устоял, или сработало условие `<breaking>` — вернуть `kBreak`.
6. Иначе вернуть `kSuccess` и продолжить цепочку.

## 11. `CastToPoints`, `CastToAlterObjs`, `CastCreation`, `CastManual`

Эти этапы управляются своим флагом `kMag…` и выполняют специализированную логику
в `magic.cpp`. Они **частично конфигурируются через данные**:

* `CastToPoints` (восстановление HP / сил / жажды / голода) полностью управляется
  блоком `<points>` (issue.mag-points). Все четыре количества делят один бросок
  силы и один шлюз `prob`. Каждое ненулевое количество выдаёт собственное
  сообщение (`kHealToVict` / `kMovesToVict` / `kThirstToVict` / `kFullToVict`).
* `CastToAlterObjs` выполняет обработчик по заклинанию (благословить, проклясть,
  невидимость, яд, зачаровать, починить и т.д.) и читает сообщения из `spell_msg.xml`.
* Призывы теперь — действие `<summon>` из данных (§3.8.5) для призывов по vnum
  (хранитель / огненный защитник / клон) и обработчики `<manual_cast>` (§3.8.4)
  для призывов из трупа и не-мобных (поднять/оживить мертвеца, ангел-хранитель,
  ментальная тень). Сообщения остаются в `spell_msg.xml` (`kSummonToRoom`,
  `kSummonFail`, `kSummonNoCorpse` и т.д.). Старого флага `kMagSummons` больше нет.
* `CastCreation` — чистый этап кода.
* `CastManual` запускает обработчик, написанный вручную, но **какой именно** —
  теперь задаётся данными: укажите его через `<manual_cast handler="SpellX"/>`
  внутри действия (§3.8.4). Сам обработчик по-прежнему живёт в `spells.cpp`; XML
  лишь выбирает его и шлюзует как стадию.

Настраиваемые через данные части этих этапов: `<flags>`, `<targets>`, `<misc>`,
`<mana>` и сообщения заклинания.

### 11.5 Комнатные эффекты (`kMagRoom`) — та же схема `<affects>`, **длительность в сырых пульсах**

Комнатные заклинания (флаг `kMagRoom`, обрабатываются через `CallMagicToRoom` в
`magic_rooms.cpp`) повторно используют **ту же схему `<affects>`**, что и
эффекты персонажей (§8). Два отличия от семантики эффектов персонажей:

1. **Единица длительности — СЫРЫЕ ПУЛЬСЫ, не часы.** Для комнатных эффектов
   значение используется напрямую без умножения на `kSecsPerMudHour /
   kSecsPerPlayerAffect`.
2. **Перечисление флагов эффекта — `ERoomAffect`, а не `EAffect`.** Большинство
   комнатных эффектов сегодня не устанавливают флаг. Для комнатных эффектов
   укажите `id="kUndefinded"`.

**Биты флагов, важные для комнатных эффектов:**

| Флаг | Эффект |
|---|---|
| `kAfUpdateDuration` | Повторное применение вашего комнатного эффекта обновляет длительность. |
| `kAfAccumulateDuration` | Повторное применение суммирует длительности. |
| `kAfMustBeHandled` | Эффект тикает каждый пульс — `HandleRoomAffect` выполняет его `tick_spell` (ниже), если задан, иначе его `tick_handler`. |
| `kAfUnique` | Перед наложением удалить предыдущее применение того же заклинания тем же кастером. |

**Пообработка по тику — `tick_spell` (на основе данных).** Комнатный эффект с
`kAfMustBeHandled` может указать **тик-заклинание** в `<affects tick_spell="...">` —
`kService`-заклинание, чьи действия обработчик тика выполняет каждый пульс, без кода:

* **Однофазное** (обычное `kEnabled`-заклинание, напр. `kThunderStone`): всё заклинание
  площадно применяется к врагам в комнате каждый пульс — прежний приём «перекастовать
  заклинание каждый раунд». `kMeteorStorm` → `kThunderStone`; `kBlackTentacles` → `kDamageSerious`.
* **Многофазное** (`kService`-заклинание с несколькими действиями): обработчик выполняет
  **одно действие за пульс**, циклически `action[(apply_time-1) % N]`, поэтому каждый раунд
  продвигает фазу, и последовательность зацикливается. У каждого действия свой `target=`
  (`kTarFoes`, `kTarRoomThis`, …) и эффекты (`<side_spell>`, `<damage>`, …). `kDeadlyFog` →
  `kDeadlyFogTick` (8 действий, kPoison → … → kMassCurse по врагам в комнате).
* **Повествование по тику** берётся из слотов `kCustomMsgOne…Ten` *накладывающего*
  заклинания, циклически по той же фазе, поэтому сообщение и эффект согласованы (kDeadlyFog задаёт 8).
* **`tick_handler`** → для тиков, невыразимых данными (смена погоды `kThunderstorm`), укажите
  C++-обработчик по строке в `<affects tick_handler="...">`. Он регистрируется в таблице
  обработчиков тика комнат и получает эффект (может читать его `duration`) — механизм manual-cast
  для комнатных тиков. В `HandleRoomAffect` **нет** свитча по заклинаниям: каждый тик разрешается
  в `tick_spell` или `tick_handler`.

**Создание нового комнатного заклинания:**

```xml
<spell id="kMyRoomSpell" element="kEarth" mode="kEnabled">
    <name rus="..." eng="..."/>
    <misc pos="kStand" violent="N" danger="1"/>
    <mana .../>
    <targets val="kTarIgnore"/>
    <flags val="kMagRoom"/>
    <potency_roll>
        <base_skill id="kEarthMagic" low_skill_bonus="0" hi_skill_bonus="0"/>
    </potency_roll>
    <talent_actions>
        <action>
            <affects type="kMyRoomSpell" saving="kNone" resist="kVitality">
                <!-- Длительность в СЫРЫХ ПУЛЬСАХ. -->
                <duration base="20" skill_divisor="5" min="0" max="0"/>
                <flags val="kAfMustBeHandled"/>
            </affects>
        </action>
    </talent_actions>
</spell>
```

Если заклинанию нужна логика по тику, задайте `tick_spell` на `kService`-заклинание
(см. «Пообработка по тику» выше) — без кода. Только сугубо кодовые тики (напр. погода)
по-прежнему требуют case в `HandleRoomAffect`.

## 12. Сообщения (`lib/cfg/spell_msg.xml`)

Строки сообщений для каждого заклинания, ключи — id заклинания (пучок) и тип
`ESpellMsg`. `id="kDefault"` — общий пучок-запасной для сообщений, не
переопределённых конкретным заклинанием.

```xml
<spell_msg>
    <sheaf id="kDefault">
        <message type="kNoeffect" val="Заклинание не оказало никакого эффекта."/>
        <message type="kAffExpired" val="Магия, наложенная на вас, рассеялась."/>
        …
    </sheaf>
    <sheaf id="kFireball">
        <message type="kAffImposedToChar" val="Огонь обжигает вас!"/>
        <message type="kAffImposedToRoom" val="$n окутывает $N3 пламенем!"/>
        <message type="kAffDispelledToChar" val="Пламя гаснет."/>
        <message type="kAffExpired" val="Жар, охвативший вас, утихает."/>
    </sheaf>
    …
</spell_msg>
```

### Общие ключи сообщений

| Ключ | Когда выдаётся | Аудитория |
|---|---|---|
| `kNoeffect` | Универсальный исход «нет эффекта». | Кастер |
| `kAffImposedToChar` | Эффект от заклинания был наложен. | Цель |
| `kAffImposedToRoom` | То же, для свидетелей. | Комната |
| `kAffDispelledToChar` | Эффект данного **типа** был рассеян. Пучок привязан к **снятому** эффекту, а не к заклинанию снятия. | Цель |
| `kAffDispelledToRoom` | То же, для свидетелей. | Комната |
| `kAffExpired` | Эффект истёк естественно. | Цель |
| `kAffDispelledToOwner` | Ваш эффект был рассеян с того, на кого вы его наложили. | Исходный кастер |
| `kHealToVict` | `<heal>` дало ненулевое количество. | Цель |
| `kMovesToVict` | `<moves>` дало ненулевое количество. | Цель |
| `kThirstToVict` | `<thirst>` дало ненулевое количество. | Цель |
| `kFullToVict` | `<full>` дало ненулевое количество. | Цель |
| `kReflectedToChar` | Отражение: заклинание отскочило. | Кастер |
| `kReflectedToVict` | Отражение: цель, которая отразила. | Исходная жертва |
| `kReflectedToRoom` | Отражение: свидетели. | Комната |
| `kCastForbiddenToChar` | Срыв из-за флага комнаты. | Кастер |
| `kCastForbiddenToRoom` | Объявление свидетелям. Пустое — без объявления. | Комната |
| `kCantCastPosition`, `kCantCastSleeping`, … | Неудача предусловий применения. | Кастер |
| `kCantCastNotAlly` | Шлюз цели `kTarAllyOnly` — жертва не является собой/членом группы. | Кастер |
| `kAreaToChar`, `kAreaToRoom`, `kAreaToVict` | Объявление площадного применения. | Различные |
| `kCastIncantToChar`, `kCastDisappearToRoom`, `kCastAppearToRoom` | Баннер произнесения + появление/исчезновение названия заклинания в комнате. | Различные |
| `kCastSayToSelf`, `kCastSayToOther`, `kCastSayToObj`, … | Строки «$n смотрит на $N3 и произносит…» при произнесении заклинания. | Комната / жертва |
| `kCastInterruptedToChar`, `kCastPreparedToChar` | Концентрация прервана / заклинание готово. | Кастер |
| `kNoTarget`, `kWrongTarget` | Аргумент цели не найден или тип цели отклонён. | Кастер |
| `kSummonToRoom`, `kSummonFail`, `kSummonNoCorpse`, … | Исходы призыва/воскрешения. | Различные |
| `kFightDeathToChar`, `kFightHitToChar`, `kFightMissToChar` (+ToVict, +ToRoom) | Боевые сообщения об уроне. | Различные |
| `kCustomMsgOne` … `kCustomMsgTen` | Универсальные слоты для разовых строк (без своей именованной константы). Также несут повествование комнатных эффектов по тику, циклически по `Affect::apply_time` — один слот на фазу (kDeadlyFog использует 8). | Различные |

Сообщения хранятся **без** завершающего `\r\n` — `act()` и `SendMsgToChar`
добавляют его сами.

Поиск сообщения сначала обращается к собственному пучку заклинания, затем
к `kDefault`. Исключения:

* **Наложение** (`kAffImposedToChar` / `kAffImposedToRoom`) и **отражение**
  (`kReflectedToChar` / …) ищутся напрямую **без запасного kDefault** —
  отсутствующий ключ остаётся молчаливым.
* **Повествование снятия** (`kAffDispelledToChar` / `kAffDispelledToRoom`)
  теперь *(issue.dispel-default-msg)* проходит через поиск с запасным kDefault.
  Пучок kDefault содержит универсальный запасной вариант — раньше большинство
  эффектов (kSanctuary / kBless / kArmor / kFly и т.д.) снимались молча.

## 13. Проработанные примеры

### 13.1 Простое заклинание урона с дополнительными ударами

```xml
<spell id="kMagicMissile" element="kAir" mode="kEnabled">
    <name rus="волшебная стрела" eng="magic missile"/>
    <misc pos="kFight" violent="Y" danger="1"/>

    <flags val="kMagDamage|kNpcDamagePc"/>
    <potency_roll>
        <dices ndice="6" sdice="4" adice="10"/>
        <base_skill id="kFireMagic" low_skill_bonus="3" hi_skill_bonus="0.75"/>
        <base_stat id="kWis" threshold="22" weight="5"/>
    </potency_roll>
    <talent_actions>
        <action>
            <damage saving="kReflex">
                <amount min="0" dices_weight="1.0" alpha="0.5" beta="12.5"/>

            </damage>
        </action>
    </talent_actions>

```

* Спасбросок `kReflex`, успех вдвое уменьшает урон.
* Количество = `кубики · 1.0 + компетенции · 25`.
* До 2 дополнительных ударов, срабатывающих каждый раз (`prob=100`), масштабируемых `min(kFireMagic, 75) / 12`.

```xml

    <name rus="лечить яд" eng="remove poison"/>
    <misc pos="kFight" violent="N" danger="0"/>
    <mana max="60" min="45" change="2"/>
    <targets val="kTarCharRoom|kTarFightSelf|kTarObjInv|kTarObjRoom"/>
    <flags val="kMagUnaffects|kMagAlterObjs|kNpcUnaffectNpc"/>
    <potency_roll>

        <base_skill id="kLifeMagic" low_skill_bonus="3" hi_skill_bonus="1.25"/>
```

* `affect_flags="kAfCurable"` — не трогает только `kAfDispellable` эффекты.

```xml

    <name rus="блестящая пыль" eng="glitterdust"/>
    …
    <talent_actions>
        <action>
            <area>
                <targets skill_divisor="15" dice_size="3" min="5" max="20"/>
                <distribution type="kStepped" decay="0.05" free_targets="5"/>
            </area>
            <unaffect>
                <remove all_of="kInvisible|kCamouflage|kHide" breaking_by_failure="true"/>
            </unaffect>
            <affects type="kGlitterDust" saving="kReflex" resist="kEarth">
                <flags val="kAfDispellable|kAfCurable"/>
                <duration base="4" skill_divisor="0" min="0" max="0"/>
                <apply id="kGlitterDust" location="kSavingReflex">
                    <modifier min="0.0" dices_weight="0.0" alpha="0" beta="4.4" factor="1"/>
                </apply>
            </affects>
        </action>

</spell>
```

* `<unaffect>` выполняется первым: пытается снять `kInvisible|kCamouflage|kHide`.

* Если снятие успешно (или эффекты отсутствовали), дебаф применяется нормально.

### 13.8 Мультидейственное заклинание: `kSacrifice` *(пер-действенные цели + cast-chain)*

`kSacrifice` («высосать жизнь») — эталонное мультидейственное заклинание: оно
наносит урон врагу, затем лечит кастера на величину нанесённого урона, затем лечит
**нежить**-миньонов кастера на ту же величину. Три блока `<action>`, у каждого своя
цель.

```xml
<spell id="kSacrifice" element="kDark" mode="kEnabled">
    <name rus="высосать жизнь" eng="sacrifice"/>
    <components><verbal/><weave/></components>
    <misc pos="kFight" violent="Y" danger="10"/>
    <mana max="140" min="75" change="3"/>
    <targets val="kTarCharRoom|kTarFightVict|kTarNotSelf"/>
    <flags val="kMagDamage|kNpcDamagePc|kNpcDamagePcMinhp"/>
    <potency_roll>
        <dices ndice="8" sdice="15" adice="60"/>
        <base_skill id="kDarkMagic" low_skill_bonus="2" hi_skill_bonus="2.5"/>
        <base_stat id="kWis" threshold="32" weight="4.5"/>
    </potency_roll>
    <talent_actions>
        <!-- A1 — урон по ОДНОМУ врагу (входное действие; цель уровня заклинания). -->
        <action>
            <damage saving="kWill">
                <amount min="0" dices_weight="1.0" alpha="0.5" beta="62"/>
            </damage>
            <target_conditions>
                <blocking><affect_flags val="kCharmed"/></blocking>   <!-- никогда по своему питомцу -->
            </target_conditions>
        </action>
        <!-- A2 — лечит КАСТЕРА на только что нанесённый урон (base=kDamage). -->
        <action target="kTarFightSelf" base="kDamage">
            <points extra="300">
                <heal min="0" dices_weight="0" alpha="0" beta="1.0" npc_coeff="0"/>
            </points>
        </action>
        <!-- A3 — лечит НЕЖИТЬ-миньонов кастера на ту же величину. -->
        <action target="kTarMinions" base="kDamage">
            <points extra="300">
                <heal min="0" dices_weight="0" alpha="0" beta="1.0" npc_coeff="0"/>
            </points>
            <area>
                <targets max="-1"/>             <!-- все миньоны, без масштабирования -->
                <distribution type="kUniform"/>
            </area>
            <target_conditions>
                <required><mob_flags val="kCorpse"/></required>   <!-- только нежить -->
            </target_conditions>
        </action>
    </talent_actions>
</spell>
```

Что делает каждая часть:

* **A1 (входное действие)** использует собственную область заклинания
  (`<targets>`/`kMagDamage`) — один выбранный враг — и наносит урон со спасброском
  `kWill`. `<target_conditions><blocking>` по `kCharmed` отказывается высасывать
  собственного питомца. Как входное действие, оно также несёт шлюзы заклинания и
  реакцию жертвы (отложенную до конца).
* **A2 (`target="kTarFightSelf"`, `base="kDamage"`)** перенацеливается на кастера и
  **лечит**. `base="kDamage"` заменяет скаляр компетентности на *фактически снятые
  A1 HP*; при `dices_weight="0"` и `beta="1.0"` лечение равно урону (чистый перенос
  1:1). `<points extra="300">` позволяет перелечить до 4× от максимума.
* **A3 (`target="kTarMinions"`, `base="kDamage"`)** повторяет это лечение на
  очарованных NPC-последователях кастера, но `<required><mob_flags val="kCorpse"/>`
  ограничивает его **нежитью**, а `<area><targets max="-1"/>` раздаёт лечение
  **всем** им равномерно. Заметьте: `<targets max="-1"/>` намеренно опускает
  `skill_divisor`/`dice_size`/`min`/`stat_weight` — они берутся по умолчанию (§7).

Что иллюстрирует пример:

* **Пер-действенные цели** — A1 бьёт врага, A2 кастера, A3 миньонов; заклинание
  перенацеливается по ходу применения (§3.8.1).
* **Cast-chain** — A2 и A3 читают накопленный **урон** вместо повторного броска
  компетентности (§3.8.2). Аккумулятор фиксирует *фактически снятые HP* (с уже
  учтённым оверкиллом и сопротивлениями), поэтому лечение не может превысить урон.
* **Пер-действенная область** — охватывает несколько целей только A3; A1/A2
  одноцелевые.
* **Одна реакция** — враг отвечает один раз, после A3, а не между действиями.

> Одноцелевой вариант «нанести урон, затем вылечиться на него» требует только
> A1 + A2; A3 — бонус для нежити-миньонов. *Массовый* вариант вместо этого заставил
> бы охватывать несколько целей само A1 (`kMagMasses` + `<area>` на A1).

## 15. Быстрый чек-лист для добавления нового заклинания

1. **Выберите id `ESpell`.** Добавьте его в `spells_constants.h`, если оно
   действительно новое (задача для инженера). **Прототипирование?** Используйте
   один из зарезервированных слотов `kTestOne … kTestFive` (см. §17).
2. **Добавьте элемент `<spell>`** в `lib/cfg/spells.xml`. Обязательны:
   заголовок, `<name>`, `<misc>`, `<mana>`, `<targets>`, `<flags>`.
3. **Выберите этапы**, перечислив нужные флаги `kMag…` в `<flags val>`.
4. **Создайте `<potency_roll>`** — хотя бы один из `<dices>`, `<base_skill>`,
   `<base_stat>`. Без броска эффекты нулевые.
5. **Напишите блок `<talent_actions>`** — сначала шлюзы, затем эффекты. Для
   мультидейственного заклинания перечислите действия в порядке выполнения;
   действиям 2+ задайте `target=` (и `base=`, если они должны масштабироваться от
   результата раннего действия) и `<area>` для любого действия, охватывающего
   несколько целей (§3.8, §7, §13.8).
6. **Добавьте сообщения** в `lib/cfg/spell_msg.xml`: как минимум сообщения
   о наложении / рассеивании / истечении (для эффектов) или описания урона
   (для атакующих заклинаний).
7. **Обновите снимок сборки** (см. §2.3) и проведите тест запуска:
   `./circle -d small <port>`. **Читайте правильный лог:** SYSERR при загрузке
   попадают в `build/syslog` (режим дозаписи) и `build/log/errlog.txt`, **а не** в
   stdout (в stdout только стартовый баннер). Сначала очистите `build/syslog`,
   запуститесь, затем `grep SYSERR build/syslog` — некорректный `<action>`
   перехватывается и логируется (всё действие отбрасывается), а не роняет сервер,
   поэтому ошибка видна только в syslog. Ищите `Incorrect value '' in
   'talent_actions'` (обязательный атрибут прочитан как пустой) и строки с тегами
   вашего заклинания.

## 16. Что это руководство **пока не охватывает**

* **Тела написанных вручную обработчиков** (`CastCreation`, блоки switch в
  `CastToAlterObjs`, обработчики пост-призыва из `<summon>` и сами обработчики
  `<manual_cast>`) — они по-прежнему находятся в `spells.cpp` и `magic.cpp`. Их
  *выбор и порядок* теперь задаются данными (§3.8.4, §3.8.5), но логика внутри
  остаётся кодом. Её перенос в систему на основе данных ведётся постепенно.
* **Использование броска успеха** — `<success_roll>` разбирается и вычисляется,
  но ещё не подключён к принятию решения об успехе применения. Классический
  `CalcCastSuccess` на основе процентов по-прежнему является шлюзом попадания.
* **Модификаторы помех навыка** — умения, усиливающие конкретные заклинания
  (например, `kMagicArrows` удваивает дополнительные удары `kMagicMissile`),
  по-прежнему жёстко заданы в `CalcExtraHits` и аналогичных вспомогательных функциях.

Если вам нужна любая из этих возможностей — создайте задачу, и работа будет
оценена.

## 17. Константы тестовых заклинаний для прототипирования

Чтобы дизайнеры могли итерировать XML нового заклинания **без перекомпиляции
движка**, постоянно зарезервировано пять заглушек `ESpell`:

| Константа | id | Отображаемое имя (рус.) |
|---|---|---|
| `kTestOne`   | 359 | «тест 1» |
| `kTestTwo`   | 360 | «тест 2» |
| `kTestThree` | 361 | «тест 3» |
| `kTestFour`  | 362 | «тест 4» |
| `kTestFive`  | 363 | «тест 5» |

Каждый слот поставляется в `lib/cfg/spells.xml` в виде минимальной заглушки:

```xml
<spell id="kTestOne" mode="kTesting">
    <name rus="тест 1" eng="test 1"/>
</spell>
```

`mode="kTesting"` не включает слот в списки заклинаний классов в продакшне
(загружается, но не выдаётся), при этом позволяет бессмертному принудительно
применить его для проверки.

**Рабочий процесс:**

1. Выберите свободный слот (`kTestOne` … `kTestFive`).
2. Отредактируйте его элемент `<spell>` в `spells.xml` — добавьте настоящие
   `<misc>`, `<mana>`, `<targets>`, `<flags>`, `<potency_roll>` и нужный
   блок `<talent_actions>`. При желании переименуйте (сохраните `mode="kTesting"`).
3. Добавьте сообщения в `spell_msg.xml` под тем же id.
4. Обновите снимок сборки (см. §2.3) и проведите тест:
   `./circle -d small <port>`, войдите как бессмертный, примените на манекене.
5. Итерируйте только в XML — перекомпиляция не требуется.
6. Когда прототип готов, инженер переименовывает константу и атрибут `id=` в XML
   (например, `kTestOne` → `kFrostshield`), меняет режим на `kEnabled` и добавляет
   в списки заклинаний нужных классов. Весь дизайн, созданный на тестовом слоте,
   переносится дословно.

Держите тестовые слоты краткосрочными: сразу после выпуска прототипа (или отказа
от него) верните слот к минимальной заглушке, чтобы следующий дизайнер нашёл его
пустым.

---

*Последнее обновление соответствует состоянию `sventovit.work` после серии
`issue.ambiguous-spells` / `issue.dispel-default-msg` / `issue.weave-component` /
`issue.caster-blocking-refine` / `issue.affects-potency-weight`:*

*`<misc violent>` получил третье значение `A` («двусмысленное»), разрешаемое
по цели через `IsViolentAgainst(caster, target)` — полезное для себя / очарованного
питомца / членов группы / NPC против NPC, полностью враждебное (с полной
ответственностью за ПК) для посторонних (§3.3; `kDispellMagic` — первый носитель);
новый блок `<components>` на уровне `<spell>` (§3.9) объединяет требования
`<verbal/>` + `<weave/>` + `<material>`, причём `<weave/>` теперь является
единственным источником истины для «это магия» и канонического шлюза комнаты
`kNoMagic`; шлюз кастера перенесён в блок уровня заклинания `<caster_conditions>`,
оборачивающий ту же структуру `<blocking>`/`<required>`, что и сторона цели (§4.3),
взамен старой одно-теговой формы `<caster_blocking>` (так удержанного кастера или
кастера с неподходящим мировоззрением можно запереть через `<align>`/`<affect_flags>`);
новый атрибут
`<affects potency_weight="0.4">` (§8.1) масштабирует сохранённую силу эффекта
при наложении, симметрично `<unaffect potency_weight=>` на стороне снятия; ключи
`kAffDispelledToChar` / `kAffDispelledToRoom` теперь используют запасной пучок
kDefault, поэтому каждое снятие выдаёт хоть что-то — ранее большинство эффектов
(kSanctuary / kBless / kArmor / kFly и т.д.) снимались молча (§12).*

*Сохраняются более ранние дополнения: `<cond>` → `<full>` в `<points>`, ключи
`kHealToVict` / `kMovesToVict` / `kThirstToVict` / `kFullToVict` по категориям,
мигрированное семейство нарратива `kCastSay*` / `kCastIncantToChar` / `kCastHereForbidden*`
/ `kNoTarget` / `kWrongTarget` / `kCastPreparedToChar` / `kCastInterruptedToChar`
/ `kSummonFail` + `kCustomMsgOne…Ten`, переработка первой помощи `BYLINS_FIRSTAID_NEW`,
и прототипирующие слоты `kTestOne…kTestFive` (§17).*
