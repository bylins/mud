# Bylins MUD — Руководство по конфигурации способностей

*Для гейм-дизайнеров, описывающих способности (перки) в `lib/cfg/feats.xml`.*

**Способность** (`EFeat`) — это пассивная черта персонажа: она может давать бонусы
к статам и статусные эффекты, настраивать перезарядку навыков/способностей и —
более новая возможность — **изменять поведение заклинаний и эффектов**, которые
персонаж произносит или носит. Всё в этом руководстве — данные; *доступность*
способности (какому классу, за какую цену) задаётся у классов, а у части
способностей тело всё ещё зашито (см. §6).

Два независимых слоя данных:

* **`<talent_effects>`** — *пассивный* слой (исходный): бонусы к статам/эффектам и
  правки перезарядки навыков/способностей, действующие просто потому, что
  способность есть у персонажа.
* **`<talent_patches>`** — *активный* слой (новее, «патчинг действий перками»):
  способность правит цепочку `<action>` заклинания или эффекта, когда носитель его
  произносит/носит.

Руководство предполагает знание общего словаря действий
[Руководства по заклинаниям](SPELL_MANUAL_RU.md) (`<action>`, `<damage>`,
`<points>`, `<affects>`, …) и [Руководства по эффектам](AFFECT_MANUAL_RU.md)
(триггеры, действия эффектов). Нагрузка `talent_patch` *и есть* этот словарь; здесь
описано, как способность его нацеливает и правит.

---

## 1. Элемент `<feat>`

```xml
<feat id="kDodger" mode="kEnabled">
    <talent_effects> … </talent_effects>     <!-- пассивный слой, §3 -->
    <talent_patches> … </talent_patches>     <!-- активный слой, §4 -->
</feat>
```

### 1.1 Атрибуты заголовка

| атрибут | значение |
|---|---|
| `id` | имя из перечисления `EFeat`. Обязателен. |
| `mode` | состояние жизненного цикла (`EItemMode`): `kEnabled` (в игре), `kTesting` (только разработка/прототипирование), `kDisabled`, `kFrozen`, `kService` (внутреннее). В игре обычно используются только `kEnabled`; `kTesting` — для обкатки новых перков без выпуска. |

Способность может иметь **пустое тело** (`<feat id="kBerserker" mode="kEnabled"/>`)
— как чистая метка, которую боевой/навыковый код проверяет через `CanUseFeat`.
Добавление данных позже это не ломает.

---

## 2. Доступность и гейтинг (как персонаж «владеет» способностью)

Данные способности **не** объявляют свои требования. Закомментированный блок
`<requirements>` вверху `feats.xml` — это желаемый формат, он **не парсится**.
Доступность решается в другом месте, а `CanUseFeat(ch, feat)` (единый гейт,
используемый всем игровым кодом) проверяет по порядку:

1. способность существует и персонаж ею **владеет** (`HaveFeat`);
2. NPC проходят сразу (мобы используют любую данную им способность);
3. **бюджет слотов** покрывает стоимость слота способности —
   `CalcMaxFeatSlotPerLvl(ch) ≥ MUD::Class(class).feats[feat].GetSlot()`;
4. **реморт** удовлетворяет минимуму — `remort ≥ …feats[feat].GetMinRemort()`;
5. у части способностей есть **зашитый гейт по стату** (напр.
   `kWeaponFinesse`/`kFInesseShot` требуют `dex > str` и `dex > 17`;
   `kPowerAttack` требует `str > 19`).

Стоимость **слота** и **мин. реморт** по классам живут в конфигах классов
(`pc_*.xml`), не здесь. Чтобы сделать способность доступной, нужно: (а) добавить её
`EFeat`, (б) дать ей данные здесь, (в) выдать её нужным классам со стоимостью
слота/реморта в конфиге класса.

## 3. `<talent_effects>` — пассивный слой

Контейнер эффектов, действующих, пока способность есть у персонажа. Дети:

### 3.1 `<applies>` — пассивные бонусы к статам / выдача эффектов

```xml
<applies>
    <apply location="kMagicResist" affect="kUndefinded" val="1"
           lvl_bonus="0.0" remort_bonus="1.0" cap="10"/>
    <apply location="kNone" affect="kInfravision" val="0"
           lvl_bonus="0.0" remort_bonus="0.0" cap="0"/>
</applies>
```

| атрибут `<apply>` | значение |
|---|---|
| `location` | `EApply` — стат/производная величина для изменения (`kMagicResist`, `kSavingReflex`, `kSpelledBlinkPhys`, …). `kNone` = только выдать флаг `affect`, без числа. |
| `affect` | флаг `EAffect` для выдачи вместе (напр. `kInfravision`, `kBlink`). `kUndefinded` (sic) = только число, без флага. |
| `val` | базовый модификатор. |
| `lvl_bonus` | прирост за **уровень**, добавляемый к `val`. |
| `remort_bonus` | прирост за **реморт**, добавляемый к `val`. |
| `cap` | ограничение: `cap > 0` — верхняя граница, `cap < 0` — нижняя, `0` — без ограничения. |

**Формула** (`Applies::Impose`):

```
mod = val ± (lvl_bonus·level + remort_bonus·remort)     // знак: '-' для «отрицательного» EApply
                                                        //       (напр. спасброски, где меньше = лучше)
mod = clamp(mod, cap)                                   // cap>0 → min; cap<0 → max
affect_modify(ch, location, mod, affect)                // применяет число И выдаёт флаг эффекта
```

Для «отрицательных» локаций (спасброски, где меньше — лучше) бонус
уровня/реморта **вычитается**, поэтому `remort_bonus` *улучшает* спасбросок. Это
встроенный способ дать масштабируемую пассивку (напр. `kDodger` даёт блинк +
масштабируемые бонусы к спасброскам; `kImpregnable` — сопр. магии/эффектам,
растущее с ремортами, с кэпом 10; `kNightVision` выдаёт эффект `kInfravision`).

> **Внимание.** Эта грамматика apply у способностей (плоские `val`/`lvl_bonus`/
> `remort_bonus`) отличается от грамматики apply у *эффектов* (вложенный
> `<modifier>` с `beta·компетентность`, §4.2 Руководства по эффектам). Способности
> масштабируются от **уровня/реморта**; эффекты — от **компетентности** кастера. Не
> путайте.

### 3.2 `<skill_mods>` — плоские бонусы к навыкам

```xml
<skill_mods>
    <mod id="kRescue" val="20"/>
</skill_mods>
```
Добавляет `val` к эффективному навыку персонажа (`SetAddSkill`).

### 3.3 `<skill_timer_mods>` / `<feat_timer_mods>` — правки перезарядки

```xml
<skill_timer_mods><mod id="kZoneName" val="-2"/></skill_timer_mods>
<feat_timer_mods><mod id="kRelocate" val="-1"/></feat_timer_mods>
```
Правят таймер перезарядки названного навыка / способности, пока способность есть
(читается через `GetTimerMod`).

## 4. `<talent_patches>` — правка заклинаний и эффектов *(активный слой)*

`<talent_patch>` правит цепочку `<action>` **заклинания** (§3.8 Руководства по
заклинаниям) или **эффекта** (§5 Руководства по эффектам), когда носитель
способности его произносит/носит. Это целиком данные — тот же словарь действий,
что у заклинаний/эффектов, плюс *селектор* (какое заклинание/эффект),
*область/родственник* (чью способность проверять, на кого целить добавленное
действие) и *операция* (как править цепочку).

**Как работает.** На старте `BuildTalentPatchIndex` разрешает селектор каждого
патча в конкретные заклинания (или раскладывает по типу эффекта) и валидирует id.
При произнесении/триггере список патчей цели применяется, каждый под гейтом
`CanUseFeat(holder, feat)`. Цена — один тест `.empty()` на незапатченных
заклинаниях; широкий селектор разворачивается только на старте (без цены за каст).
Ошибочно нацеленные патчи громко логируются на старте (совпало 0 заклинаний, нет
id действия, отсутствует проявление).

```xml
<talent_patches>
    <talent_patch spell="kBurningHands" op="replace" action="Main" effect="MainDamage">
        <action> … замена … </action>
    </talent_patch>
</talent_patches>
```

### 4.1 Селекторы — *какие* заклинания / эффект

Все присутствующие селекторы объединяются по **И**; отсутствующий истинен по
умолчанию.

| селектор | совпадает с |
|---|---|
| `spell="kX"` | одно конкретное заклинание |
| `element="kFire"` | все заклинания с таким `EElement` |
| `base_skill="kFireMagic"` | все заклинания, чей базовый навык potency/success совпадает |
| `flag="kMagDamage"` | все заклинания с таким `ESpellFlag` |
| `category="curse"` | все заклинания с таким семантическим тегом |
| `has_effect="kArea"` | все заклинания, чья цепочка содержит такое проявление |
| `all="true"` | все заклинания (осторожно — с гейтом) |
| `affect="kVampirism"` | **эффект** вместо заклинания — патч правит цепочку действий эффекта (§5 Руководства по эффектам). Раскладывается по типу эффекта на старте. |

### 4.2 Область и родственник — *чья* способность, *чья* цель

* `by="target"` — проверять/применять по способностям **цели**, а не кастера (по
  умолчанию — кастер). (Область `kCaster`/`kTarget`.)
* `relative="self|master|group_leader"` — *(патчи эффектов)* чья способность
  гейтит патч относительно носителя эффекта: `self` (по умолчанию), `master`
  (хозяин очарования носителя), `group_leader`. Так `kSoulLink` позволяет
  вампиризму **миньона** лечить его **хозяина** — патч на `kVampirism`, гейт по
  способности хозяина.
* Добавленное действие может целить `kTarMaster` (§5.3 Руководства по эффектам),
  чтобы попасть в этого хозяина.

### 4.3 Операции — *как* править цепочку

| `op` | эффект | требует |
|---|---|---|
| `append` *(по умолч.)* | добавить нагрузку `<action>` в конец цепочки | нагрузка |
| `insert` | вставить нагрузку у якоря | `before="id"` / `after="id"`; нагрузка |
| `replace` | заменить блок (или одно проявление в нём) нагрузкой | `action="id"` (+ опц. `effect="id"`); нагрузка |
| `remove` | удалить блок (или одно проявление) | `action="id"` (+ опц. `effect="id"`) |
| `add_effect` | влить проявления нагрузки в названный блок | `action="id"`; нагрузка |
| `replace_all` | очистить цепочку и поставить нагрузку | нагрузка |
| `modify` | масштабировать/сместить/задать числовое поле проявления | `effect="kKind"` + `<modify>` |

**Адресация по id.** `action="…"` — это `id` блока; `effect="…"` — `id` проявления
внутри блока (для `replace`/`remove`/`add_effect`), **или** — для `modify` — **вид**
проявления (`kDamage`/`kArea`/`kPoints`/`kAffect`). Давайте `<action>`/`<tag>`
целевого заклинания/эффекта стабильные атрибуты `id=`, чтобы операции по id их
находили.

**`modify`** масштабирует одно поле:

```xml
<talent_patch has_effect="kArea" op="modify" effect="kArea">
    <modify field="decay" mul="0.6"/>     <!-- mul= / add= / set= -->
</talent_patch>
```
Поля — числовые члены проявления (`Area`: `decay`/`free_targets`/`max_targets`/…;
`Damage`: `dices_weight`/`beta`/`min`/…). `mul`/`add`/`set` выбирают арифметику.

### 4.4 Нагрузка `<action>`

Для `append`/`insert`/`replace`/`replace_all`/`add_effect` дочерний `<action>` —
обычный блок действия, та же грамматика, что у `<talent_actions>` заклинания (§3.8
Руководства по заклинаниям) или `<actions>` эффекта (§5 Руководства по эффектам).
Он может нести `<trigger>` (для патчей эффектов), `target=` (в т.ч. `kTarMaster`) и
любое проявление (`<damage>`, `<points><heal>`, `<affects>`, `<side_spell>`, …).

## 5. Разобранные примеры (в игре / тестовые)

* **Пассивное сопротивление (масштаб), `kImpregnable`:** `<applies>` для
  `kMagicResist` + `kAffectResist`, `remort_bonus="1.0" cap="10"` — +1 за реморт,
  кэп 10.
* **Пассивная выдача эффекта, `kNightVision`:** один `<apply location="kNone"
  affect="kInfravision"/>` — выдаёт флаг эффекта, без числа.
* **Пассивное число + эффект, `kDodger`:** блинк (`kSpelledBlinkPhys`, эффект
  `kBlink`) + масштабируемые бонусы к спасброскам.
* **Патч эффекта (родственник), `kSoulLink`:** `<talent_patch affect="kVampirism"
  relative="master" op="append"><action target="kTarMaster" …><trigger
  val="kPostHit"/><points><heal beta="0.05"/></points></action>` — вампиризм
  миньона лечит ещё и хозяина. Гейт по способности *хозяина*.
* **Замена в заклинании, `kEmberFury`:** `<talent_patch spell="kBurningHands"
  op="replace" action="Main" effect="MainDamage">` — заменить блок урона заклинания.
* **Классовый селектор append, `kPyromancer`:** `<talent_patch element="kFire"
  flag="kMagDamage" op="append">` — добавить эффект каждому огненному урон-заклинанию.
* **Modify, способность на area:** `<talent_patch has_effect="kArea" op="modify"
  effect="kArea"><modify field="decay" mul="0.6"/>` — смягчить затухание area у
  каждого area-заклинания.

## 6. Что *пока* не данные

* **Зашитые тела способностей.** Многие способности всё ещё проверяются прямо в
  C++ (боевые приёмы, передвижение/восприятие, гейты торговли/крафта, баффы
  призывов). По сути это отдельные способности, и им нужен ability-пайплайн, прежде
  чем стать данными — они **не** кандидаты для `talent_effects`/`talent_patch`.
* **Часть модификаторов заклинаний** (напр. флаг «игнорировать поглощение против
  NPC» у `kPowerMagic`, баффы призывов вроде `kFuryOfDarkness`) трогают *флаги*
  заклинаний / *призванных мобов* — форм, которые текущая грамматика `talent_patch`
  (правки числовых полей + операции над цепочкой) пока не выражает. Их чисто
  числовые части уже живут в `<talent_effects>` (напр. +50% маг. урона у
  `kPowerMagic`).

## 7. Справочник

* **`EItemMode`** — `kEnabled`/`kTesting`/`kDisabled`/`kFrozen`/`kService`.
* **`EApply`** (`location=` у apply) — `entities_constants.h`; общий с заклинаниями/эффектами.
* **`EAffect`** (`affect=` у apply) — [Руководство по эффектам](AFFECT_MANUAL_RU.md).
* **Операции патча** — `append`/`insert`/`replace`/`remove`/`add_effect`/`replace_all`/`modify`.
* **Селекторы патча** — `spell`/`element`/`base_skill`/`flag`/`category`/`has_effect`/`all`/`affect`.
* **Грамматика действий/проявлений** — §3.8 / §5–§8 Руководства по заклинаниям; §5
  Руководства по эффектам.

## 8. Чек-лист для добавления / миграции способности

1. **Добавьте id `EFeat`** и блок `<feat id= mode=>` (`kTesting` на время обкатки).
2. **Выдайте её классу** в `pc_*.xml` со стоимостью слота + мин. ремортом (именно
   это делает её получаемой).
3. **Пассивный стат/эффект?** → `<talent_effects><applies>` (`val` + `lvl_bonus`/
   `remort_bonus`, `cap`, опц. `affect=`).
4. **Перезарядка навыка/способности или бонус к навыку?** → `<skill_mods>` /
   `<skill_timer_mods>` / `<feat_timer_mods>`.
5. **Правит заклинание/эффект?** → `<talent_patches>`: выберите селектор
   (конкретный или классовый), операцию и (для патчей эффектов) `relative`; дайте
   цели стабильные `id=`, если используете операции по id.
6. **Переключите `mode="kEnabled"`**, когда готово; проверьте на старте (нет
   предупреждений патчей) и в mud-sim (выдайте способность, произнесите/триггерните
   затронутое заклинание/эффект, подтвердите, что эффект есть у носителей и нет у
   невладеющих).
